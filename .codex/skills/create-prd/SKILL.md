# Create PRD

## 개요

`docs/spec/spec-fixed.md`를 원천 명세로 삼아 전체 허브 PRD와 기능별 PRD를 생성하거나 갱신한다. feature 단위는 고정 목록을 쓰지 않고, 현재 명세의 사용자 가치, 기능 책임, 상태 경계, 수용 기준, 테스트 가능성을 기준으로 매번 도출한다.

## 기본 원칙

- 모든 질문, 분석, 문서 초안은 한국어로 작성한다.
- 먼저 `docs/spec/spec-fixed.md`를 끝까지 읽고 큰 번호 섹션 목록을 확인한다.
- `docs/spec/spec-fixed.md`는 읽기 전용 원천 문서로 취급하고 수정하지 않는다.
- `spec-interview`처럼 요구사항을 새로 인터뷰하거나 원천 명세를 보완하지 않는다.
- `spec-review`처럼 큰 번호 섹션별 리뷰나 검토 확정 절차를 수행하지 않는다.
- TRD, 기술 아키텍처, 하네스, CI gate, STOMP/WebSocket 상세 약속, DB schema, endpoint payload는 작성하지 않는다.
- issue 파일과 ADR 파일은 생성하지 않는다.
- feature 목록을 하드코딩하지 않는다. 도출한 feature 이름은 소문자 hyphen-case 영문 폴더명으로 정한다.
- `docs/prd.md`는 긴 상세 문서가 아니라 허브/요약/링크 문서로 작성한다.
- 상세 제품 요구사항은 `docs/features/{feature}/prd.md`에 작성한다.
- 원천 명세의 큰 번호 섹션은 feature 폴더명으로 쓰지 않고 `source_sections`와 `docs/traceability.md`로 추적한다.
- 사용자가 명시적으로 일괄 진행을 요청하지 않는 한, 허브 PRD와 기능별 PRD는 문서별 사용자 승인 게이트를 통과한 뒤 다음 문서로 진행한다.

## 대상 범위

사용자가 `{feature}` 인자를 주면 해당 feature만 처리한다. 예를 들어 `$create-prd matchmaking`처럼 요청하면 `docs/features/matchmaking/prd.md`만 생성하거나 갱신한다.

인자가 없으면 전체 feature 후보를 도출하고 `docs/prd.md`와 기능별 PRD 세트를 순서대로 생성하거나 갱신한다.

지정된 `{feature}`가 기존 문서와 매칭되지 않으면 유사한 feature를 추측해 진행하지 않는다. 원천 명세 분석 결과와 기존 `docs/features/*/prd.md`를 기준으로 후보를 제시하고 사용자 확인을 받는다.

## 진행 절차

1. `docs/spec/spec-fixed.md` 전체를 읽는다.
2. spec-review 스킬 내 `scripts/extract_spec_sections.py --file docs/spec/spec-fixed.md --list`로 큰 번호 섹션 목록과 라인 번호를 확인한다.
3. `{feature}` 인자가 있는지 확인하고 처리 범위를 확정한다.
4. 원천 섹션을 다음 기준으로 분류한다.
   - 허브 PRD에 반영할 전역 섹션: 개요, 문서 목적, 용어, MVP 범위, 전체 수용 기준, 확장 후보 등
   - 기능별 PRD로 분리할 섹션: 사용자 기능, 게임 기능, 운영 기능처럼 사용자 가치와 수용 기준을 가질 수 있는 도메인
   - TRD로 넘길 섹션: 상세 기술 계약, 서버 내부 책임, 상태 불변식, 테스트 하네스, 운영/관측, 구현 순서
5. feature 후보를 도출한다.
6. 원천 명세의 `예외 및 엣지 케이스`, `핵심 수용 기준`, `구현 전 필수 확정 사항`, `MVP 제외 및 이후 확장 후보`를 관련 feature PRD로 재배치할 후보를 표시한다.
7. 파일 작성 전에 feature map, 처리 범위, 작성 순서, 생성/갱신 대상 문서를 세션에서 제시하고 사용자 승인을 받는다.
8. 인자가 없으면 `docs/prd.md`를 생성하거나 갱신하고 확정받은 뒤 기능별 PRD로 진행한다.
9. 기능별 `docs/features/{feature}/prd.md`를 하나씩 생성하거나 갱신한다. 각 문서 초안 작성 후 확정 여부를 묻고, 확정 전에는 후속 문서의 기준으로 삼지 않는다.
10. 필요하면 `docs/traceability.md`에 원천 섹션과 PRD 산출물의 연결을 생성하거나 보강한다.
11. 작성 후에는 생성/수정한 파일, 도출한 feature 목록, 남은 모호함, 원천 섹션 추적 방식, 승인 완료된 문서 목록을 간단히 보고한다.

## Feature 도출 기준

feature를 만들 때는 `spec-fixed.md`의 큰 번호 섹션을 그대로 폴더로 옮기지 않는다. 섹션은 문서 설명 순서이고, feature는 제품 요구사항과 구현 협업의 경계다.

다음 경우는 feature보다 허브 PRD에 가깝다.

- 문서 목적과 범위
- 용어 정의
- MVP 포함/제외 범위
- 전체 사용자 시나리오
- 전체 수용 기준
- 구현 전 확정 사항
- 이후 확장 후보

다음 경우는 feature 후보로 검토한다.

- 사용자가 직접 사용하는 독립 기능
- 운영자가 직접 사용하는 독립 기능
- 별도 도메인 상태 또는 상태 전이를 갖는 기능
- 별도 사용자 시나리오와 수용 기준을 가질 수 있는 기능
- 후속 TRD와 구현 issue로 나눌 수 있는 책임 경계가 있는 기능

## 문서 작성 규칙

- 원천 명세에서 확정된 요구사항과 추론을 구분한다. 추론은 "원천 명세로부터의 정리" 또는 "문서화 가정"으로 표시한다.
- `docs/prd.md`는 목적, 원천 문서, 전체 원칙, feature PRD 링크, 미확정 사항 요약을 담는 허브로 작성한다.
- 기능별 PRD에는 대상 사용자, 사용자 가치, 포함/제외 범위, 주요 흐름, 상세 사용자 시나리오, 상태/권한별 엣지 케이스, 사용자에게 보이는 결과, 수용 기준, 미확정 사항을 작성한다.
- 기능별 PRD의 상세 사용자 시나리오는 최소 2개 이상 작성한다. 성공 흐름만 쓰지 말고 실패, 제한, 경계 조건 중 적어도 1개를 포함한다.
- 실패 흐름에는 사용자에게 보이는 결과와 변경되면 안 되는 상태를 함께 적는다.
- 원천 명세에 없는 endpoint, payload, DB schema, 정책값은 임의 확정하지 않고 "계약 후보" 또는 "미확정 사항"으로 표시한다.
- 기술 구현 방식, 서버 내부 구조, 테스트 하네스, 운영 관측, 구체적인 오류 코드 명칭은 TRD 또는 아키텍처 산출물로 넘긴다.
- 확정된 문서의 결정사항은 후속 문서에서 기준으로 사용할 수 있다.
- 미확정 문서의 내용은 후속 문서에 단정적으로 반영하지 않는다.
- 문서별 승인 요청은 초안 요약, 생성/수정 파일, 확정 여부 질문, 수정 요청 가능성을 짧게 포함한다.

## 문서 구조

다음 구조만 생성하거나 갱신한다.

```text
docs/
  prd.md
  traceability.md
  features/
    {feature}/
      prd.md
```

다음 구조는 생성하지 않는다.

```text
docs/trd.md
docs/milestones.md
docs/websocket-spec.md
docs/features/{feature}/trd.md
docs/features/{feature}/issues/
docs/features/{feature}/adr/
```

## Feature PRD 필수 섹션

기능별 `prd.md`는 다음 섹션을 포함한다.

- `# {Feature Name} PRD`
- `## 목적`
- `## 원천 섹션`
- `## 대상 사용자`
- `## 범위`
- `## 주요 사용자 흐름`
- `## 상세 사용자 시나리오`
- `## 상태/권한별 엣지 케이스`
- `## 사용자에게 보이는 결과`
- `## 수용 기준`
- `## 미확정 사항`
- `## 관련 문서`

상세 사용자 시나리오는 다음 형식을 따른다.

```markdown
### 시나리오 N. {성공, 실패, 제한, 경계 흐름 이름}

- 시작 조건: {계정 상태, 데이터 상태, 경기/콘텐츠 상태}
- 사용자 행동: {사용자가 수행하는 행동}
- 시스템 반응: {시스템이 검증하고 확정하는 내용}
- 사용자에게 보이는 결과: {사용자가 확인할 수 있는 확정 결과}
- 완료 조건: {성공 조건 또는 변경되지 않아야 하는 상태}
```

## 보조 리소스

- 큰 번호 섹션 목록 추출과 단일 섹션 발췌에는 `scripts/extract_spec_sections.py`를 사용한다.

## 검증

문서 생성 후 다음을 확인한다.

- `docs/spec/spec-fixed.md`가 수정되지 않았다.
- TRD, STOMP/WebSocket 상세 약속, 기술 아키텍처, 하네스, CI gate, issue, ADR이 생성되지 않았다.
- 허브 PRD와 기능별 PRD가 문서별 사용자 승인 게이트를 통과한 뒤 다음 문서로 진행되었다.
- feature 목록이 스킬에 하드코딩된 기본값이 아니라 현재 원천 명세 분석 결과로 설명된다.
- `docs/traceability.md`에서 원천 섹션과 PRD 산출 문서의 연결을 추적할 수 있다.
- `docs/prd.md`는 허브 문서이고, 상세 제품 요구사항은 기능별 PRD에 위치한다.
- 각 기능별 PRD에 최소 2개 이상의 상세 사용자 시나리오가 있다.
- 각 기능별 PRD에 상태/권한별 엣지 케이스, 사용자에게 보이는 결과, 수용 기준, 미확정 사항이 있다.
- 원천 명세의 예외와 수용 기준이 허브 PRD에만 머물지 않고 관련 기능별 PRD에 재배치되어 있다.

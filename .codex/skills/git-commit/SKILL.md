# Git Commit

## 개요

현재 작업트리의 변경 사항을 하나의 의도로 묶어 커밋한다. 변경 분석, 관련 파일 stage, 커밋 메시지 작성, hook 실패 대응까지 수행하되 `--no-verify`로 검증을 우회하지 않는다.

## 기본 원칙

- 사용자-facing 설명과 커밋 메시지는 한국어로 작성한다.
- 커밋 전에 반드시 `git status --short --branch`, `git diff`, `git diff --staged`로 변경 범위를 확인한다.
- 하나의 커밋에는 하나의 대표 의도만 담는다.
- 사용자의 별도 승인 없이 커밋을 실행한다.
- 사용자가 명시하지 않은 기존 변경을 되돌리거나 삭제하지 않는다.
- hook이 실패하면 우회하지 않고 실패 원인을 분석한다.

## Stage 정책

1. staged 변경이 있으면 staged 변경의 의도를 우선한다.
2. 같은 의도의 unstaged 파일이 누락되어 있으면 함께 stage한다.
3. staged 변경이 없으면 전체 diff를 읽고 하나의 작업으로 판단되는 관련 파일만 stage한다.
4. 서로 무관한 변경이 섞여 있거나 커밋 범위를 신뢰할 수 없으면 커밋하지 않고 분리 필요성을 보고한다.
5. `git add -A`는 모든 변경이 명확히 같은 의도일 때만 사용한다.

## 접두어 선택

커밋 내용에 따라 아래 접두어 중 하나를 고른다. 여러 종류에 해당하면 가장 대표적인 의도를 선택한다.

| 접두어 | 기준 |
| --- | --- |
| `feat` | 기능 추가 및 기존 기능 변경 |
| `fix` | 에러, 버그, 실패하는 동작 수정 |
| `docs` | Markdown, txt 등 문서 파일만 수정 |
| `chore` | 빌드 도구, 설정, 의존성, 운영 보조 작업처럼 제품 기능과 직접 관련 없는 수정 |
| `refactor` | 기능 추가나 동작 변경 없이 기존 기능 구조만 개선 |
| `style` | 코드 스타일, 포맷, 공백, import 정렬처럼 의미 변화가 없는 수정 |
| `test` | 테스트 코드 추가 또는 수정이 대표 의도인 경우 |

## 메시지 규칙

형식:

```text
{접두어}: {커밋 메시지}

{body}
```

- 제목은 50자를 넘기지 않는다.
- 제목은 명사형으로 끝내 간결하게 쓴다.
- 제목에는 변경의 대표 결과를 요약한다.
- 이유나 구현 방식이 50자 안에 들어가지 않으면 제목은 요약만 쓰고 body에 적는다.
- body는 반드시 쓰지 않아도 된다.
- body를 쓰면 `왜`, `어떻게`를 중심으로 bullet로 작성한다.
- body bullet도 가능한 한 명사형으로 끝낸다.

예:

```text
feat: 문자열 병합 기능 추가

- A 클래스의 필요성에 의한 추가
- 기존 라이브러리에 B 클래스 기능을 넣어 안정성 보강
```

## 커밋 실행 절차

1. 변경 범위를 확인하고 커밋 의도를 한 문장으로 파악한다.
2. Stage 정책에 따라 관련 파일만 stage한다.
3. `git diff --staged`를 다시 읽고 staged 범위가 하나의 의도인지 확인한다.
4. 접두어와 제목을 정하고 필요한 경우 body를 작성한다.
5. `git commit -m "<title>"` 또는 body가 있으면 여러 `-m` 인자를 사용해 커밋한다.
6. 커밋 성공 후 커밋 해시와 최종 메시지를 간단히 보고한다.

## Hook 실패 처리

- `commit-msg` 실패처럼 메시지 형식 문제이면 hook 출력의 요구사항을 근거로 메시지를 고쳐 재시도한다.
- 메시지 재작성 재시도는 최대 2회까지 수행한다.
- `pre-commit`, 테스트, 정적 분석 실패처럼 구현 문제이면 커밋을 중단하고 실패한 명령, 핵심 원인, 수정 필요 지점을 보고한다.
- hook 실패 후에도 `--no-verify`를 사용하지 않는다.

## 저장소 Hook 규칙

- `.githooks/pre-commit`은 staged 파일 중 Java/Gradle 컴파일 관련 변경이 있을 때 `./gradlew compileJava` 또는 `./gradlew compileTestJava`를 실행해 컴파일 에러가 있는 커밋을 막는다.
- staged 컴파일 관련 파일에 unstaged 변경이 함께 있으면 커밋 대상 기준 검증이 불명확하므로 커밋을 중단한다.
- `.githooks/commit-msg`는 `{접두어}: {커밋 메시지}` 형식, 허용 접두어, 50자 이하 제목, 제목/body 구분 공백, body bullet 형식을 검증한다.
- hook이 메시지를 거부하면 출력된 원인을 반영해 메시지를 다시 작성한다.

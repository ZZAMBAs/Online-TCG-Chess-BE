# Spec Interview

## 개요

`docs/spec/spec-original.md`를 기준 자료로 삼아 요구사항의 빈칸, 모호함, 예외 상황, 정책 결정을 사용자와 인터뷰로 채운다. 모든 질문과 산출물은 한국어로 작성하고, 최종 산출물은 `docs/spec/spec-fixed.md`에 저장한다.

## 기본 원칙

- 먼저 `docs/spec/spec-original.md`를 끝까지 읽고, 이미 존재하는 `docs/spec/spec-fixed.md`가 있으면 이전 산출물 또는 참고용으로 확인한다.
- 질문은 반드시 한 번에 하나만 한다.
- 질문은 충분하다고 판단될 때까지 최대한 많이 한다. 기능 요구사항뿐 아니라 비기능 요구사항, 권한, 데이터, 정책, 오류 처리, 엣지 케이스, 운영 조건, 추후 확장 가능성까지 점검한다.
- 각 질문에는 추천 답변과 추천 이유를 포함한다.
- 추천 답변은 프로젝트 맥락과 일반적인 제품 설계 원칙에 근거해 제시한다.
- 법, 보안, 결제, 개인정보, 표준, 외부 API, 플랫폼 정책처럼 정확한 검증이 필요한 내용은 신뢰할 수 있는 공식 문서나 기관 자료를 확인한 뒤 근거를 짧게 밝힌다. 최신성이 필요한 경우 반드시 현재 자료를 확인한다.
- 사용자의 답변이 새 모호함을 만들면 후속 질문을 이어간다.
- 사용자가 "추천대로", "알아서", "계속"처럼 위임하면 추천 답변을 채택한 것으로 기록하고 다음 질문으로 넘어간다.
- 충분히 구체화되기 전에는 `docs/spec/spec-fixed.md`를 작성하지 않는다.
- 실제 파일에 작성하기 전에는 반드시 세션 응답으로 반영 예정 요약을 먼저 보여주고, 사용자가 반영을 승인한 뒤에만 파일을 수정한다.
- 최종 작성 시에는 추측과 확정 사항을 섞지 않는다. 확정되지 않은 항목은 "미결정" 또는 "가정"으로 분명히 표시한다.

## 진행 절차

1. `docs/spec/spec-original.md`를 읽고 요구사항 후보를 도메인별로 나눈다.
2. 다음 관점에서 모호하거나 누락된 지점을 찾는다.
   - 목표, 사용자, 주요 시나리오
   - 반복 사용되는 용어, 동의어, 코드 용어, 대화에서 섞여 쓰인 표현
   - 기능 범위와 제외 범위
   - 사용자 역할, 권한, 인증/인가
   - 화면, API, 데이터 모델, 상태 전이
   - 게임/비즈니스 규칙, 정책, 우선순위
   - 오류 처리, 동시성, 중복 요청, 취소/복구
   - 보안, 개인정보, 감사 로그, 보존 정책
   - 성능, 확장성, 운영, 관측성
   - 외부 연동, 버전 호환, 마이그레이션
   - 엣지 케이스와 악용 가능성
3. 가장 블로킹이 큰 항목부터 질문한다.
4. 사용자의 답변을 내부 결정 기록으로 유지하고, 다음 질문을 이어간다.
5. 반복되는 용어, 코드 식별자, UI 문구, 사용자 표현이 같은 개념을 가리키는지 점검하고, 필요한 경우 용어 통일 질문을 한다.
6. 더 이상 의미 있는 질문이 없거나 남은 사항을 명시적 가정으로 처리해도 되는지 확인되면, 파일을 쓰기 전에 세션에서 반영 예정 요약을 보여주고 승인을 받는다.
7. 사용자가 승인하면 `docs/spec/spec-fixed.md`를 작성한다. 사용자가 수정 요청을 하면 추가 질문 또는 요약 수정을 먼저 진행한다.
8. 작성 후에는 변경한 파일과 남은 미결정 사항을 간단히 보고한다.

## 질문 형식

질문은 다음 구조를 유지한다.

```markdown
질문: [하나의 구체적인 질문]

추천 답변: [권장하는 선택 또는 문장]

추천 이유: [왜 이 답변이 현재 스펙에 적합한지 설명]

필요하면 참고 근거: [공식 문서나 신뢰 가능한 기관 자료를 확인한 경우만 짧게 기재]
```

한 번에 여러 질문을 묶지 않는다. 선택지가 필요하면 한 질문 안에서 2-4개의 선택지를 제시하되, 사용자가 답해야 하는 쟁점은 하나로 유지한다.

## 용어 정의 관리

인터뷰 중 용어 사전을 계속 갱신한다. 기능 안에서 반복되는 용어, 코드 용어, UI 문구, 사용자 대화에서 나온 표현이 사실상 같은 개념을 가리킬 수 있으므로, 최종 문서에 반드시 `용어 정의` 섹션을 둔다.

용어를 다룰 때는 다음을 수행한다.

- `표준 용어`, `동의어/이전 표현`, `정의`, `사용 맥락`, `관련 코드/API 이름`을 구분해 기록한다.
- 같은 개념에 여러 이름이 있으면 어떤 용어로 통일할지 질문한다.
- 사용자가 동의하면 이후 질문과 최종 문서에서는 정의된 표준 용어를 사용한다.
- 코드나 DB에 이미 굳어진 이름이 있으면 문서 용어와 코드 용어의 매핑을 남긴다.
- 용어 정의가 기능 요구사항의 해석을 바꿀 수 있으면 기능 질문보다 먼저 확인한다.

용어 질문 예시는 다음 형식을 따른다.

```markdown
질문: `[표현 A]`, `[표현 B]`, `[코드 용어 C]`가 같은 개념을 가리키는 것으로 보입니다. 최종 문서에서는 어떤 표준 용어로 통일할까요?

추천 답변: `[표준 용어]`로 통일하고, 용어 정의에 `[표현 A]`, `[표현 B]`, `[코드 용어 C]`를 동의어 또는 코드 매핑으로 남긴다.

추천 이유: 문서, 코드, 대화에서 같은 개념을 같은 이름으로 다루면 이후 요구사항 변경과 구현 검토 시 혼선을 줄일 수 있다.
```

## 작성 전 확인

`docs/spec/spec-fixed.md` 또는 사용자가 요청한 실제 스펙 파일을 수정하기 전에 반드시 세션에서만 요약을 먼저 제시한다. 이 단계에서는 파일을 쓰지 않는다.

요약에는 다음을 포함한다.

- 반영할 주요 요구사항
- 확정된 용어 정의와 용어 통일 방식
- 주요 정책 결정과 엣지 케이스 처리
- 명시적 가정
- 미결정 사항
- 작성 또는 수정 대상 파일

요약 끝에는 "이 내용으로 파일에 반영해도 될까요?"라고 확인한다. 사용자가 명확히 승인한 뒤에만 파일을 작성하거나 수정한다.

## 최종 문서 작성 기준

`docs/spec/spec-fixed.md`는 한국어로 자세히 작성한다. 프로젝트에 맞게 섹션을 조정하되, 가능한 한 다음 내용을 포함한다.

- 문서 목적과 범위
- 제품/기능 개요
- 용어 정의
- 사용자 역할과 권한
- 주요 사용자 시나리오
- 기능 요구사항
- 비기능 요구사항
- 데이터 요구사항
- API 또는 인터페이스 요구사항
- 상태, 규칙, 정책
- 예외 및 엣지 케이스
- 보안 및 개인정보 요구사항
- 운영, 로깅, 모니터링 요구사항
- 수용 기준
- 미결정 사항과 명시적 가정

기존 `docs/spec/spec-fixed.md`가 있으면 무조건 덮어쓰기 전에 내용을 읽고, 새 인터뷰 결과를 반영해 정리한다. 사용자나 시스템이 기존 내용을 보존하라고 지시한 경우 그 지시를 따른다.

`용어 정의` 섹션은 생략하지 않는다. 정의된 용어가 이후 문서 본문에서 반복될 때는 표준 용어를 우선 사용하고, 필요한 곳에서만 코드 용어 또는 이전 표현을 괄호로 병기한다.

# Spec Review

## 개요

요구사항 명세를 큰 번호 섹션 단위로 검토한다. 메인 에이전트가 대상 섹션 본문만 추출해 서브에이전트에게 전달하고, 기획자·QA/보안·FE/BE 개발자 관점의 독립 결과를 종합해 사용자와 하나씩 확정한다.

## 기본 원칙

- 모든 질문, 분석, 추천안, 요약은 한국어로 작성한다.
- 대상 문서의 섹션 개수를 고정하지 않는다. 현재 파일에서 발견되는 모든 `## N. 제목` 큰 번호 섹션을 기준으로 한다.
- 서브에이전트에게 전체 `spec-fixed.md` 경로나 전체 본문을 넘기지 않는다.
- 서브에이전트에게는 메인 에이전트가 추출한 대상 섹션 본문과 필요한 최소 주변 맥락만 전달한다.
- 한 번에 하나의 큰 번호 섹션을 사용자와 확정한다.
- 사용자가 "추천대로", "확정", "좋아", "OK"처럼 동의하면 추천안을 채택한 결정으로 기록하고 다음 섹션으로 넘어간다.
- 사용자 승인 전에는 스펙 파일을 수정하지 않는다.
- 모든 섹션 검토가 끝난 뒤 반영 예정 요약을 먼저 제시하고, 사용자가 승인한 뒤에만 파일을 수정한다.

## 섹션 추출

대상 파일이 명시되지 않으면 `docs/spec/spec-fixed.md`를 기본 대상으로 사용한다.

먼저 큰 번호 섹션 목록을 확인한다.

```powershell
python .codex\skills\spec-review\scripts\extract_spec_sections.py --file docs\spec\spec-fixed.md --list
```

특정 섹션 본문만 추출한다.

```powershell
python .codex\skills\spec-review\scripts\extract_spec_sections.py --file docs\spec\spec-fixed.md --section 10
```

검토 중 관련 맥락이 필요하면 메인 에이전트가 직접 판단해 인접 섹션 제목이나 관련 섹션의 짧은 발췌만 추가한다. 서브에이전트에게 "파일을 직접 읽어라"라고 지시하지 않는다.

## 검토 절차

1. 대상 스펙 파일의 큰 번호 섹션 목록을 추출한다.
2. 사용자가 지정한 범위가 있으면 해당 섹션만, 없으면 발견된 큰 번호 섹션을 오름차순으로 검토한다.
3. 각 섹션마다 메인 에이전트가 섹션 본문을 추출한다.
4. 동일 섹션에 대해 세 서브에이전트를 병렬로 호출한다.
5. 각 서브에이전트에는 같은 섹션 발췌본을 주되, 서로 다른 관점과 산출 형식을 지시한다.
6. 메인 에이전트는 세 결과를 중복 제거하고 충돌 지점을 표시한다.
7. 사용자에게 해당 섹션의 추천 확정안을 제시한다.
8. 사용자의 결정이 모이면 결정 기록을 유지하고 다음 섹션으로 넘어간다.
9. 모든 대상 섹션 검토가 끝나면 전체 반영 예정 요약을 제시한다.
10. 사용자가 승인한 뒤에만 대상 스펙 파일을 수정한다.

## 서브에이전트 프롬프트 기준

### 기획자 관점

전달할 내용:

- 대상 섹션 번호와 제목
- 대상 섹션 본문
- 필요한 최소 주변 맥락

검토 지시:

- 사용자 시나리오가 자연스러운지 검토한다.
- MVP 포함/제외 범위가 명확한지 검토한다.
- 용어, 정책, 상태, 확장 후보가 다른 섹션과 충돌하지 않는지 검토한다.
- 사용자에게 추가 확인이 필요한 제품 결정을 찾는다.
- 파일은 수정하지 않는다.

### QA/보안 관점

전달할 내용:

- 대상 섹션 번호와 제목
- 대상 섹션 본문
- 필요한 최소 주변 맥락

검토 지시:

- 악의적 사용자가 시도할 우회, 남용, 조작 가능성을 검토한다.
- 인증/인가, 권한, 세션, 개인정보, 감사 로그, 보존 정책 누락을 찾는다.
- 동시성, 중복 요청, 재시도, 실패 후 상태 불변성, 경계값 테스트 가능성을 검토한다.
- 수용 기준이나 오류 코드가 테스트 가능하게 표현되어 있는지 확인한다.
- 파일은 수정하지 않는다.

### FE/BE 개발자 관점

전달할 내용:

- 대상 섹션 번호와 제목
- 대상 섹션 본문
- 필요한 최소 주변 맥락

검토 지시:

- 프론트엔드 표시 계약, pending/확정 상태, 오류 표시, 재동기화 요구를 검토한다.
- 백엔드 REST, STOMP, 도메인 서비스, 데이터 모델, 상태 전이 구현 가능성을 검토한다.
- API/event/schema가 불완전하거나 구현자가 임의 결정해야 하는 부분을 찾는다.
- FE와 BE 사이 책임 경계가 불명확한 부분을 찾는다.
- 파일은 수정하지 않는다.

## 종합 응답 형식

섹션별 사용자 응답은 다음 구조를 따른다.

```markdown
섹션: [번호와 제목]

검토 요약:
- [핵심 이슈]

추천 확정안:
- [문서에 반영할 구체 문장 또는 정책]

추천 이유:
- [기획/QA/보안/개발 관점의 근거]

확인 필요:
- [사용자 결정이 필요한 항목. 없으면 "없음"]
```

사용자가 확정하기 전에는 다음 섹션으로 넘어가지 않는다. 단, 사용자가 "계속", "추천대로 계속"처럼 명시적으로 위임하면 추천안을 채택하고 다음 섹션으로 이동한다.

## 파일 반영 전 요약

스펙 파일을 수정하기 전에는 반드시 다음을 세션 응답으로 먼저 제시한다.

- 섹션별 주요 반영 사항
- 확정된 용어와 정책
- 주요 보안/QA/구현 보완 사항
- 유지할 미결정 사항과 가정
- 수정 대상 파일

요약 끝에는 "이 내용으로 파일에 반영해도 될까요?"라고 확인한다. 사용자가 명확히 승인한 뒤에만 파일을 수정한다.

## 서브에이전트가 없을 때

서브에이전트 도구를 사용할 수 없으면 메인 에이전트가 같은 세 관점을 순차적으로 검토한다. 이 경우 사용자에게 서브에이전트 대신 단일 에이전트 순차 검토로 진행한다고 알린다.

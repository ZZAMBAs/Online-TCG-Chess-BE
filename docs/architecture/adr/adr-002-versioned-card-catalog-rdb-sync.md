# ARCH-ADR-002. 버전 카드 catalog를 RDB와 동기화하고 활성 버전을 경기별로 고정

## Context

카드는 영구 식별자와 불변 버전을 가지며, 규칙·조건·공개 의미가 바뀌면 새 버전을 발행해야 한다. 진행 중 경기와 과거 기보는 이후 카드 변경으로 의미가 달라지면 안 되고, 신규 경기만 선택된 새 버전을 사용해야 한다. 동시에 카드 컬렉션·덱·통계는 MySQL 관계와 조회를 필요로 한다.

기존 아키텍처는 MySQL/JPA/Flyway와 `GameEvent` 기반 기보를 확정했지만 카드/팩 master 파일 포맷과 seed 연결 방식은 미확정이었다. JSON만 수정해 새로운 실행 규칙이 자동 생성되는 것으로 간주하면 정의와 효과 처리의 불일치가 배포될 위험이 있다.

## Decision

버전 관리되는 `docs/cards` 카드 JSON과 별도 활성 버전 목록을 카드 작성·배포 원천으로 사용한다. RDB는 검증·배포된 카드의 실행·조회 원천이다. 검증된 정의를 애플리케이션 산출물에 포함하고, 배포된 모든 카드 식별자·버전을 RDB에 불변으로 동기화한다. 이미 존재하는 동일 식별자·버전의 내용이 다르면 기동을 실패시키며 새 버전만 추가한다.

카드별 활성 버전은 버전 정의와 분리해 하나만 선택한다. 활성 버전은 RDB 직접 수정이 아니라 활성 버전 목록을 변경한 새 배포에서만 전환한다. 신규 경기 생성 시 활성 버전을 조회해 경기 덱에 고정하며, 활성 버전 변경은 이미 생성된 경기와 과거 기보에 영향을 주지 않는다. 과거 기보는 저장된 `GameEvent` 결과와 당시 카드 식별자·버전을 사용하고 최신 효과 처리를 재실행하지 않는다.

활성화하거나 롤백 대상으로 유지하는 모든 버전에는 효과 처리와 선택·공개 계약·fixture가 동반되어야 한다. 기존 처리 방식으로 표현할 수 없는 효과 변경에는 해당 버전의 새 처리를 추가한다. catalog, 활성 버전, 효과 처리와 계약의 완결성이 검증되지 않으면 배포 가능한 산출물을 만들지 않는다.

## Alternatives

| 대안 | 장점 | 단점/위험 | 현재 상황에서의 적합성 | 선택 또는 제외 이유 |
| --- | --- | --- | --- | --- |
| 저장소 JSON 원천 + 패키징 + RDB 불변 동기화 | 코드 리뷰와 제품 정의 변경을 함께 추적하고 RDB 관계·조회와 롤백을 지원한다. | 동기화기와 이중 표현 간 일치 검증이 필요하다. | 이미 `docs/cards`, MySQL/JPA와 버전 고정 요구가 있다. | 선택. 작성 원천과 실행 저장소의 역할을 분리하고 Hard gate로 drift를 차단할 수 있다. |
| RDB/Flyway만 원천 | 실행 데이터와 원천이 하나라 동기화기가 필요 없다. | 카드 제품 정의·FE 계약·효과 코드와 migration이 분리되고 운영 데이터 수정이 작성 흐름이 된다. | 카드 정의가 저장소 문서와 계약의 입력으로 이미 사용된다. | 제외. 현재 문서 중심 협업과 불변 버전 검증 경계를 약화한다. |
| 패키징 JSON만 실행 원천 | RDB 동기화가 없고 구현이 단순하다. | 컬렉션·덱 관계, 활성 버전 조회, 운영 확인과 모든 배포 버전 보존 책임이 애플리케이션 파일에 집중된다. | MySQL이 카드·덱 영속 원천인 현재 결정과 충돌한다. | 제외. 기존 persistence 결정을 불필요하게 변경한다. |

## Consequences

- RDB에는 초안이 아니라 검증·배포된 모든 카드 버전이 남고, 기존 버전은 정상 운영에서 수정·삭제하지 않는다.
- 신규 카드에는 새 식별자와 버전 1을, 기존 카드 변경에는 같은 식별자의 새 버전을 사용한다.
- 카드 활성화와 버전 정의가 분리되어 이전 버전을 보존한 채 신규 경기 적용 버전을 되돌릴 수 있다.
- 활성 버전 전환에는 저장소의 활성 버전 목록 변경, 검증과 새 배포가 필요하며 운영 중 RDB 직접 변경으로 우회할 수 없다.
- 새 버전을 추가하는 작업은 JSON만으로 끝나지 않을 수 있으며 효과 처리·계약·fixture 완결성 검증 비용을 수용한다.
- 상세 resource 경로, 테이블·컬럼, 동기화 use case와 테스트 입력은 후속 TRD·issue·TDD에서 결정한다.
- 외부 운영 UI에서 카드 정의를 직접 편집하거나 RDB를 작성 원천으로 바꿀 때, 또는 여러 애플리케이션 버전이 동시에 서로 다른 활성 목록을 적용해야 할 때 이 결정을 재검토한다.

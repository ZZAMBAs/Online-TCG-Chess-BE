# Trust, Safety & Admin PRD

## 목적

사용자가 다른 사용자를 차단하고 문제 콘텐츠를 신고할 수 있도록 하며, 관리자가 신고를 검토하고 게시글·사용자·금칙어를 운영할 수 있게 한다. 모든 보호 조치는 계정 상태와 실시간 채널에 일관되게 반영되어야 한다.

## 대상 사용자

- 다른 사용자를 차단하거나 차단 해제하려는 일반 사용자
- 채팅·게시글·사용자 프로필을 신고하려는 일반 사용자
- 신고를 검토하고 게시글·사용자·금칙어를 관리하는 관리자

## 범위

### 포함

- 사용자 차단 목록 조회·차단·해제
- 대국 중 상대방 선택 후 차단 버튼으로 즉시 차단
- 차단 관계의 매칭·채팅 전달 제한 연계
- 채팅·게시글·사용자 프로필 신고
- 신고 중복·대상 상태·자동 숨김 연계
- 채팅 신고 증거 스냅샷
- 신고 목록·상세 조회와 관리자 처리
- 게시글 숨김·복구
- 커뮤니케이션 정지·계정정지와 기간·사유·관리자·신고 출처 기록
- 금칙어 목록 조회·추가·수정·삭제
- 관리자 원문 조회와 감사 로그
- 카드 목록 조회

### 제외

- 댓글 신고, 허위 신고 자동 제재, 이의 제기
- 카드 효과·확률·팩 가격 편집
- 고급 관리자 검색·정렬·엑셀 다운로드
- 고급 금칙어 우회 탐지·외부 필터 서비스
- 관리자 권한 부여·회수의 제품 UI

## 주요 사용자 흐름

1. 사용자가 상대를 차단하거나 신고 대상을 선택한다.
2. 서버가 대상 상태·중복 여부·신고 권한을 검증한다.
3. 신고가 저장되고 게시글 신고 5건이면 자동 숨김이 적용된다.
4. 관리자가 신고 목록·상세와 필요한 원문을 조회한다.
5. 관리자가 게시글 복구·숨김 또는 사용자 제재를 처리한다.
6. 제재가 REST·STOMP·기존 세션에 즉시 반영된다.
7. 관리자 행위와 원문 조회가 감사 로그에 남는다.

## 상세 사용자 시나리오

### 시나리오 1. 사용자 차단과 매칭·채팅 제한

- 시작 조건: 사용자가 다른 사용자를 차단할 수 있는 정상 계정이다.
- 사용자 행동: 사용자가 상대를 차단하고 이후 빠른 매치 또는 경기 채팅을 이용한다.
- 시스템 반응: 서버는 자기 자신 차단을 거부하고 차단 관계를 저장한다. 해당 상대를 매칭 후보 탐색과 매칭 확정 검사에서 제외하며, 차단된 상대의 채팅은 차단자에게 전달하지 않는다.
- 사용자에게 보이는 결과: 사용자는 차단 목록과 해제 결과를 확인하고, 차단한 상대와 매칭되지 않으며 채팅을 받지 않는다.
- 완료 조건: 차단 여부가 상대에게 노출되지 않고 중복 차단은 멱등적으로 처리된다.

### 시나리오 2. 대국 중 상대방 차단

- 시작 조건: 두 사용자가 진행 중인 대국에 참가하고 있고 상대방의 채팅이 전달되는 상태다.
- 사용자 행동: 사용자가 대국 화면에서 상대방을 선택하고 `차단`을 누른다.
- 시스템 반응: 서버는 현재 세션 사용자의 차단 요청을 저장하고 즉시 이후 상대방 채팅의 개인 채널 전달을 제한한다.
- 사용자에게 보이는 결과: 사용자는 차단 완료를 확인하고 이후 상대방의 채팅을 받지 않는다. 상대방에게는 차단 사실을 알리지 않는다.
- 완료 조건: 진행 중인 대국은 계속되며 차단은 채팅 전달과 이후 매칭 후보 검증에 적용된다.

### 시나리오 3. 게시글 신고 누적과 자동 숨김

- 시작 조건: `ACTIVE` 게시글이 있고 서로 다른 유효 사용자들이 신고할 수 있다.
- 사용자 행동: 사용자들이 같은 게시글을 신고한다.
- 시스템 반응: 서버는 사용자당 게시글당 한 번만 신고를 허용하고, 서로 다른 유효 사용자 5건이 누적되면 게시글을 `AUTO_HIDDEN`으로 전환한다.
- 사용자에게 보이는 결과: 게시글이 일반 목록과 검색에서 사라지고 신고 수·임계치 진행 상황은 노출되지 않는다.
- 완료 조건: 동시 신고에서도 자동 숨김이 한 번만 적용되고 숨김·삭제 게시글에는 추가 신고를 받지 않는다.

### 시나리오 4. 관리자 신고 검토와 게시글 복구

- 시작 조건: 신고가 저장되었고 관리자가 `ADMIN` 권한을 가진다.
- 사용자 행동: 관리자가 신고 목록·상세를 조회하고 게시글 숨김 유지 또는 복구를 처리한다.
- 시스템 반응: 서버는 관리자 권한을 확인하고 신고 대상과 필요한 원문을 제공하며, 처리 행위와 원문 조회를 감사 로그에 남긴다.
- 사용자에게 보이는 결과: 관리자는 처리 결과와 게시글 상태를 확인하고, 일반 사용자는 복구된 게시글만 다시 조회할 수 있다.
- 완료 조건: 비관리자에게 신고 원문·관리자 기능이 노출되지 않는다.

### 시나리오 5. 커뮤니케이션 정지와 계정정지

- 시작 조건: 관리자가 유효한 신고 또는 운영 사유를 근거로 제재를 처리한다.
- 사용자 행동: 관리자가 기간·사유를 입력해 커뮤니케이션 정지 또는 계정정지를 적용하거나 해제한다.
- 시스템 반응: 서버는 시작·종료 시각, 사유, 처리 관리자, 출처 신고 ID를 저장하고 기존 세션과 REST·STOMP 처리에 즉시 반영한다.
- 사용자에게 보이는 결과: 커뮤니케이션 정지 사용자는 채팅·커뮤니티 작성/수정/삭제/신고가 제한되고, 계정정지 사용자는 제한 안내 외 주요 기능을 사용할 수 없다.
- 완료 조건: 제재 기간이 지나거나 해제되면 해당 제한이 정책에 맞게 변경되고, 제재 이력이 보존된다.

### 시나리오 6. 금칙어 관리

- 시작 조건: 관리자가 금칙어 관리 권한을 가진다.
- 사용자 행동: 관리자가 금칙어 목록을 조회·추가·수정·삭제한다.
- 시스템 반응: 서버는 빈 값·중복·과도하게 긴 값을 거부하고 변경 사유와 변경 전·후 값을 감사 대상으로 저장한다.
- 사용자에게 보이는 결과: 이후 작성·전송되는 채팅과 게시글에 갱신된 금칙어 정책이 적용된다.
- 완료 조건: 기존 게시글에는 금칙어 변경이 자동 소급되지 않는다.

## 상태/권한별 엣지 케이스

- 사용자는 자기 자신을 차단할 수 없고, 자신이 차단한 목록만 조회·관리할 수 있다.
- 자신을 차단한 사용자 목록은 조회할 수 없다.
- 차단 관계는 빠른 매치 후보 탐색·확정과 경기 채팅 전달에 반영된다.
- 신고 대상은 채팅, 게시글, 사용자 프로필로 제한한다.
- 같은 사용자는 같은 게시글을 한 번만 신고할 수 있다.
- 숨김·삭제·만료 대상에는 추가 신고를 허용하지 않는다.
- 사용자 프로필 신고는 자동 숨김 없이 관리자 검토 대상으로만 저장한다.
- 신고자 정보와 차단 관계는 신고 대상자에게 노출하지 않는다.
- 관리자 원문 조회는 권한 검사와 감사 로그를 거친다.
- 커뮤니케이션 정지와 계정정지는 기간제이며 제재 이력을 남긴다.
- 계정 상태 제한은 관리자 역할보다 우선한다.
- 카드 관리자 기능은 카드 목록 조회만 제공하고 효과·확률·팩 가격은 수정하지 않는다.
- 금칙어 변경은 기존 게시글에 소급하지 않는다.
- 일반 채팅 로그는 7일, 신고 증거 스냅샷은 신고 처리 완료 후 30일 기준으로 보관·삭제 또는 비식별화한다.

## 사용자에게 보이는 결과

- 차단·차단 해제와 차단 목록
- 신고 접수 결과와 숨김 처리 결과
- 신고 수나 신고자 정보가 노출되지 않는 게시글 상태
- 관리자에게만 제공되는 신고 상세·원문 검토
- 게시글 숨김·복구 결과
- 제재 사유·기간 안내와 기능 제한
- 금칙어가 적용된 채팅·게시글 표시

## 수용 기준

- 사용자는 다른 사용자를 차단·해제하고 자신의 차단 목록을 관리할 수 있다.
- 사용자는 대국 도중 상대방을 선택해 차단할 수 있다.
- 차단 관계가 매칭과 채팅 전달에 반영되고 상대에게 노출되지 않는다.
- 채팅·게시글·프로필 신고가 저장된다.
- 게시글 신고 5건은 서로 다른 유효 사용자 기준으로 자동 숨김을 발생시킨다.
- 관리자만 신고 목록·상세·원문을 조회하고 게시글 숨김·복구를 수행할 수 있다.
- 커뮤니케이션 정지와 계정정지가 기간·사유·관리자·출처 신고와 함께 기록된다.
- 제재가 REST·STOMP·기존 세션에 즉시 반영된다.
- 관리자는 금칙어 CRUD를 수행할 수 있고 변경 감사 정보가 남는다.
- 카드 관련 관리자 기능은 목록 조회로 제한된다.
- 일반 채팅·신고 증거 보관 정책과 민감 정보 로그 제외 정책이 적용된다.

## 미확정 사항

- 신고 상태값과 관리자 처리 상태의 상세 목록
- 제재 기간 선택지와 해제 사유의 화면 표현
- 신고 증거 스냅샷의 삭제·비식별화 실행 방식
- 금칙어 변경 사유의 필수 입력 형식

관리자 UI의 상세 구성, endpoint와 payload, 감사 로그 schema와 보관 구현은 기술 문서로 넘긴다.

# TCG Online Chess MVP PRD

## 목적

이 문서는 `docs/spec/spec-fixed.md`를 원천으로 하는 TCG Online Chess 백엔드 MVP의 제품 요구사항 허브다. 인증된 사용자가 유효한 덱으로 빠른 매치에 참여하고, 서버 권위의 실시간 1:1 경기를 완료하며, 결과와 기보를 확인하는 전체 제품 범위를 기능별 PRD로 연결한다.

상세 제품 요구사항은 기능별 PRD가 소유한다. 통신·저장·내부 설계와 운영 구현은 이 문서의 범위가 아니며 후속 기술 문서에서 다룬다.

## 원천 문서

- 원천 명세: [`docs/spec/spec-fixed.md`](spec/spec-fixed.md)
- 원천 명세의 큰 번호 섹션과 PRD 산출물 연결: [`docs/traceability.md`](traceability.md)

## MVP 제품 원칙

- 서버가 인증, 권한, 계정 상태, 덱 유효성, 게임 상태, 카드 효과, 타이머, 승패와 결과를 권위적으로 판단한다.
- 클라이언트는 행동 의도와 표시 상태를 전달하며, 클라이언트가 계산한 상태와 결과는 신뢰하지 않는다.
- MVP는 로그인한 사용자의 1:1 빠른 매치를 제공한다.
- 경기 중 공개 정보와 비공개 정보를 구분하고, 비공개 정보는 권한 있는 사용자에게만 제공한다.
- 모든 카드는 발동 가능성 검증, 비용과 카드 소비가 확정되는 발동, 실제 효과 적용을 구분한다.
- 카드가 발동된 뒤 효과가 실패해도 카드·마나·행동 기회 소비와 공개는 유지하고 부분 보드 효과는 남기지 않는다.
- 카드 식별자와 버전은 경기 시작 시 고정하며 과거 기보는 경기 당시 확정된 카드 결과를 현재 카드 효과로 다시 해석하지 않는다.
- 변경 명령은 중복 요청과 동시성 상황에서도 도메인 상태가 한 번만 확정되도록 처리한다.
- 계정 상태와 기능 제재는 일반 사용자 기능보다 우선하며, 관리자 기능은 관리자 역할을 요구한다.
- MVP에서 제외한 기능은 구현하지 않고, 확장 시 필요한 권한·데이터 경계만 명시한다.

기능 간 공통 권한, 오류 안내, 공개 범위와 기록 원칙은 특정 기능의 전용 책임이 아니라 전체 제품 기준으로 추적한다.

## 전체 사용자 여정

1. 사용자가 자체 로그인 또는 OAuth로 가입·로그인하고 필수 설정을 완료한다.
2. 서버가 신규 사용자에게 스타터 컬렉션과 기본 덱을 멱등적으로 지급한다.
3. 사용자가 유효한 덱으로 빠른 매치에 진입한다.
4. 서버가 매칭과 경기 생성을 확정하고, 양쪽에 초기 경기 상태를 제공한다.
5. 플레이어가 서버 권위의 체스·카드·턴·시간 규칙에 따라 경기를 진행하고, 대상 선택 카드는 발동 후 서버가 제공한 후보 중 효과 대상을 선택한다.
6. 사용자는 필요한 범위에서 경기 채팅을 이용하고, 종료 후 결과와 자신의 기보를 확인한다.
7. 사용자는 카드팩·컬렉션·덱을 관리하거나 커뮤니티 기능을 이용할 수 있다.
8. 신고·차단·제재 정책은 경기와 커뮤니티의 보호 기능으로 적용된다.

## MVP 포함 범위

- 자체 이메일/비밀번호 인증과 Google·Kakao OAuth
- 로그인 상태 유지와 안전한 사용자 인증
- 빠른 매치 기반 1:1 PvP
- 표준 체스와 제한된 카드·마나 시스템
- 경기 타이머, 재접속 유예, 시간패·몰수패·항복
- 기본 카드팩, 컬렉션, 덱 편성
- 경기 결과, 전적, 참가자 전용 기보
- 경기 채팅
- 게시글 중심 커뮤니티
- 신고, 차단, 자동 숨김, 관리자 제재
- 서비스 상태와 주요 운영 지표 확인

## MVP 제외 범위

솔로 봇, 친구 대전·초대 코드, 관전, 친구·팔로우, 합의 무승부, 댓글·좋아요, 이미지 업로드, 유료 경제, 공개 랭크·시즌·리더보드, 공개 기보와 모바일 앱 전용 지원은 MVP에 포함하지 않는다.

## 후속 기술 결정

- MVP 카드 12장의 제품 정의와 버전 정책은 `docs/cards`와 원천 명세에 확정되어 있다.
- 카드·카드팩 정의의 저장·배포·변경 이력과 과거 기보 제공 방식은 확정된 제품 정책을 만족하도록 하위 기술·아키텍처 문서에서 결정한다.
- 구체적인 통신 형식, 오류 코드, 저장 구조와 검증 자동화는 PRD 범위에 포함하지 않는다.

## 후속 문서 순서

1. 허브와 기능별 PRD 승인
2. [`docs/traceability.md`](traceability.md)에 원천 섹션과 PRD 연결 기록
3. 승인된 PRD를 기준으로 계약·아키텍처·기술 요구사항과 구현 이슈를 별도 산출

# TCG Online Chess MVP 요구사항 명세

## 1. 개요

TCG Online Chess는 표준 체스 규칙을 기반으로, 경기 중 카드 효과를 제한적으로 결합한 실시간 1:1 온라인 웹 게임이다. 본 문서는 MVP에서 사용자가 경험해야 하는 기능, 규칙, 제한과 수용 기준을 정의한다.

기본 언어는 한국어이며, 서비스 운영 기준 시간대는 KST(Asia/Seoul)로 한다.

## 2. 문서 목적과 범위

이 문서는 TCG Online Chess MVP의 제품 요구사항과 검토 기준을 정의한다. 화면 디자인, 구체적인 통신 데이터 형식, 내부 상태 이름, 저장 구조, 기술 스택과 배포 절차는 하위 계약·기술·아키텍처 문서에서 다룬다.

MVP 완료 기준은 인증된 사용자가 유효한 덱으로 빠른 매치에 진입하고, 서버 권위의 실시간 1:1 경기를 끝까지 완료하며, 결과와 기보가 서버에 저장되는 것이다.

- 사용자는 웹 클라이언트에서 실시간으로 경기에 참여한다.
- MVP는 1:1 온라인 PvP 경험을 완성하는 것을 우선한다.
- 서버는 모든 인증, 권한, 게임 상태, 카드 효과, 타이머, 승패, 보상, 기록을 권위적으로 판단한다.
- 클라이언트는 행동 의도와 표시 상태만 전달하며, 클라이언트가 계산한 게임 상태나 결과는 신뢰하지 않는다.
- UI/UX의 구체적 화면 구성, 디자인 시스템, 애니메이션은 이 문서 범위가 아니다.
- 프론트엔드와 백엔드 사이의 구체적인 통신 형식과 오류 코드는 별도 계약 문서에서 확정한다.
- MVP 제외 기능은 제공하지 않는다. 필요한 경우 이후 확장 시 충돌하지 않도록 권한, 정보와 기록 경계를 남긴다.

## 3. 용어 정의

문서와 사용자 안내에서 같은 개념을 다른 단어로 부르지 않도록 다음 표준 용어를 사용한다.

| 표준 용어 | 동의어/이전 표현 | 정의 |
| --- | --- | --- |
| 사용자 | 회원, 유저 | 서비스 계정을 가진 주체다. 경기 안에서는 플레이어 역할을 가진다. |
| 플레이어 | 경기 참가자 | 특정 경기에서 백 또는 흑으로 배정된 사용자다. |
| 자체 로그인 사용자 | 이메일 로그인 사용자 | 이메일과 비밀번호로 가입한 사용자다. 이메일 인증 완료 전에는 주요 기능 사용이 제한된다. |
| OAuth 사용자 | 소셜 로그인 사용자 | Google 또는 Kakao로 로그인하는 사용자다. 서비스 고유 닉네임 설정과 필수 약관 동의가 필요하다. |
| 닉네임 | 표시 이름, 유저명 | 서비스 내 공개 식별자다. MVP에서는 가입 후 변경할 수 없다. |
| 빠른 매치 | 매치메이킹, PvP 매칭 | 유효한 덱을 가진 사용자를 비공개 MMR 기준으로 자동 매칭하는 MVP의 유일한 매칭 방식이다. |
| 경기 | 게임, 대국, 매치 | 백·흑 두 플레이어가 하나의 체스 보드와 카드 상태를 공유하며 진행하는 단위다. |
| 턴 | 차례 | 한 플레이어가 턴 시작, 이동 전 카드 행동, 말 이동, 이동 후 행동을 순서대로 수행하는 진행 단위다. |
| 턴 시작 | 시작 단계 | 현재 턴 플레이어에게 턴 시작 효과를 적용하는 단계다. MVP에서는 마나 증가만 수행한다. |
| 이동 전 카드 행동 | 선 카드 행동 | 말 이동 전에 카드 1장 사용 또는 아무것도 하지 않음을 선택하는 단계다. |
| 이동 후 행동 | 후 행동, 이동 후 카드 행동 | 말 이동 후 카드 1장 사용, 일반 카드 1장 드로우, 아무것도 하지 않음 중 하나를 선택하는 단계다. |
| 카드 발동 | 카드 사용 확정 | 최초 발동 조건을 통과해 카드·마나·행동 기회를 소비하고 카드를 패에서 제거·공개하는 단계다. |
| 카드 효과 선택 대기 | 효과 선택 단계 | 대상 선택 카드가 발동된 뒤 사용자가 효과 대상을 선택하는 동안 다른 카드 행동과 말 이동을 할 수 없는 상태다. |
| 일반 카드 | 공개 카드 | 경기 중 드로우할 수 있고 상대에게 공개되는 카드다. |
| 레전드 카드 | 궁극기 카드, 비공개 카드 | 경기당 1장만 사용 가능한 1회성 카드이며 사용 전까지 상대에게 공개되지 않는다. |
| 마나 | 비용 자원 | 카드 사용 비용으로 쓰는 경기 중 자원이며 턴 시작과 체스 행동 결과로 획득한다. |
| 덱 | 카드 덱 | 한 경기에서 사용할 레전드 카드 1장과 일반 카드 5장의 조합이다. |
| 유효한 덱 | 사용 가능 덱 | 보유 수량, 카드 종류, 중복 제한과 필수 카드 수를 모두 만족해 빠른 매치에 사용할 수 있는 덱이다. |
| 손패 | 핸드 | 경기 중 플레이어가 현재 보유한 카드 목록이다. 일반 카드 손패는 상대에게 공개된다. |
| 비공개 MMR | 내부 MMR | 빠른 매치에만 쓰며 사용자에게 수치나 티어로 공개하지 않는 실력 점수다. |
| 공개 정보 | 공개 상태 | 양 플레이어에게 모두 보여 줄 수 있는 보드, 일반 카드 손패, 마나와 시간 정보다. |
| 비공개 정보 | 숨김 상태 | 사용 전 레전드 카드와 상대에게 공개하지 않는 정보다. |
| 기보 | 경기 기록, 수순 기록 | 말 이동, 카드 발동과 효과, 드로우, 마나 변화와 주요 결과를 포함한 경기 재현용 기록이다. |
| 시간패 | 시간사 | 개인 시계가 0초가 된 뒤 유효 행동이 완료되지 않아 발생하는 패배다. |
| 몰수패 | 접속 종료 패배 | 연결 끊김 후 제한 시간 안에 재접속하지 못해 발생하는 패배다. |
| 신고 | 리포트 | 대전 채팅, 커뮤니티 게시글, 사용자 프로필을 관리자 검토 대상으로 제출하는 행위다. |
| 차단 | 블록 | 특정 사용자와의 매칭 및 채팅 노출을 제한하는 사용자 설정이다. |
| 제재 | 정지, 관리자 조치 | 관리자가 사용자에게 적용하는 커뮤니케이션 정지 또는 계정정지 조치다. |
| 커뮤니케이션 정지 | 채팅정지, 커뮤니티 작성 제한 | 대전 채팅과 커뮤니티 활동을 함께 제한하는 기능 제재다. |
| 관리자 | 운영자 | 신고 처리, 숨김·복구, 제재와 금칙어 관리 등 제한된 운영 기능을 수행하는 사용자다. |
| 로그인 상태 | 세션 | 로그인 후 서비스가 사용자를 식별하고 인증 상태를 유지하는 기간이다. |
| 재접속 유예 시간 | 복귀 제한 시간 | 연결 끊김 후 몰수패 전까지 복귀할 수 있는 시간이다. MVP에서는 20초다. |
| 감사 기록 | 감사 로그 | 관리자 행위, 카드팩 개봉, 원문 조회와 제재 등 추적이 필요한 행위의 기록이다. |

## 4. 사용자 역할과 권한

기능 접근은 사용자 역할, 계정 상태와 기능 제재를 함께 판단한다. 계정 상태 제한은 관리자 역할보다 우선한다.

### 4.1 역할

| 역할 | 의미 | 허용 기능 |
| --- | --- | --- |
| 일반 사용자 | 서비스 사용자 | 인증, 빠른 매치, 경기, 카드팩, 컬렉션, 덱, 마이페이지, 커뮤니티 게시글, 신고, 차단 |
| 관리자 | 운영 관리자 | 일반 사용자 기능과 신고 검토, 게시글 숨김·복구, 사용자 제재, 카드 목록 조회, 금칙어 관리 |

관리자 권한은 승인된 운영 절차로만 부여·회수하며 모든 변경을 감사 대상으로 기록한다.

### 4.2 계정 상태

| 상태 | 의미 | 허용 기능 | 제한 기능 |
| --- | --- | --- | --- |
| 정상 | 인증과 필수 설정을 완료해 일반 기능을 사용할 수 있는 상태다. | 일반 사용자 기능 | 관리자 기능은 관리자 역할 필요 |
| 인증·설정 미완료 | 자체 로그인 이메일 미인증, OAuth 닉네임 미설정, 필수 약관 미동의 등을 포함한다. | 로그인, 로그아웃, 이메일 인증 메일 재요청, 비밀번호 재설정, 제한 안내 조회, 최소 계정 상태 조회 | 매칭, 경기, 카드팩, 덱, 마이페이지 주요 기능, 커뮤니티 조회·작성·신고, 대전 채팅 |
| 계정정지 | 로그인은 가능하지만 제한 안내와 제재 사유·기간 확인만 허용한다. | 로그인, 제한 안내 조회, 로그아웃 | 매칭, 경기, 카드팩, 덱, 커뮤니티, 채팅, 신고, 마이페이지 주요 기능, 관리자 기능 |
| 탈퇴 | 로그인할 수 없고 모든 사용자 기능이 차단된다. 기존 기록의 표시명은 탈퇴 사용자로 처리한다. | 없음 | 모든 사용자 기능 |

### 4.3 기능 제재

- MVP의 기능 제재는 커뮤니케이션 정지 하나로 단순화한다.
- 커뮤니케이션 정지를 당하면 대전 채팅 전송과 커뮤니티 게시글 작성/수정/삭제/신고가 모두 제한된다.
- 커뮤니티 작성 제한도 동일한 제재로 취급하며, 별도 제재 종류로 분리하지 않는다.
- 커뮤니케이션 정지는 조회성 기능과 경기 진행 자체를 막지 않는다.
- 계정정지는 커뮤니케이션 정지보다 강한 제한이며 대부분의 보호 기능을 차단한다.
- 사용자의 모든 보호 기능 접근과 실시간 행동에서 역할, 계정 상태와 제재를 검증한다.

## 5. 주요 사용자 시나리오

### 5.1 첫 플레이 흐름

1. 사용자는 이메일/비밀번호 또는 OAuth로 가입한다.
2. 자체 로그인 사용자는 이메일 인증을 완료한다.
3. OAuth 사용자는 서비스 고유 닉네임을 설정한다.
4. 모든 가입 방식은 필수 약관 동의를 완료한다.
5. 서버는 신규 사용자에게 스타터 컬렉션과 기본 덱을 한 번만 지급한다.
6. 인증과 필수 설정을 완료한 사용자는 기본 덱으로 빠른 매치에 진입할 수 있다.
7. 서버는 빠른 매치를 통해 상대를 찾고 경기를 생성한다.
8. 사용자는 실시간 경기, 재접속, 종료 처리를 경험한다.
9. 종료된 경기 결과, 전적, MMR, 기보가 서버에 저장된다.
10. 사용자는 다음 경기 또는 덱 편성으로 이어갈 수 있다.

### 5.2 빠른 매치와 경기 시작

1. 사용자는 유효한 덱을 보유한 상태로 빠른 매치 대기열에 진입한다.
2. 서버는 계정 상태, 제재, 덱 유효성, 차단 관계, 비공개 MMR을 검증한다.
3. 서버는 한 사용자당 하나의 대기열 항목만 허용한다.
4. 매칭이 확정되면 경기 생성, 백·흑 무작위 배정, 덱 고정과 초기 상태 생성이 일부만 반영되지 않도록 한 번에 확정한다.
5. 서버는 양쪽 클라이언트에 경기 시작과 확정된 초기 상태를 전달한다.

### 5.3 턴 진행과 카드 사용

1. 서버는 턴 시작 단계에서 마나 지급 등 턴 시작 효과를 적용한다.
2. 현재 턴 플레이어는 이동 전 카드 행동을 선택한다.
3. 카드 사용을 선택하면 서버는 최초 발동 조건과 후보 존재를 검증하고, 성공한 발동의 카드·마나·행동 슬롯 소비와 공개를 확정한다.
4. 대상 선택 카드는 효과 선택 대기에서 사용자가 서버 후보 중 대상을 제출한다. 효과 성공 또는 실패가 확정되면 비용을 유지하고 다음 단계로 진행한다.
5. 플레이어는 합법 말 이동을 수행한다.
6. 프로모션이 필요한 경우 플레이어는 즉시 승격 기물을 선택한다.
7. 서버는 이동 결과, 마나 획득, 체크/체크메이트/무승부/승패 여부를 확정한다.
8. 경기가 끝나지 않았다면 플레이어는 이동 후 행동으로 카드 사용, 일반 카드 드로우, 아무것도 하지 않음 중 하나를 선택한다.
9. 이동 후 카드 효과로 체크메이트가 충족되면 게임은 즉시 종료된다.
10. 서버는 턴 종료 후 다음 플레이어에게 턴을 넘긴다.

### 5.4 카드팩, 컬렉션, 덱 편성

1. 사용자는 MVP에서 기본 카드팩을 0골드로 개봉한다.
2. 서버는 슬롯별 독립 확률로 카드 3장을 지급하고 감사 기록을 남긴다.
3. KST 기준 하루 10회 성공 개봉 제한은 동시 요청으로 초과할 수 없게 적용한다.
4. 중복 카드는 보유 수량으로 누적한다.
5. 사용자는 보유 카드와 덱 중복 제한을 만족하는 덱을 편성한다.

### 5.5 커뮤니티 신고와 관리자 처리

1. 로그인 사용자는 커뮤니티 게시글을 작성, 조회, 수정, 삭제, 검색할 수 있다.
2. 로그인 사용자는 게시글, 사용자 프로필, 대전 채팅을 신고할 수 있다.
3. 게시글 신고가 서로 다른 유효 사용자 5명에게 누적되면 자동 숨김 처리한다.
4. 관리자는 신고 내용을 검토하고 숨김 유지, 복구, 사용자 제재를 결정한다.
5. 커뮤니티 댓글과 좋아요는 MVP에서 제공하지 않는다.

## 7. 인증 및 계정

### 7.1 인증 방식

- MVP는 같은 사이트의 웹 클라이언트를 대상으로 로그인 상태를 안전하게 유지한다.
- 로그인과 실시간 연결 모두 현재 인증 사용자, 권한, 계정 상태와 제재를 검증한다.
- 요청 위조와 허용되지 않은 출처의 접근을 차단해야 한다.
- 로그인과 OAuth 완료, 비밀번호 재설정 후에는 이전 인증 상태를 그대로 재사용하지 않는다.
- 로그아웃, 탈퇴, 계정정지와 비밀번호 재설정이 완료되면 기존 로그인 상태와 실시간 연결을 무효화한다.
- 모바일 앱과 외부 공개 서비스용 별도 인증은 MVP에서 제공하지 않는다.

### 7.2 회원가입 및 로그인

- 자체 로그인은 이메일, 비밀번호, 고유 닉네임을 사용한다.
- 닉네임은 서비스 내 공개 식별자이며 가입 후 변경할 수 없다.
- 닉네임은 2~16자이며 한글, 영문, 숫자, 언더스코어만 허용한다.
- 닉네임 입력값의 앞뒤 공백은 제거한다.
- 닉네임 중복은 대소문자를 구분하지 않고 검사한다.
- 금칙어가 포함된 닉네임은 사용할 수 없다.
- 자체 로그인 사용자는 이메일 인증을 완료해야 주요 기능을 사용할 수 있다.
- OAuth 제공자는 Google, Kakao를 지원한다.
- OAuth 계정은 제공자와 해당 제공자의 사용자 식별자를 함께 사용해 구분한다.
- OAuth 사용자는 별도 이메일 인증을 요구하지 않는다.
- OAuth 사용자는 서비스 고유 닉네임 설정과 필수 약관 동의를 완료해야 주요 기능을 사용할 수 있다.
- 모든 가입 방식은 서비스 이용약관과 개인정보 처리방침 동의를 필수로 요구한다.
- 약관 동의 시각과 약관 버전을 저장한다.

### 7.3 이메일, 비밀번호, 토큰 보안

- 비밀번호는 8~72자로 제한한다.
- 비밀번호는 복구할 수 없는 안전한 방식으로 보호한다.
- 같은 계정 기준 5회 연속 로그인 실패 시 10분 동안 계정을 잠근다.
- 계정 잠금 중에도 비밀번호 재설정 요청은 허용한다.
- 비밀번호 재설정이 완료되면 로그인 실패 횟수와 계정 잠금을 초기화한다.
- 접속 환경 기준 요청 제한은 계정 잠금과 별도로 적용한다.
- 이메일 인증과 비밀번호 재설정에 사용하는 비밀값은 원문으로 보관하지 않고 탈취되더라도 그대로 사용할 수 없게 보호한다.
- 이메일 인증 토큰과 비밀번호 재설정 토큰은 모두 5분 만료 시간을 가진다.
- 이메일 인증 토큰과 비밀번호 재설정 토큰은 1회 사용 후 즉시 무효화한다.
- 새 이메일 인증 토큰을 발급하면 같은 사용자의 기존 미사용 이메일 인증 토큰은 모두 무효화한다.
- 새 비밀번호 재설정 토큰을 발급하면 같은 사용자의 기존 미사용 재설정 토큰은 모두 무효화한다.
- 비밀번호 재설정 요청 응답은 이메일 존재 여부를 노출하지 않는다.
- 이메일 인증 메일과 비밀번호 재설정 메일은 요청 시 발송한다.
- 발송에 실패하면 사용자에게 성공으로 오인되지 않는 결과를 제공한다.
- MVP에서는 실패한 발송을 자동으로 반복하지 않는다.

### 7.4 OAuth와 동일 이메일 충돌

- MVP에서는 자체 로그인 계정과 OAuth 계정을 자동 병합하지 않는다.
- 동일 이메일의 활성 자체 로그인 계정이 이미 존재하면, 새 OAuth 가입은 차단하고 기존 로그인 방식 안내를 제공한다.
- 동일 이메일이더라도 OAuth 제공자나 제공자 사용자 식별자가 다르면 같은 계정으로 자동 연결하지 않는다.
- 탈퇴 계정 재활성화는 기존 가입 방식에 맞는 검증을 거쳐 수행한다.
- 설정 화면에서 자체 로그인 계정과 OAuth 계정을 연동하여 같은 사용자 계정으로 로그인할 수 있게 하는 기능은 이후 확장 후보로 둔다.

### 7.5 계정 탈퇴

- 사용자는 계정 탈퇴를 요청할 수 있다.
- MVP에서는 계정을 즉시 물리 삭제하지 않고 탈퇴 상태로 비활성화한다.
- 탈퇴 사용자는 로그인할 수 없다.
- 전적, 기보, 커뮤니티 기록은 서비스 무결성을 위해 보존하되, 작성자 표시는 탈퇴 사용자로 처리한다.
- 동일 이메일 또는 동일 OAuth 식별자로 재가입하면 새 계정을 만들지 않고 기존 비활성화 계정을 재활성화한다.
- 탈퇴 계정 재활성화 시 기존 닉네임을 그대로 유지한다.

## 8. 실시간 상태 동기화

### 8.1 서버 권위

- 클라이언트는 사용자의 행동 의도만 전달하고, 서버는 게임 상태, 합법 수, 카드 조건, 시간과 승패를 권위적으로 판단한다.
- 클라이언트가 계산한 게임 결과는 신뢰하지 않는다.
- 사용자가 행동을 제출한 뒤 서버 결과가 도착하기 전까지 화면에는 처리 중임을 표시할 수 있다.
- 처리 결과 통지와 실제 경기 상태 갱신의 도착 순서가 바뀌더라도 화면에는 서버가 최종 확정한 상태만 반영한다.
- 동기화가 누락되거나 불일치하면 사용자가 볼 권한이 있는 최신 전체 상태로 복구한다.

### 8.2 중복과 순서

- 같은 행동이 네트워크 문제로 반복 전달되어도 결과와 비용을 한 번만 반영한다.
- 이미 처리한 행동과 의미가 다른 요청을 같은 재시도로 가장하면 거부한다.
- 동시에 도착한 경기 행동은 서버가 정한 하나의 순서로 처리하며, 이미 바뀐 상태를 기준으로 나머지 행동을 다시 검증한다.

### 8.3 공개와 비공개 정보

- 양 플레이어가 공통으로 볼 수 있는 보드, 일반 카드, 마나, 시간과 경기 결과는 동일한 의미로 제공한다.
- 사용 전 레전드, 카드 후보, 사용자의 구체적인 선택과 상세 실패 사유는 권한이 있는 사용자에게만 제공한다.
- 재접속과 기보 조회에서도 같은 공개 범위를 적용한다.

### 8.4 보드 표시와 좌표

- 모든 경기 상태와 기보의 칸은 표준 `a1`~`h8` 좌표를 기준으로 한다.
- 실시간 플레이어 화면은 자신의 진영이 아래쪽에 보이도록 표시한다.
- 화면 방향은 서버의 보드 상태와 좌표를 바꾸지 않는다.
- 카드 방향은 화면 방향이 아니라 카드를 사용한 플레이어의 진영을 기준으로 해석한다.

## 9. 클라이언트 상호작용 요구사항

### 9.1 권한과 신뢰 경계

- 사용자는 자신의 권한과 소유 범위 안에서만 조회와 행동을 수행할 수 있다.
- 서버는 현재 로그인 사용자, 역할, 계정 상태, 제재와 리소스 소유권을 확인한다.
- 사용자가 제출한 사용자 식별자, 게임 결과, 카드 효과와 승패 결과를 권위 정보로 사용하지 않는다.

### 9.2 실시간 경기 경험

- 경기 참가, 행동 제출, 채팅과 재접속 상태 복구는 사용자가 지연이나 새로고침 때문에 결과를 중복 적용하지 않도록 제공한다.
- 공개 경기 정보와 개인 정보는 서로 다른 권한 범위로 제공한다.
- 발동 전 거부된 행동은 사용자가 원인을 이해하고 다시 선택할 수 있게 안내한다.
- 발동 후 카드 효과 실패는 이미 소비된 카드와 비용, 공개 상태가 유지된 결과로 양 플레이어에게 일관되게 보인다.

### 9.3 오류와 복구

- 사용자 오류 안내는 내부 구현, 저장소 정보, 예외 내용과 보안 정책 세부를 노출하지 않는다.
- 최신 상태가 필요한 실패에서는 사용자가 서버의 현재 상태로 복구할 수 있어야 한다.
- 기술적인 오류 분류와 통신 형식은 별도 프론트엔드·백엔드 계약에서 정의한다.

### 9.4 변경 관리

- 프론트엔드와 백엔드는 같은 제품 릴리스에서 함께 변경할 수 있다.
- 사용자에게 보이는 상태 의미, 공개 범위와 행동 결과가 양쪽에서 일치해야 한다.
- 구체적인 통신 방식, 데이터 형식과 호환성 규칙은 별도 계약 문서에서 관리한다.

## 10. 매치메이킹

### 10.1 빠른 매치

- MVP는 빠른 매치만 제공한다.
- 친선 코드, 랭크 모드, 관전 모드는 제공하지 않는다.
- 사용자는 유효한 덱을 보유해야 빠른 매치 대기열에 진입할 수 있다.
- 사용자당 활성 빠른 매치 대기열 항목은 하나만 허용한다.
- 이미 대기열에 있는 사용자가 다시 진입을 요청하면 새 항목을 만들지 않고 현재 대기 상태를 반환한다.
- 매칭 전에는 사용자가 대기열을 취소할 수 있다.
- 매칭 확정 후 취소 요청은 실패하거나 무시되며, 이미 생성된 경기 상태가 우선한다.

### 10.2 대기열 상태

빠른 매치 대기열은 최소한 다음 상태를 가진다.

- 상대 대기
- 사용자 요청 또는 조건 불만족으로 인한 취소
- 양 플레이어 매칭 확정
- 경기 생성 완료
- 서비스 오류 또는 최종 검증 실패

### 10.3 매칭 기준

- 사용자는 비공개 단일 MMR을 가진다.
- 신규 사용자의 최초 MMR은 1400으로 시작한다.
- 빠른 매치는 MMR이 가까운 상대를 우선한다.
- 매칭 시작 후 0~10초 동안은 자신의 MMR 기준 ±50 범위에서 상대를 찾는다.
- 이후 매칭 대기 시간이 10초 늘어날 때마다 허용 범위를 두 배로 확대한다. 즉 10~20초는 ±100, 20~30초는 ±200으로 적용한다.
- MMR은 표준 Elo 기대 승률 공식과 `K=32`를 사용해 갱신한다. 승리는 1, 패배는 0, 무승부는 0.5의 실제 결과값을 사용한다.
- 경기 결과에 따라 비공개 MMR을 갱신한다.
- 공개 티어나 랭크 표시는 MVP에 포함하지 않는다.
- 차단 관계가 있는 두 사용자는 빠른 매치에서 서로 매칭되지 않는다.
- 차단 관계는 후보 탐색 시점과 매칭 확정 직전에 모두 검사한다.
- MVP에서는 대기 시간이 길어져도 차단 관계를 무시하지 않는다.

### 10.4 매칭 확정과 경기 생성

- 대기열 진입 시점과 매칭 확정 직전에 계정 상태, 커뮤니케이션 외 제재, 덱 유효성, 차단 관계를 다시 검증한다.
- 매칭 확정, 경기 생성, 백·흑 배정, 덱 고정과 초기 상태 생성은 일부만 반영되지 않도록 한 번에 확정한다.
- 경기가 생성되면 선택된 덱 구성을 고정한다.
- 경기 생성 후 사용자가 덱을 수정해도 진행 중 경기에는 영향을 주지 않는다.

### 10.5 색 배정

- 서버가 경기 시작 시 서버 난수로 백/흑을 무작위 배정한다.
- MVP에서는 최근 색 이력 보정이나 공평성 누적 보정을 하지 않는다.
- 백이 먼저 둔다.

## 11. 체스 룰

### 11.1 기본 규칙

- 보드는 표준 8x8 체스 보드다.
- 기물 종류, 수량, 초기 배치는 표준 체스를 따른다.
- 기본 승리 조건은 체크메이트다.
- 킹을 직접 잡는 방식으로 승리하지 않는다.
- 자기 턴에는 합법 수 하나를 반드시 둬야 한다.
- 둘 수 있는 합법 수가 없을 때는 체크메이트 또는 스테일메이트로 판정한다.

### 11.2 특수 수

MVP는 표준 체스 특수 규칙을 모두 지원한다.

- 캐슬링
- 앙파상
- 프로모션

프로모션은 표준 말 이동으로 폰이 끝 행에 도달한 경우에만 발생한다. 프로모션이 필요한 말 이동 직후 플레이어는 퀸, 룩, 비숍, 나이트 중 하나를 즉시 선택해야 한다.

카드 효과로 폰이 끝 행에 도달하더라도 프로모션은 발생하지 않는다.

카드 효과로 특정 폰이 앙파상으로 잡힐 수 있는 위치에 도달한 경우, 서버가 앙파상 가능 상태를 기록하면 다음 적용 가능한 말 이동 기회에 앙파상을 허용한다. 카드 효과로 만들어진 앙파상 가능 상태는 다음 적용 가능한 말 이동 기회에만 유효하다.

카드 효과가 킹 또는 룩의 이동 이력에 영향을 줄 수 있는 경우, 캐슬링 권리는 보수적으로 제거한다.

### 11.3 무승부

다음 표준 체스 무승부 조건을 지원한다.

- 스테일메이트
- 기물 부족
- 50수 규칙
- 3회 반복

카드 효과로 보드 상태가 바뀌면 해당 변화도 보드 이력에 반영해 50수 규칙과 3회 반복 판단에 사용한다.

3회 반복은 기물 배치와 소유 진영뿐 아니라 현재 차례와 카드 행동 단계, 캐슬링·앙파상 가능 여부, 이동 이력, 손패·미사용 레전드와 마나처럼 실제로 가능한 행동이 같은 상태만 동일 상태로 인정한다. 남은 시간, 화면의 처리 중 표시와 보드 표시 방향은 동일 상태 판단에 포함하지 않는다.

플레이어 간 합의 무승부는 MVP에서 제외한다.

### 11.4 체크, 체크메이트, 게임 종료

- 말 이동 결과뿐 아니라 카드 효과 적용 후의 보드 결과도 체크, 체크메이트, 스테일메이트, 무승부 판단 대상이다.
- 이동 전 카드 행동으로 체크메이트가 충족되면 말 이동 단계로 가지 않고 즉시 게임을 종료한다.
- 말 이동으로 체크메이트 또는 무승부가 충족되면 이동 후 행동으로 가지 않고 즉시 게임을 종료한다.
- 이동 후 카드 사용으로 체크메이트가 충족되면 즉시 게임을 종료한다.
- 카드 효과는 표준 체스 승패와 무승부 판정을 우회할 수 없다.

### 11.5 항복

- 플레이어는 경기 중 항복할 수 있다.
- 항복은 경기 참가자만 요청할 수 있다.
- 항복은 확인 단계를 거친 뒤 확정된다.
- 항복이 확정되면 즉시 해당 플레이어의 패배로 종료한다.
- 같은 항복 요청이 반복되어도 경기 결과를 중복 확정하지 않는다.

### 11.6 체스 규칙 정확성

- 표준 체스 합법 수와 특수 규칙은 검증 가능한 기준에 따라 일관되게 판정한다.
- 카드 효과가 기물의 위치·소유권·이동 이력을 바꾸더라도 표준 체스 승패와 무승부 규칙을 정확히 적용한다.
- 구체적인 규칙 검증 도구와 상태 표현 방식은 기술 문서에서 결정한다.

## 12. 턴 진행

한 턴은 다음 순서로 진행한다.

1. 턴 시작
2. 이동 전 카드 행동 대기
3. 말 이동 대기
4. 프로모션 선택 대기
5. 이동 후 행동 대기
6. 턴 종료
7. 경기 종료

`프로모션 선택 대기`는 표준 말 이동으로 프로모션이 필요한 경우에만 발생한다. `경기 종료`는 어느 단계에서든 종료 조건이 확정되면 즉시 진입한다.

대상 선택 카드가 이동 전 또는 이동 후 카드 단계에서 발동되면 카드 효과 선택 대기 상태가 된다. 효과가 성공 또는 실패로 끝나기 전에는 다른 카드 행동과 말 이동을 처리하지 않으며, 완료 후에는 발동한 카드 단계의 다음 순서로 진행한다.

### 12.1 턴 시작

- 턴 시작 단계에서는 현재 턴 플레이어에게 턴 시작 효과를 적용한다.
- MVP의 턴 시작 효과는 현재 턴 플레이어의 마나를 1 증가시키는 것이다.
- 단, 최초 턴 플레이어의 최초 턴에는 턴 시작 마나를 지급하지 않는다.
- 마나 증가에는 최대 보유량 10 제한을 적용한다.
- 턴 시작 처리는 서버가 자동으로 확정하며 중복 처리하지 않는다.

### 12.2 이동 전 카드 행동

플레이어는 다음 중 하나를 선택한다.

- 카드 1장 사용
- 아무것도 하지 않음

이 단계에서 발동할 수 있는 카드는 카드 보유·마나·timing·체크 상태의 최초 제한과 발동 가능한 후보 존재 조건을 만족해야 한다. 최종 보드 불변식은 발동 후 효과 적용 단계에서 검증하며 위반하면 소비가 유지되는 효과 실패로 종료한다.

아무것도 하지 않음을 선택하면 이동 전 카드 행동 단계는 소비되며 같은 턴의 이 단계로 돌아갈 수 없다.

이동 전 카드 행동 후 게임이 끝나지 않았다면 말 이동은 필수다.

### 12.3 말 이동

- 말 이동은 항상 필수다.
- 서버는 이동의 합법성을 검증한다.
- 이동 결과 체크, 체크메이트, 무승부, 캡처, 프로모션, 앙파상 여부를 판정한다.
- 프로모션이 필요한 경우 승격 기물 선택이 완료되어야 다음 단계로 넘어간다.
- 말 이동으로 게임 종료 조건이 충족되면 이동 후 행동 없이 즉시 종료한다.

### 12.4 프로모션 선택

- 프로모션은 표준 말 이동으로 폰이 끝 행에 도달한 경우에만 발생한다.
- 프로모션 선택은 말 이동 후 즉시 수행해야 한다.
- 프로모션 선택이 완료되기 전에는 이동 후 행동, 턴 종료, 상대 행동을 처리하지 않는다.
- 프로모션 선택 후 최종 보드를 기준으로 체크, 체크메이트, 무승부를 판정한다.

### 12.5 이동 후 행동

플레이어는 다음 중 하나를 선택한다.

- 카드 1장 사용
- 일반 카드 1장 드로우
- 아무것도 하지 않음

이동 후 행동은 하나만 수행할 수 있다. 카드 사용 또는 드로우 또는 패스가 성공적으로 확정되면 턴을 종료한다.

일반 카드 덱이 비어 있을 때 드로우를 요청하면 실패하며, 이동 후 행동 슬롯은 소비하지 않는다.

### 12.6 체크 상태의 카드 제한

- 자신의 킹이 체크 상태일 때는 체크 해소 가능 후보가 하나 이상 있는 카드만 발동할 수 있다.
- 발동 후 효과 결과가 자신의 킹을 체크 상태로 남기면 부분 보드 효과를 폐기하고 이미 소비된 카드·마나·행동 기회는 유지한 채 효과를 실패로 종료한다.
- 일반 카드와 레전드 카드 모두 이 제한을 따른다.
- 이동 전 카드 행동에서 체크가 발생하거나 해소되는 경우도 최종 보드 기준으로 판단한다.

### 12.7 반복되거나 동시에 전달된 행동

- 같은 단계에서 동시에 요청이 도착하면 서버가 먼저 확정한 요청만 적용한다.
- 나머지 요청은 현재 단계와 상태를 다시 확인해 적용하거나 거부한다.
- 같은 사용자 행동이 반복 전달되어도 비용과 결과를 중복 적용하지 않는다.
- 발동 전 거부된 카드 사용과 실패한 드로우·말 이동은 보드, 손패, 덱, 마나, 턴 단계와 결과를 변경하지 않는다. 발동 후 카드 효과 실패는 발동 소비·공개와 다음 단계 진행을 유지하고 부분 보드 효과만 남기지 않는다.

## 13. 시간 규칙

### 13.1 기본 시간

- 각 플레이어는 개인 시계 30분을 가진다.
- 자기 턴 동안에만 자신의 시간이 감소한다.
- 상대 턴 동안에는 자신의 시간이 감소하지 않는다.
- 턴 시작, 이동 전 카드 행동, 말 이동, 프로모션 선택, 이동 후 행동은 모두 현재 턴 플레이어의 시간을 소비한다.
- 클라이언트 시계는 신뢰하지 않으며, 모든 시간 판정은 서버 기준 시간으로 수행한다.
- 서버는 각 행동의 제한 시각을 기준으로 시간을 계산한다.

### 13.2 시간패

- 개인 시간이 0초가 되면 3초 유예 시간을 적용한다.
- 0초 도달 후 3초 이내에 유효 행동이 서버에서 완료되면 해당 행동을 인정한다.
- 3초를 초과하면 시간패를 확정한다.
- 시간패 판정은 서버가 행동을 확정한 시각을 기준으로 한다.
- 시간패가 확정되면 이후 복귀나 지연 도착 행동으로 되돌릴 수 없다.

### 13.3 연결 끊김과 재접속

- 서버가 해당 사용자의 모든 활성 실시간 연결 종료를 감지하면 연결 끊김으로 본다.
- 같은 사용자의 다른 탭 또는 다른 기기가 아직 연결되어 있으면 연결 끊김으로 보지 않는다.
- 연결 끊김 후 20초 안에 재접속하면 서버는 최신 경기 상태를 재동기화한다.
- 재접속한 사용자는 재접속 전과 동일한 게임 화면에서 최신 경기 상태를 확인한다.
- 연결 끊김 중에도 해당 플레이어의 턴이면 시간은 계속 감소한다.
- 연결 끊김 중 상대 턴이면 해당 플레이어의 개인 시계는 감소하지 않는다.
- 연결 끊김 후 복귀했더라도 이미 시간이 초과되어 시간패가 확정되었으면 시간패는 그대로 유지한다.
- 연결 끊김 후 20초 안에 재접속하지 못하면 몰수패를 확정한다.

### 13.4 종료 사유 경쟁

- 시간패와 몰수패 조건이 경쟁하면 실제 제한 시각이 더 이른 종료 사유를 적용한다.
- 동일 시각으로 판단되면 시간패를 우선한다.
- 경기 종료 사유는 경쟁 상황에서도 한 번만 확정한다.

## 14. 카드 시스템

### 14.1 카드 종류

- MVP 초기 카드 규모는 일반 카드 10장, 레전드 카드 2장이다.
- 일반 카드는 공개형 전술 보조 수단이다.
- 레전드 카드는 비공개 1회성 변수다.
- 실제 MVP 카드 12장의 이미지, 카드명, 상세 효과, 사용 조건, 마나 비용과 UI 문구는 `docs/cards`에 정의한다.
- 카드 문서의 형식과 용어는 `docs/cards/information.md`를 따른다.
- 카드 식별자는 영구적으로 유지하고 효과 또는 조건 변경은 카드 버전과 기보 재현 정책에 따라 관리한다.
- 같은 카드 식별자와 버전의 정의는 배포 후 수정하거나 삭제하지 않는다. 비용, 사용 시기, 발동 조건, 후보 규칙, 효과, 효과 실패 시 소비 또는 공개 의미가 바뀌면 새 버전을 발행한다.
- 경기 생성 시 덱에 적용할 카드 식별자와 버전을 고정하며 이후 카드 변경은 진행 중 경기와 과거 기보에 영향을 주지 않는다.

### 14.2 덱 구성

- 한 덱은 레전드 카드 1장과 일반 카드 5장으로 구성한다.
- 덱은 사용자가 보유한 카드 수량 안에서만 구성할 수 있다.
- 동일 일반 카드는 덱 안에 최대 2장까지 포함할 수 있다.
- 빠른 매치 진입과 경기 생성 직전 모두 덱 유효성을 검증한다.
- 경기 생성 시 덱 구성을 고정한다.

### 14.3 경기 시작 카드 상태

- 경기 시작 시 각 플레이어는 선택한 레전드 카드 1장을 가진다.
- 경기 시작 시 각 플레이어는 일반 카드 덱에서 무작위 일반 카드 1장을 손패로 받는다.
- 일반 카드 덱 순서는 서버가 경기 시작 시 셔플하고 고정한다.
- 클라이언트는 드로우 결과, 셔플 결과, 난수 결과를 제출할 수 없다.
- 일반 카드 손패는 상대에게 공개한다.
- 레전드 카드는 사용 전까지 상대에게 공개하지 않는다.

### 14.4 공개 정보와 비공개 정보

- 상대에게 공개되는 일반 카드 정보는 이름, 비용, 효과 설명, 사용 조건 등 의사결정에 필요한 정보를 포함한다.
- 남은 일반 카드 덱은 장수만 공개하고, 남은 구성과 순서는 공개하지 않는다.
- 사용 전 레전드 카드는 상대가 보는 현재 상태, 실시간 알림과 재접속 후 복구 상태에 포함하지 않는다.
- 사용되지 않은 상대 레전드 카드는 경기 종료 후 기보에서도 공개하지 않는다.

### 14.5 카드 사용

- 이동 전 카드 행동에서 최대 1장, 이동 후 행동에서 최대 1장을 사용할 수 있다.
- 한 턴에 사용할 수 있는 카드는 최대 2장이다.
- 모든 카드는 발동 전 검증, 발동, 효과 적용을 구분한다. 대상 선택 여부와 관계없이 사용자는 카드 발동 전에 명시적으로 확인한다.
- 발동 전에는 경기 참가·차례·카드 행동 단계·카드 보유·마나, 카드별 최초 발동 조건과 발동 가능한 후보 존재를 검증한다. 이 검증이 실패하면 카드, 마나, 행동 기회, 손패, 공개 상태와 보드를 변경하지 않는다.
- 카드 발동이 확정되면 카드와 마나 및 해당 카드 행동 슬롯을 소비하고 카드를 패에서 제거한다. 발동한 일반·레전드 카드는 효과 성공 여부와 관계없이 상대에게 공개하며 레전드는 사용 완료로 처리한다.
- 사용자가 대상을 선택하는 카드는 발동과 비용 소비가 먼저 확정된 뒤 카드 효과 선택 대기 상태가 된다. 서버는 해당 사용자에게 선택 가능한 후보와 선택 개수·중복·순서 조건을 제공한다.
- 카드 효과 선택 대기 중에는 다른 카드 행동과 말 이동을 허용하지 않는다. 별도 선택 제한시간은 두지 않고 현재 플레이어의 개인 시간은 계속 흐르며, 재접속하면 대기 상태와 후보를 다시 확인할 수 있다.
- 클라이언트는 후보와 효과를 계산하지 않고 사용자의 선택만 전달한다. 선택 내용 자체를 해석할 수 없으면 선택 대기를 끝내지 않으며 사용자가 다시 제출할 수 있다.
- 형식은 올바르지만 확정 후보 밖인 선택, 대상 전환 불가 또는 최종 공통 규칙 위반은 효과 실패로 끝난다. 카드·마나·행동 기회를 환급하거나 재선택하지 않으며, 효과의 부분 보드·특수 상태 변경은 전부 남기지 않는다.
- 대상 선택이 없는 카드는 한 번의 사용자 행동 안에서 발동과 효과 성공 또는 실패가 연속해서 확정된다.
- 효과 성공 또는 실패로 카드 처리가 끝나면 이동 전 카드 단계에서는 말 이동으로, 이동 후 카드 단계에서는 턴 종료로 진행한다. 시간패가 먼저 확정되면 미해결 효과보다 경기 종료 상태가 우선한다.
- 상대에게는 발동 카드, 사용자, 소비·사용 완료, 효과 성공 여부와 관찰 가능한 결과만 보여 준다. 후보, 사용자의 구체적인 선택과 상세 실패 사유는 카드 사용자에게만 보여 준다.

### 14.6 카드 효과 불변식

카드 효과는 다음 규칙을 깨뜨릴 수 없다.

- 양쪽 킹은 정상적으로 존재해야 한다.
- 상대 킹을 직접 제거할 수 없다.
- 자신의 킹이 체크 상태로 남는 결과를 만들 수 없다.
- 표준 체스의 체크메이트, 스테일메이트, 무승부 판정을 우회할 수 없다.
- 카드 효과 후 체크메이트가 되면 즉시 게임이 종료된다.
- 카드 효과로 폰이 끝 행에 도달해도 프로모션은 발생하지 않는다.
- 카드 효과가 앙파상 가능 상태를 만들 수 있으며, 서버가 이를 기록하면 다음 적용 가능한 말 이동에서만 유효하다.
- 카드 효과가 캐슬링 권리에 영향을 줄 수 있으면 보수적으로 캐슬링 권리를 제거한다.
- 카드 사용자 진영 기준 방향은 다음과 같이 계산한다.
  - 백의 왼쪽, 오른쪽, 왼쪽 앞, 오른쪽 앞은 표준 보드의 열·행 변화량으로 각각 `(-1,0)`, `(+1,0)`, `(-1,+1)`, `(+1,+1)`이다.
  - 흑은 각각 `(+1,0)`, `(-1,0)`, `(+1,-1)`, `(-1,-1)`이다.
  - 방향은 경기에서 확정된 카드 사용자의 색상을 기준으로 하며 클라이언트가 표시한 색상이나 보드 방향에 따라 달라지지 않는다.
- 카드 효과 자체는 50수 규칙의 진행 횟수를 추가로 늘리지 않는다.
- 카드 효과로 폰이 이동하거나 기물이 잡히면 50수 규칙 진행 횟수를 초기화한다. 그 외 위치 변경, 교환, 소유권 변경과 발동 후 효과 실패는 현재 값을 유지하고, 같은 턴의 표준 말 이동은 일반 체스 규칙대로 계산한다.
- 폰 일제 전진 카드는 효과 시작 상태에서 한 번도 이동하지 않았고 전방 칸이 비어 있는 폰만 대상으로 삼는다. 막힌 폰은 제외하고 나머지를 동시에 전진시킨다. 대상 폰이 하나도 없으면 발동할 수 없고, 발동 후 전체 적용 결과가 공통 규칙을 위반하면 어떤 폰 이동도 남기지 않되 발동 비용과 공개는 유지한다.
- 기물 강제 교환 카드는 상대 킹과 퀸을 제외한 서로 다른 살아 있는 상대 기물 두 개를 순서 의미 없이 선택한다. 교환된 룩의 캐슬링 권리를 제거하고, 교환된 폰 관련 기존 앙파상 권리를 제거하며 새 앙파상 권리를 만들지 않는다. 끝 행에 도달한 폰은 프로모션하지 않는다.
- 왕의 포섭 카드는 상대 킹을 제외한 살아 있는 상대 기물 중 자신의 킹과 맨해튼 거리가 최소인 후보를 서버가 계산하고, 카드 사용자가 그 후보 중 한 개를 선택한다. 비킹 후보가 없으면 카드를 발동할 수 없고 비용을 소비하지 않는다. 발동 후 후보 밖 기물을 선택하거나 전환할 수 없으면 이미 소비된 비용을 유지한 채 효과를 실패로 종료한다.
- 왕의 포섭 카드로 전환된 기물은 종류·위치·기존 이동 이력을 유지하고 소유 진영만 변경한다. 이후 이동·공격·폰 전진 방향과 프로모션 도달 행은 새 진영 기준을 따르며 이동하지 않은 상태로 초기화하지 않는다.
- 전환된 룩에는 캐슬링 권리를 부여하지 않고 전환된 폰 관련 기존 앙파상 권리를 제거하며 새 권리를 만들지 않는다. 전환 시 폰이 끝 행에 있어도 즉시 프로모션하지 않는다.
- 왕의 포섭으로 전환된 이동 이력이 없는 폰의 2칸 전진은 보존된 이동 이력과 새 진영의 시작 행 조건을 모두 만족할 때만 허용한다.
- 위치 교환과 소유권 변경을 포함한 각 효과 적용 결과는 중간 불법 상태를 노출하지 않고 최종 보드에서 체크·체크메이트·스테일메이트·무승부를 다시 판정한다. 효과 실패 시에는 발동 소비 상태만 유지하고 후보 보드 변경은 전부 폐기한다.

### 14.7 드로우

- 이동 후 행동에서 일반 카드 1장을 드로우할 수 있다.
- 드로우는 서버가 고정된 일반 카드 덱 순서에 따라 처리한다.
- 일반 카드 덱이 비어 있으면 드로우 요청은 실패한다.
- 실패한 드로우는 이동 후 행동 슬롯을 소비하지 않는다.

### 14.8 카드 정의 범위

- MVP에서는 `docs/cards`에 확정된 12장만 사용한다.
- 관리자 편집과 자동 카드 생성은 MVP에서 제공하지 않는다.
- 카드 효과 적용 뒤에는 공통 체스 규칙과 게임 종료 조건을 다시 판정한다.

## 15. 마나 시스템

### 15.1 기본 규칙

- 각 플레이어의 시작 마나는 0이다.
- 최대 보유 마나는 10이다.
- 최대 보유량을 초과하는 마나는 버린다.
- 마나는 서버가 계산하며 클라이언트 입력을 신뢰하지 않는다.
- 모든 마나 변화는 일부만 반영되지 않도록 적용하고 변화 사유를 기록한다.

### 15.2 획득 규칙

- 최초 턴 플레이어의 최초 턴에는 턴 시작 마나를 받지 않는다.
- 그 이후 모든 턴 시작마다 현재 턴 플레이어는 마나 1을 얻는다.
- 합법적인 말 이동 자체는 마나를 주지 않는다.
- 기물을 잡으면 이동한 플레이어가 마나 1을 얻는다.
- 앙파상은 캡처 보상 1에 추가 앙파상 보상 1을 더해 총 2마나를 얻는다.
- 체크를 건 플레이어는 마나를 얻지 않는다.
- 각 플레이어는 자신이 체크를 2회 당할 때마다 마나 1을 얻는다.
- 체크메이트도 체크를 당한 1회로 계산한 뒤 게임을 종료한다.
- 한 행동에서 더블 체크가 발생해도 체크를 당한 횟수는 1회로 계산한다.
- 카드 효과로 발생한 체크도 체크를 당한 횟수에 포함한다.

### 15.3 예시

- 턴 시작 마나 1, 캡처 1이면 총 2마나를 얻는다.
- 앙파상 캡처는 캡처 1과 앙파상 추가 1을 합산해 총 2마나를 얻는다.
- 어떤 플레이어가 체크를 두 번째로 당하면, 체크를 당한 플레이어가 마나 1을 얻는다.
- 체크를 건 플레이어는 체크 자체로 마나를 얻지 않는다.

## 16. 카드팩 및 컬렉션

### 16.1 골드

- 서비스 재화 이름은 골드다.
- MVP에서는 골드 잔액 필드를 미래 확장용으로 둘 수 있다.
- MVP 기본 카드팩 가격은 0골드다.
- 출석 보상, 경기 보상, 골드 구매는 MVP 이후 확장 대상으로 둔다.

### 16.2 카드팩

- MVP에서는 기본 카드팩 1종만 제공한다.
- 카드팩 1개는 카드 3장을 지급한다.
- 각 카드 슬롯은 독립적으로 레전드 여부를 판정한다.
- MVP의 레전드 확률은 각 슬롯당 1%다.
- 따라서 카드팩 1개에서 레전드 카드가 0장, 1장, 2장, 3장 모두 나올 수 있다.
- 레전드 슬롯은 활성 레전드 카드 중 균등 무작위로 지급한다.
- 일반 슬롯은 활성 일반 카드 중 균등 무작위로 지급한다.
- MVP 카드팩은 0골드로 개봉할 수 있다.
- 사용자별 카드팩 개봉은 KST 기준 하루 10회 성공 개봉으로 제한한다.
- 일일 제한은 KST 00:00:00에 초기화한다.
- 신규 카드팩 종류 추가는 MVP 이후 확장 대상으로 둔다.

### 16.3 개봉 처리

- 카드팩 개봉 결과는 서버가 결정한다.
- 클라이언트는 지급될 카드, 확률이나 개봉 결과를 정할 수 없다.
- 카드팩 개봉은 일일 횟수, 골드 확인·차감, 카드 지급과 감사 기록이 일부만 반영되지 않도록 한 번에 확정한다.
- 0골드 카드팩이라도 동시 요청으로 하루 10회 성공 제한을 넘을 수 없어야 한다.
- 같은 개봉 행동이 반복 전달되어도 카드와 성공 횟수를 중복 지급하지 않는다.

### 16.4 중복 카드

- 이미 보유한 카드를 다시 얻으면 보유 수량이 증가한다.
- 중복 카드를 자동 환급하거나 결과를 보정하지 않는다.
- 천장, pity, 확률 보정은 MVP에서 제공하지 않는다.
- 덱 편성 시 보유 수량과 덱 중복 제한을 함께 검증한다.

### 16.5 스타터 컬렉션

- 신규 사용자는 즉시 플레이 가능한 스타터 컬렉션과 기본 덱을 지급받는다.
- 스타터 덱은 기본 레전드 카드 1장과 일반 카드 5장으로 구성한다.
- 스타터 카드는 사용자별로 한 번만 지급한다.
- 이미 지급된 사용자에게 중복 지급하지 않는다.

### 16.6 카드팩 감사 기록

카드팩 개봉 결과는 감사 기록으로 보존한다.

- 개봉 행동을 구분하는 정보
- 사용자
- 카드팩 종류
- 개봉 시각
- KST 기준 날짜
- 사용 골드와 개봉 전/후 골드
- 결과 카드 3장과 슬롯 순서
- 각 카드의 일반/레전드 여부
- 확률 버전
- 개봉 전/후 보유 수량
- 중복 여부
- 실패한 경우 실패 사유

## 17. 전적 및 기보

### 17.1 전적 요약

마이페이지에서는 자신의 전적만 제공한다.

- 총 종료 경기 수
- 승리 수
- 패배 수
- 무승부 수
- 승률
- 최근 경기 목록

승률은 승리 수를 종료 경기 수로 나눈 값이다. 무승부는 분모에 포함한다. 사용자에게는 소수점 첫째자리까지 표시하고, 둘째자리에서 반올림한다.

최근 경기 목록은 종료 시각 내림차순으로 정렬하고 한 번에 최대 10개씩 무한 스크롤로 조회한다.

최근 경기 목록은 최소한 다음 정보를 포함한다.

- 상대 닉네임
- 백/흑 여부
- 경기 시작 시각
- 경기 종료 시각
- 종료 사유
- 결과

전적에는 서버가 최종 종료한 경기만 반영한다. 진행 중, 취소, 미시작, 무효 경기는 제외한다.

### 17.2 경기 종료 저장

- 경기 종료는 종료 사실, 경기 결과, 플레이어 전적, MMR 갱신과 기보 확정을 일부만 반영하지 않도록 한 번에 처리한다.
- 경기 결과, 종료 사유, 종료 시각은 서버만 확정한다.
- 서비스 장애로 무효 처리된 경기는 전적과 MMR에 반영하지 않는다.
- 무효 경기는 무효 처리 직전까지의 확정 경기 기록과 기보를 보존한다. 기보 마지막 결과 표시는 승자 대신 무효 경기임을 표시한다.

### 17.3 기보 저장

- 기보는 서버가 확정한 경기 행동과 결과를 순서대로 보존한 기록을 기준으로 구성한다.
- 말 이동, 카드 발동, 효과 선택 대기, 효과 적용 성공·실패, 카드 드로우, 마나 변화, 턴 단계, 보드 상태와 종료 결과를 보존한다.
- 기보 상세는 턴별로 구분하고, 각 턴에서 시작 마나 회복·시작 덱 드로우·기물 이동 전 행동·기물 이동·기물 이동 후 행동을 단계별로 표시한다. 시작 덱 드로우는 조회자에게 공개 가능한 카드만 표시하며, 상대의 비공개 정보는 숨긴다.
- 특별한 행동이나 종료 조건이 있는 경우 기본 단계 외의 상세 결과를 추가로 표시할 수 있다.
- 기보에서 승·무·패가 결정된 경우 승자를 표시한다.
- 기보는 MVP에서 해당 경기 당사자만 조회할 수 있다.
- 관리자가 기보를 조회하는 권한은 MVP에서 별도로 부여하지 않는다.
- 공개 기보 기능은 이후 확장 대상으로 둔다.

### 17.4 기보 공개 범위

- 기보는 조회자의 권한에 맞는 정보만 보여 준다.
- 사용되지 않은 상대 레전드 카드는 경기 종료 후에도 공개하지 않는다.
- 발동된 상대 레전드는 효과 성공 여부와 관계없이 사용된 카드로 공개하되, 카드 사용자에게만 보이는 후보·구체적인 선택·상세 실패 사유는 상대 기보에서 숨긴다.
- 난수 생성의 비공개 정보, 운영 기록, 비공개 MMR, 접속 정보와 차단 관계는 기보에 포함하지 않는다.
- 탈퇴 사용자는 탈퇴 사용자로 표시한다.
- 차단 관계가 있어도 과거 경기 기보 접근 권한은 제거하지 않는다.
- 단, 기보 응답에서 차단 여부를 노출하지 않는다.

## 18. 대전 채팅

### 18.1 기본 정책

- 대전 중 두 플레이어는 자유 텍스트 채팅을 사용할 수 있다.
- 채팅 전송과 구독은 경기 참가자만 가능하다.
- 서버는 클라이언트가 보낸 senderId, 닉네임, 권한 값을 신뢰하지 않는다.
- 커뮤니케이션 정지 사용자는 채팅을 전송할 수 없다.
- 계정정지 사용자는 채팅을 포함한 주요 기능을 사용할 수 없다.
- 경기 종료 후 채팅 전송은 거부한다.
- 채팅 로그는 일반 보관본 기준 7일간 보관한다.

### 18.2 차단과 전달

- 채팅은 수신자별 개인 채널로 전달한다.
- 서버는 차단한 사용자의 채팅을 차단자에게 전달하지 않는다.
- 차단 여부는 송신자에게 노출하지 않는다.
- 차단은 채팅 저장 여부와 신고 증거 저장 정책을 우회하지 않는다.

### 18.3 메시지 형식과 제한

- 채팅은 plain text만 허용한다.
- 빈 문자열과 공백만 있는 메시지는 거부한다.
- 채팅 메시지 최대 길이는 100자다.
- 사용자 화면과 관리자 화면 모두 HTML escape를 적용한다.
- 채팅 전송은 사용자와 경기 기준으로 1초 3건, 10초 10건까지 허용한다.

### 18.4 금칙어 처리

- 서버는 기본 금칙어 필터를 가진다.
- 금칙어 사전은 운영 중 관리할 수 있도록 보존한다.
- MVP의 금칙어 매칭은 대소문자 구분 없는 단순 부분 문자열 포함 방식으로 처리한다.
- 중첩 금칙어는 긴 단어 우선으로 처리한다.
- 모든 금칙어 출현 위치를 마스킹한다.
- 공백, 특수문자 우회 탐지, 초성 분리, Unicode confusable, 고급 패턴 매칭은 MVP 이후 확장 대상으로 둔다.
- 사용자에게 송출되는 채팅 메시지는 금칙어가 마스킹된 내용이다.
- 금칙어는 금칙어 전체 길이만큼 `*`로 치환한다.
- 원문과 마스킹 결과를 모두 보존한다.
- 관리자 원문 조회는 권한 검사를 거치고 감사 로그를 남긴다.

### 18.5 신고 증거

채팅 신고 증거 사본은 일반 채팅 기록의 보관 기간과 별도로 관리한다.

- 메시지 식별 정보
- 경기 식별 정보
- 발신자
- 신고자
- 원문
- 마스킹 결과
- 발신 시각
- 신고 사유

신고 증거는 신고 처리 완료 후 30일이 지나면 삭제 또는 비식별화한다.

## 19. 커뮤니티

### 19.1 접근 권한

- 커뮤니티 조회와 작성은 모두 로그인 사용자만 가능하다.
- 계정정지 사용자는 커뮤니티를 사용할 수 없다.
- 커뮤니케이션 정지 사용자는 게시글 작성, 수정, 삭제, 신고를 할 수 없다.
- 커뮤니티 댓글, 게시글 좋아요, 댓글 좋아요는 MVP에서 제외한다.

### 19.2 게시글

게시글은 다음 기능을 제공한다.

- 작성
- 목록 조회
- 상세 조회
- 수정
- 삭제
- 검색
- 신고
- 자동 숨김
- 관리자 숨김/복구

이미지 업로드는 MVP에서 제외한다.

### 19.3 게시글 상태

게시글은 최소한 다음 상태를 가진다.

- 정상 노출
- 작성자 또는 관리자가 삭제
- 신고 기준 충족으로 자동 숨김
- 관리자가 수동 숨김

숨김 또는 삭제된 게시글은 일반 목록과 검색에서 제외한다. 직접 접근은 거부하거나 숨김 안내만 제공한다.

### 19.4 작성, 수정, 삭제

- 게시글 작성자는 자신의 게시글을 수정하거나 삭제할 수 있다.
- 관리자는 게시글을 숨김 또는 복구할 수 있다.
- 사용자와 관리자 화면 모두 plain text를 HTML escape하여 표시한다.
- 제목과 본문은 원문과 마스킹 결과를 저장한다.
- 사용자 화면에는 마스킹 결과만 보여 준다.
- 금칙어 사전 변경은 기존 게시글에 자동 소급 적용하지 않는다.
- 게시글 생성 또는 수정 시점의 금칙어 사전을 기준으로 마스킹한다.

### 19.5 검색

- 제목, 내용, 작성자 닉네임을 대상으로 검색한다.
- MVP 검색은 제목, 내용과 작성자 닉네임의 단순 일치 검색을 제공한다.
- 검색 결과는 정상 노출 게시글만 포함한다.
- 사용자 화면에는 마스킹 결과만 보여 준다.
- 전문 검색 엔진 또는 고급 랭킹은 이후 확장 대상으로 둔다.

### 19.6 신고 및 자동 숨김

- 신고 대상은 게시글만이다.
- 한 사용자는 같은 게시글을 한 번만 신고할 수 있다.
- 숨김 또는 삭제된 게시글은 신고할 수 없다.
- 서로 다른 유효 사용자 5명이 신고하면 게시글은 자동 숨김 상태가 된다.
- 자동 숨김은 한 번만 적용한다.
- 신고 수와 임계치 진행 상황은 일반 사용자에게 노출하지 않는다.
- 신고 접수, 신고 수 계산과 자동 숨김은 일부만 반영되지 않도록 처리한다.
- 허위 신고 자동 제재, 이의 제기 절차는 MVP에서 제외한다.

### 19.7 커뮤니티 금칙어 처리

- 게시글 원문은 관리자 검토를 위해 보존한다.
- 사용자 화면에는 금칙어가 마스킹된 내용을 표시한다.
- 관리자 검토 시 원문을 확인할 수 있다.
- 관리자 원문 조회는 권한 검사를 거치고 감사 로그를 남긴다.
- 마스킹 방식은 대전 채팅 금칙어 처리와 동일하게 금칙어 전체 길이만큼 `*`로 치환한다.

## 20. 신고, 차단, 관리자

### 20.1 사용자 차단

- 사용자는 다른 사용자를 차단할 수 있다.
- 자기 자신은 차단할 수 없다.
- 같은 차단 요청이 반복되어도 차단 관계를 중복 생성하지 않는다.
- 사용자는 자신이 차단한 목록만 조회하고 관리할 수 있다.
- 자신을 차단한 사용자 목록은 조회할 수 없다.
- 차단 해제를 제공한다.
- 차단 관계가 있는 사용자는 빠른 매치에서 서로 매칭되지 않는다.
- 매칭 후보 탐색과 매칭 확정 직전에 모두 차단 관계를 검사한다.
- 차단한 사용자의 대전 채팅은 서버가 차단자에게 전달하지 않는다.
- 차단 여부는 상대에게 노출하지 않는다.
- 대국 도중에도 사용자는 상대방을 선택해 차단할 수 있으며, 차단 즉시 상대방의 채팅은 차단자에게 전달하지 않는다.

### 20.2 신고

MVP 신고 대상은 다음으로 제한한다.

- 대전 채팅
- 커뮤니티 게시글
- 사용자 프로필

커뮤니티 댓글 신고는 MVP에서 제외한다.

- 채팅 신고는 신고 대상 메시지와 신고 증거를 기준으로 저장한다.
- 게시글 신고는 사용자당 게시글당 한 번만 허용한다.
- 게시글 신고 5건은 서로 다른 유효 사용자 기준으로 계산한다.
- 사용자 프로필 신고는 관리자 검토 대상으로만 저장하며 자동 숨김은 없다.
- 숨김, 삭제, 만료된 대상에는 추가 신고를 허용하지 않는다.
- 신고 대상자에게 신고자 정보나 차단 관계를 노출하지 않는다.

### 20.3 관리자 기능

MVP는 최소 관리자 기능을 제공한다.

- 신고 목록 조회
- 신고 상세 조회
- 게시글 숨김 및 복구
- 사용자 커뮤니케이션 정지
- 사용자 계정정지
- 사용자 제재 이력 조회
- 카드 목록 조회
- 금칙어 목록 조회
- 금칙어 추가, 수정, 삭제

카드 효과, 확률, 팩 가격 수정은 MVP 관리자 기능에 포함하지 않는다.

관리자 화면은 기능 중심의 단순 목록, 상세, 처리 화면으로 제공한다.

관리자 목록 화면은 MVP에서 다음 검색 및 필터만 제공한다.

- 신고 목록: 상태, 대상 유형, 신고일 범위
- 사용자 제재 관련 검색: 닉네임 또는 사용자 식별자
- 카드 목록: 카드명, 카드 종류
- 금칙어 목록: 키워드

복잡한 정렬, 다중 조건 조합, 엑셀 다운로드는 MVP에서 제외한다.

### 20.4 제재

- 커뮤니케이션 정지는 기간제로 적용한다.
- 계정정지도 기간제로 적용한다.
- 제재는 시작 시각, 종료 시각, 사유, 처리 관리자, 출처 신고 ID를 저장한다.
- 제재는 이후 기능 요청과 기존 로그인·실시간 연결에 즉시 반영한다.
- 계정정지 사용자는 로그인 이후 제한 안내 외 주요 기능 사용이 제한된다.
- 커뮤니케이션 정지는 채팅 전송과 커뮤니티 게시글 작성/수정/삭제/신고를 함께 제한한다.
- 관리자 권한은 승인된 운영 절차로만 부여한다.

### 20.5 관리자 감사

다음 관리자 행위는 감사 로그를 남긴다.

- 신고 상세 조회와 원문 조회
- 게시글 숨김 및 복구
- 커뮤니케이션 정지
- 계정정지
- 제재 해제
- 금칙어 추가, 수정, 삭제
- 카드 관련 관리자 조회

금칙어는 빈 값, 중복 값, 과도하게 긴 값을 거부한다. 금칙어 변경 시 변경 사유와 변경 전/후 값을 저장한다.

## 21. 마이페이지 및 설정

### 21.1 마이페이지

마이페이지는 사용자 자신의 데이터 조회와 설정 변경 허브다. 각 기능의 기존 권한과 공개 범위를 따른다.

마이페이지는 다음 기능을 제공한다.

- 내 카드 컬렉션 조회
- 덱 편성 및 변경
- 현재 유효 덱 확인
- 전적 요약 조회
- 최근 경기 조회
- 당사자 기보 조회
- 사용자 설정 변경

모든 마이페이지와 설정 기능은 현재 로그인 사용자에게만 동작하며 다른 사용자의 정보를 조회하거나 수정할 수 없다.

닉네임 변경은 제공하지 않는다.

### 21.2 사용자 설정

사용자는 다음 설정만 변경할 수 있다.

- 대전 채팅 표시 여부
- 인앱 알림 수신 여부
- 이메일 알림 수신 여부

허용하지 않은 설정 항목은 거부한다. 역할, 계정 상태, 이메일, 닉네임과 MMR 같은 민감 정보는 사용자 설정으로 변경할 수 없다.

대전 채팅 표시 여부는 로컬 표시 선호도다. 채팅 전송 권한, 서버 저장, 신고, 차단 필터에는 영향을 주지 않는다.

인앱 알림 수신 여부는 선택 알림에만 적용한다. 제재, 보안, 관리자 중요 공지 등 필수 안내는 수신 설정과 무관하게 전달할 수 있다.

이메일 알림 수신 여부는 향후 서비스 이메일에 적용한다. 이메일 인증과 비밀번호 재설정 메일에는 적용하지 않는다.

### 21.3 알림

- MVP의 서비스 알림은 인앱 알림 중심으로 제공한다.
- MVP 인앱 알림 대상은 신고 처리 결과, 관리자 공지, 제재/제한 안내다.
- 커뮤니티 댓글과 좋아요 알림은 MVP에서 제외한다.
- 자체 로그인 이메일 인증과 비밀번호 재설정은 이메일로 처리한다.
- 계정정지 사용자는 주요 마이페이지와 설정 기능을 사용할 수 없다.
- 탈퇴 사용자는 로그인할 수 없고 알림도 받을 수 없다.

### 21.4 이후 설정 확장

- 설정 화면에서 자체 로그인 계정과 OAuth 계정을 연동하는 기능은 MVP 이후 확장 후보로 둔다.
- 연동 기능이 추가되면 자체 로그인과 연동된 OAuth 로그인 모두 같은 사용자 계정으로 로그인되어야 한다.

## 22. 운영 및 보호 장치

### 22.1 요청 제한

다음 민감 기능에는 사용자, 접속 환경 또는 기능 단위 요청 제한을 적용한다.

- 로그인: IP + 로그인 식별자
- 회원가입: IP + 이메일
- 이메일 인증 요청: IP + 이메일
- 비밀번호 재설정 요청: IP + 이메일
- 채팅 전송: 사용자 + 경기, 1초 3건, 10초 10건
- 게시글 작성: 사용자 기준과 보조 IP 기준
- 카드팩 개봉: 사용자 + KST 날짜 기준 하루 10회와 동시 요청 방지
- 반복된 인증 실패, 과도한 실시간 행동과 과도한 메시지 크기

요청 제한을 초과하면 사용자가 다시 시도할 수 있는 시점을 안내한다. 카드팩 일일 제한은 동시 요청으로 우회할 수 없어야 한다.

### 22.2 관측

- 운영자는 서비스 건강 상태, 오류, 실시간 연결, 매칭, 경기, 인증, 요청 제한, 카드팩, 채팅, 신고와 관리자 제재 현황을 관측할 수 있어야 한다.
- 외부에는 최소한의 서비스 상태만 공개하고 상세 운영 정보는 승인된 운영 주체에게만 제공한다.
- 구체적인 관측 도구와 제공 방식은 기술·운영 문서에서 결정한다.

### 22.4 운영 기록과 보안

운영 기록에는 요청 추적, 처리 결과, 소요 시간과 오류 분류처럼 문제 분석에 필요한 정보를 남긴다. 다음 값은 일반 운영 기록에 남기지 않는다.

- 비밀번호
- 인증 토큰
- 비밀번호 재설정 토큰
- 이메일 인증 토큰
- 세션 쿠키
- 요청 위조 방지용 비밀값
- 이메일 원문
- 채팅/게시글 원문
- 난수 생성의 비공개 값

관리자 원문 조회, 제재, 숨김/복구, 금칙어 변경, 요청 제한과 권한 거부는 감사 또는 보안 기록 대상이다.

### 22.5 주요 운영 지표

최소한 다음 운영 지표를 확인할 수 있어야 한다.

- 현재 실시간 연결 수
- 매칭 대기 사용자 수와 진행 중 경기 수
- 경기 생성 수와 종료 사유별 수
- 실시간 처리 실패, 로그인 실패와 요청 제한 차단 수
- 카드팩 개봉 수와 일일 제한 초과 시도 수
- 채팅 전송 차단, 신고 생성과 관리자 제재 수
- 카드 사용, 체크메이트 기물, 카드별 경기 투입·승리 통계
- 카드별 통계는 최소한 카드 사용 수, 경기 투입 수, 승리 수와 이를 기반으로 한 승률 계산을 지원해야 한다.
- 체크메이트 통계는 체크메이트를 만든 기물 유형별 횟수를 지원해야 한다.
- 사용자 요청 처리 시간과 서버 오류 수

운영 지표에는 사용자 식별자, 이메일, 경기 식별자, 접속 주소와 인증 비밀값처럼 개인 식별 또는 과도하게 세분화된 값을 포함하지 않는다.

## 23. 기록과 보존 원칙

### 23.1 경기 상태 저장

- 진행 중 경기의 현재 상태는 서버가 권위적으로 유지한다.
- 확정된 경기 행동과 결과는 순서와 중복 없이 보존해 재접속, 기보 재생, 감사와 문제 분석에 사용할 수 있어야 한다.
- 구체적인 상태 저장 방식과 복구 구조는 기술·아키텍처 문서에서 결정한다.

### 23.2 서버 재시작과 활성 경기

- 서비스 장애 후 진행 중 경기의 완전한 복구는 MVP 필수 요구사항이 아니다.
- 서비스 장애로 경기를 계속할 수 없으면 해당 경기는 무효 처리한다.
- 서버 장애로 게임을 계속할 수 없으면 사용자에게 `서버 장애로 인해 게임을 진행할 수 없습니다. 게임을 종료합니다.` 문구를 표시하고 게임을 종료한다.
- 무효 처리된 경기는 전적과 MMR에 반영하지 않는다.
- 종료된 경기의 결과와 기보는 보존되어야 한다.
- 대규모 분산 처리와 진행 중 경기 복구 고도화는 MVP 이후 확장 대상으로 둔다.

### 23.3 핵심 보존 대상

MVP에서 최소한 다음 제품 정보는 서비스 재시작과 무관하게 보존해야 한다.

- 사용자 계정, 인증 제공자, 이메일 인증 상태, 계정 상태
- 닉네임, 사용자 설정, 차단 관계
- 역할, 제재 이력, 관리자 감사 기록
- 카드 정의, 사용자 카드 보유 수량, 덱 구성
- 카드팩 개봉 감사 기록과 확률 버전
- 빠른 매치 결과, 경기 결과, 비공개 MMR
- 순서가 보존된 경기 기록과 기보 확정 상태
- 카드 사용·체크메이트 기물·카드별 경기 투입 및 승리 통계를 위한 집계 정보
- 커뮤니티 게시글, 게시글 신고, 게시글 숨김 상태
- 채팅 기록의 원문과 마스킹 결과
- 신고 증거 사본
- 운영 관측 기록과 지표

커뮤니티 댓글과 좋아요 저장은 MVP에서 제외한다.

### 23.4 개인정보, 원문, 보관 기간

- 일반 채팅 기록은 7일 보관 후 삭제하거나 원문을 제거한다.
- 신고 증거 사본은 신고 처리 완료 후 30일이 지나면 삭제 또는 비식별화한다.
- 게시글 원문은 관리자 검토와 금칙어 감사 필요에 따라 보관한다.
- 관리자 원문 조회는 권한 검사를 거치고 감사 기록을 남긴다.
- 탈퇴 사용자의 기록은 탈퇴 사용자로 표시하되 과거 기록 간 연결은 유지한다.
- 사용되지 않은 상대 레전드처럼 보존은 필요하지만 공개할 수 없는 정보는 조회자의 권한에 따라 항상 숨긴다.

### 23.5 카드 정의와 난수

- 카드와 카드팩 정의는 적용 버전을 식별할 수 있어야 하며, 진행 중 경기와 과거 기보의 의미가 이후 변경으로 달라지지 않아야 한다.
- 카드 정의의 저장·배포 방식은 제품 요구사항에서 정하지 않고 하위 기술·아키텍처 문서에서 결정한다.
- 카드팩의 무작위 결과는 서버에서만 결정한다.
- 카드팩 감사 기록에는 결과와 적용한 확률 버전을 남긴다.
- 난수 생성의 내부 비밀값은 사용자 응답, 일반 운영 기록과 기보에 노출하지 않는다.

## 24. 예외 및 엣지 케이스

서버는 다음 예외 상황을 클라이언트 상태와 무관하게 권위적으로 판정한다. 발동 전 거부와 행동 검증 실패는 경기 상태를 변경하지 않고 해당 사용자에게만 안내한다. 발동 후 카드 효과 실패는 이미 확정된 소비와 공개를 유지하는 경기 결과이므로 이 일반 거부 규칙에 포함하지 않는다.

경기 상태가 변경되지 않는다는 것은 보드, 손패, 덱, 마나, 턴 단계와 경기 결과가 변경되지 않는다는 의미다. 행동이 거부되어도 이미 흐른 경기 시간은 환급하지 않는다. 요청 제한, 보안 감사와 중복 처리 방지를 위한 기록은 별도로 남길 수 있다.

### 24.1 인증 및 권한

- 이메일 미인증 또는 필수 설정 미완료 사용자가 주요 기능을 요청하면 거부한다.
- 계정정지 사용자가 보호 기능을 요청하면 제한 안내와 함께 거부한다.
- 커뮤니케이션 정지 사용자의 채팅 전송과 커뮤니티 게시글 작성/수정/삭제/신고는 거부한다.
- 관리자 권한이 없는 사용자의 관리자 기능 접근은 거부한다.
- 로그아웃 또는 만료된 로그인 상태로 보호 기능이나 실시간 행동을 요청하면 거부한다.
- 실시간 연결과 모든 행동은 인증, 권한, 계정 상태와 제재를 검증한다.
- 로그아웃, 계정정지와 역할 회수는 기존 실시간 연결에도 반영되어야 하며 필요하면 연결을 종료한다.

### 24.2 매칭 및 경기

- 유효하지 않은 덱으로 빠른 매치 대기열 진입을 요청하면 실패한다.
- 이미 대기열에 있는 사용자가 중복으로 대기열 진입을 요청하면 중복 등록하지 않는다.
- 매칭 확정 후 취소 요청이 도착하면 이미 생성된 경기 상태를 우선한다.
- 대기열 진입 후 덱을 수정해 경기 생성 직전 덱이 유효하지 않으면 매칭 또는 경기 생성을 실패 처리한다.
- 새 차단 관계가 매칭 확정 직전에 생기면 경기 생성을 취소한다.
- 경기 참가자가 아닌 사용자의 경기 상태 조회나 행동 제출은 거부한다.
- 경기 종료 후 도착한 행동 요청은 종료 상태를 반환하고 도메인 상태를 변경하지 않는다.

### 24.3 턴, 카드, 체스 규칙

- 자기 턴이 아닌 사용자의 행동은 거부한다.
- 턴 시작 중복 처리는 거부한다.
- 최초 턴 플레이어의 최초 턴에는 턴 시작 마나를 지급하지 않는다.
- 이동 전 카드 행동, 말 이동, 프로모션 선택, 이동 후 행동 순서를 벗어난 요청은 거부한다.
- 마나가 부족하거나 카드 조건을 만족하지 못한 카드 사용은 실패한다.
- 발동 전 조건으로 체크 해소 가능성이 전혀 없는 카드, 후보가 하나도 없는 선택형 카드와 이동 가능한 폰이 하나도 없는 폰 일제 전진 카드는 소비 없이 거부한다.
- 발동 후 효과 결과가 자신의 체크를 해소하지 못하거나 킹 안전·킹 존재·표준 승패·무승부 규칙을 위반하면 부분 보드 효과를 남기지 않고 효과를 실패로 끝낸다. 발동된 카드·마나·행동 기회 소비와 공개는 유지한다.
- 이동 전 카드 사용, 말 이동, 이동 후 카드 사용으로 체크메이트가 발생하면 즉시 게임을 종료한다.
- 말 이동으로 체크메이트 또는 무승부가 발생하면 이동 후 행동은 생략한다.
- 카드 효과로 만들어진 앙파상 가능 상태는 다음 적용 가능한 말 이동 기회에만 유효하다.
- 카드 효과로 폰이 끝 행에 도달해도 프로모션은 발생하지 않는다.
- 프로모션이 필요한 상황에서 승격 기물을 선택하지 않으면 다음 단계로 진행하지 않는다.
- 일반 카드 덱이 비어 있을 때 드로우 요청은 실패한다.
- 발동 전 거부된 카드 사용과 실패한 드로우는 마나, 카드, 행동 슬롯, 손패, 덱과 보드를 변경하지 않는다. 카드 발동 후 효과 실패는 같은 단계에서 재선택하거나 재시도할 수 없다.
- 사용자가 오래된 경기 상태를 기준으로 카드를 발동하려 하면 비용 없이 거부하고 최신 상태로 복구한다.
- 카드 후보는 현재 행동할 수 있고 해당 카드를 보유한 사용자만 확인할 수 있다. 후보 확인은 카드·마나·행동 기회와 공개 경기 상태를 변경하지 않으며 후보 없음·사용 전 레전드·확인 실패를 상대에게 노출하지 않는다.
- 효과 선택 요청의 필수 정보가 없거나 형식이 잘못되면 선택 대기를 유지한다. 형식은 올바르지만 확정 후보 밖인 선택은 이미 소비된 비용을 유지한 채 효과 실패로 종료한다.
- 폰 일제 전진 카드는 시작 상태에서 막힌 폰을 제외한 이동 가능한 폰을 동시에 전진시킨다. 일부 폰의 막힘은 전체 실패 원인이 아니며 이동 가능한 폰이 없을 때만 발동 전 거부한다.
- 카드 방향은 카드 사용자 진영을 기준으로 하며 클라이언트의 보드 표시 방향이나 전달한 색상에 따라 바뀌지 않는다.
- 상대 킹만 남는 등 왕의 포섭 카드의 비킹 최단거리 후보가 없으면 카드를 발동할 수 없고 비용을 소비하지 않는다. 후보가 있어 발동한 뒤 후보 밖 기물을 선택하거나 전환할 수 없으면 이미 소비된 비용을 유지한 채 효과 실패로 종료한다.

### 24.4 시간, 연결, 종료

- 개인 시간이 0초가 되면 3초 유예 시간 안의 유효 행동만 인정한다.
- 연결 끊김 후 20초 안에 재접속하지 않으면 몰수패 처리한다.
- 연결 끊김 중 시간이 초과되어 시간패가 확정되면, 이후 재접속해도 결과를 되돌리지 않는다.
- 시간패와 몰수패 조건이 동시에 경쟁하면 먼저 도래한 제한 시각을 적용한다.
- 동일 시각이면 시간패를 우선한다.
- 종료 조건이 경쟁해도 경기 종료 결과는 한 번만 확정한다.
- 서비스 장애로 경기를 계속할 수 없으면 무효 처리하고 전적·MMR에 반영하지 않는다.

### 24.5 커뮤니티 및 신고

- 삭제되었거나 숨김 처리된 게시글에 대한 수정, 삭제, 신고 요청은 현재 상태 기준으로 검증한다.
- 댓글과 좋아요 관련 요청은 MVP에서 제공하지 않으므로 거부한다.
- 같은 사용자가 같은 게시글을 중복 신고하면 기존 신고를 반환하거나 중복 오류로 처리한다.
- 신고 5건 누적 경계에서 동시에 신고가 들어와도 자동 숨김은 한 번만 적용한다.
- 금칙어 필터가 적용된 내용은 사용자 화면에는 마스킹 결과를, 관리자 검토에는 원문을 제공한다.
- 요청 제한, 비공개 운영 정보 접근 제한과 카드팩 일일 제한을 초과하면 일관된 의미로 안내한다.

### 24.6 동시에 발생하거나 반복된 행동의 결과

- 동시에 도착한 경기 행동은 하나씩 순서대로 처리하며 먼저 확정된 행동 이후의 현재 상태를 기준으로 나머지를 다시 검증한다.
- 같은 사용자 행동이 반복 전달되어도 최초 결과를 다시 보여 줄 뿐 비용과 경기 상태를 중복 적용하지 않는다.
- 사용자 행동과 결과 기록을 함께 확정할 수 없는 기술 장애가 발생하면 일부 상태만 남기지 않는다.
- 확정된 결과를 사용자에게 전달하지 못한 경우 재접속 또는 최신 상태 복구로 동일한 결과를 확인할 수 있어야 한다.
- 거부된 행동은 보안과 문제 분석에 필요한 범위에서 별도 기록할 수 있다.

### 24.7 사용자 오류 안내

- 로그인·권한·계정 상태·제재로 사용할 수 없는 기능은 제한 이유를 안내한다.
- 덱, 차례, 카드 행동 단계, 마나, 카드 발동 조건과 체스 규칙을 만족하지 못하면 사용자가 다음 행동을 선택할 수 있는 안내를 제공한다.
- 최신 경기 상태와 다른 요청은 현재 상태로 복구한 뒤 다시 행동하도록 안내한다.
- 카드 효과 선택 요청의 형식 오류는 선택 대기를 유지하고, 후보 밖의 유효한 선택은 효과 실패로 종료되었음을 안내한다.
- 요청 제한과 카드팩 일일 제한을 초과하면 다시 시도할 수 있는 기준을 안내한다.
- 구체적인 오류 표현과 통신 형식은 별도 계약에서 정의한다.

## 25. 핵심 수용 기준

### 25.1 인증, 계정, 세션

- 사용자는 이메일/비밀번호로 가입할 수 있다.
- 이메일 인증 전 자체 로그인 사용자는 이메일 인증 관련 기능과 최소 계정 상태 조회만 사용할 수 있다.
- Google 또는 Kakao OAuth로 로그인할 수 있다.
- OAuth 사용자는 서비스 고유 닉네임 설정과 필수 약관 동의 후 주요 기능을 사용할 수 있다.
- 동일 이메일의 활성 자체 로그인 계정이 있으면 OAuth 신규 가입은 자동 병합되지 않고 차단된다.
- 모든 가입 방식에서 서비스 이용약관과 개인정보 처리방침 동의 시각 및 버전이 저장된다.
- 닉네임은 2~16자, 한글/영문/숫자/언더스코어만 허용되며 대소문자 구분 없이 중복을 막는다.
- 로그인 상태와 인증 정보는 브라우저 스크립트·위조 요청·허용되지 않은 출처로부터 보호된다.
- 로그아웃, 탈퇴, 계정정지와 비밀번호 재설정 후 기존 로그인 상태와 실시간 연결이 무효화된다.

### 25.2 계정 상태와 제재

- 인증·설정 미완료, 계정정지와 탈퇴 상태별 허용·거부 기능이 검증 가능하다.
- 계정정지 사용자는 로그인 후 제한 안내 외 주요 기능을 사용할 수 없다.
- 탈퇴 사용자는 로그인할 수 없다.
- 커뮤니케이션 정지는 채팅 전송과 커뮤니티 게시글 작성/수정/삭제/신고를 함께 제한한다.
- 제재는 기존 로그인 상태와 실시간 행동에 즉시 반영된다.

### 25.3 첫 플레이와 매칭

- 신규 사용자는 스타터 컬렉션과 기본 덱을 한 번만 지급받는다.
- 유효 덱을 가진 사용자는 빠른 매치 대기열에 진입할 수 있다.
- 유효 덱이 없으면 대기열 진입이 실패한다.
- 한 사용자는 활성 대기열 항목을 하나만 가진다.
- 중복 대기열 진입은 현재 상태를 반환하고 중복 등록하지 않는다.
- 매칭 확정 전 취소는 가능하고, 매칭 확정 후 취소는 경기 상태가 우선한다.
- 두 사용자가 매칭되면 백·흑 무작위 배정과 경기 생성이 일부만 반영되지 않도록 확정한다.
- 차단 관계가 있는 사용자는 서로 매칭되지 않는다.
- 경기 생성 후 덱 수정은 진행 중 경기에 영향을 주지 않는다.

### 25.4 게임 진행, 체스, 카드

- 서버는 표준 체스 합법 수만 허용한다.
- 턴은 턴 시작, 이동 전 카드 행동, 말 이동, 프로모션 선택, 이동 후 행동, 턴 종료 순서를 따른다.
- 최초 턴 플레이어의 최초 턴에는 턴 시작 마나를 받지 않는다.
- 이동 전 카드 행동 후 게임이 끝나지 않으면 말 이동이 필수다.
- 프로모션은 표준 말 이동 후 즉시 선택해야 한다.
- 카드 효과로 폰이 끝 행에 도달해도 프로모션은 발생하지 않는다.
- 카드 효과로 앙파상 가능 상태가 만들어지면 다음 적용 가능한 말 이동에서만 유효하다.
- 이동 전 카드 사용, 말 이동, 이동 후 카드 사용 모두 체크메이트를 만들 수 있고 즉시 게임을 종료한다.
- 체크메이트, 스테일메이트, 50수 규칙, 3회 반복, 기물 부족이 올바르게 판정된다.
- 모든 카드는 발동 전 거부, 발동 후 효과 성공과 발동 후 효과 실패를 구분한다.
- 발동 전 권한·상태·마나·최초 조건·후보 존재 검증 실패는 카드·비용·행동 기회와 공개 상태를 바꾸지 않는다.
- 발동이 성공하면 효과 성공 여부와 관계없이 카드·마나·행동 슬롯을 소비하고 패에서 제거·공개하며 레전드는 사용 완료 처리한다.
- 발동 후 효과 규칙 또는 대상 선택 실패는 비용을 유지하고 재선택 없이 종료하며, 효과의 부분 보드·특수 상태 변경은 남기지 않는다.
- 실시간 경기에서 각 플레이어는 자신의 진영 시작 행이 화면 아래쪽인 보드를 보지만, 경기 좌표와 기보는 표준 `a1`~`h8` 좌표를 유지한다.
- 백·흑 카드 사용자의 왼쪽, 오른쪽, 왼쪽 앞과 오른쪽 앞 방향이 사용자 진영 기준으로 올바르게 적용된다.
- 선택형 카드는 발동과 비용 소비 뒤 대상 선택을 기다리고, 재접속·시간 진행·잘못된 입력의 대기 유지와 후보 밖 선택의 소비 유지 실패가 일관되게 재현된다.
- 대상 선택이 없는 카드는 한 번의 행동에서 발동과 효과 성공 또는 실패가 연속해서 확정된다.
- 상대에게는 발동 카드와 일반적인 결과를, 카드 사용자에게만 후보·구체적인 선택·상세 실패 사유를 보여 준다.
- 폰 일제 전진, 기물 강제 교환과 왕의 포섭 카드의 발동 조건, 효과 성공·실패, 대상·특수 상태·50수 규칙·공개와 소비 정책이 카드 정의와 일치한다.

### 25.5 마나

- 시작 마나는 양 플레이어 모두 0이다.
- 마나는 10을 초과해 저장되지 않는다.
- 턴 시작 마나, 캡처 마나, 앙파상 추가 마나가 서버에서 계산된다.
- 체크를 건 플레이어는 마나를 얻지 않는다.
- 체크를 2회 당할 때마다 체크를 당한 플레이어가 마나 1을 얻는다.
- 카드 효과로 발생한 체크도 체크를 당한 횟수에 포함한다.
- 모든 마나 변화는 기보에 사유와 함께 기록된다.

### 25.6 시간과 재접속

- 각 플레이어는 개인 시계 30분을 가진다.
- 자기 턴 동안에만 시간이 감소한다.
- 모든 턴 단계는 현재 턴 플레이어의 시간을 소비한다.
- 시간이 0초가 된 뒤 3초 안에 유효 행동이 없으면 시간패 처리된다.
- 연결 끊김 후 20초 안에 복귀하면 경기 상태가 재동기화된다.
- 연결 끊김 중에도 본인 턴이면 시간은 계속 감소한다.
- 복귀 전에 시간패가 확정되었으면 복귀해도 결과는 되돌리지 않는다.
- 20초 안에 복귀하지 않으면 몰수패 처리된다.

### 25.7 카드팩과 컬렉션

- MVP 카드팩은 기본 카드팩 1종만 제공한다.
- MVP 카드팩은 0골드로 개봉할 수 있다.
- 카드팩 1개는 카드 3장을 지급한다.
- 각 카드 슬롯은 독립적으로 레전드 1% 확률을 적용한다.
- 사용자는 KST 기준 하루 10회까지만 카드팩을 성공 개봉할 수 있다.
- 동시 요청으로 하루 10회 제한을 초과할 수 없다.
- 중복 카드는 보유 수량으로 누적된다.
- 카드팩 개봉 감사 기록이 보존된다.

### 25.8 전적과 기보

- 종료된 경기만 전적에 반영된다.
- 경기 종료는 결과, 전적, MMR과 기보 확정이 일부만 반영되지 않도록 처리한다.
- 기보는 서버가 확정한 경기 행동과 결과의 순서를 기준으로 저장된다.
- 경기 참가자만 자신의 경기 기보를 조회할 수 있다.
- 사용되지 않은 상대 레전드는 기보에서도 공개되지 않는다.
- 발동된 일반·레전드 카드는 효과 성공 여부와 관계없이 카드 식별자·버전, 사용자, 소비·사용 완료, 효과 상태와 공개 가능한 결과를 기보에 기록한다.
- 발동 후 효과 실패한 레전드는 사용된 카드로 상대 기보에도 공개하되 카드 후보, 사용자의 구체적인 선택과 상세 실패 사유는 상대 기보에서 제외한다.
- 카드 효과로 경기가 종료되면 카드 발동·효과 결과, 경기 결과·전적·MMR과 기보를 함께 확정하고 중복 기록하지 않는다.
- 서버 재시작으로 무효 처리된 활성 경기는 전적과 MMR에 반영되지 않는다.

### 25.9 채팅, 커뮤니티, 신고

- 채팅 메시지는 최대 100자다.
- 채팅은 사용자와 경기 기준의 요청 제한을 적용한다.
- 차단한 사용자의 채팅은 서버가 차단자에게 전달하지 않는다.
- 커뮤니케이션 정지 사용자는 채팅과 커뮤니티 게시글 작성/수정/삭제/신고를 할 수 없다.
- 커뮤니티는 게시글 작성, 목록, 상세, 수정, 삭제, 검색, 신고, 자동 숨김을 제공한다.
- 커뮤니티 댓글과 좋아요는 MVP에서 제공하지 않는다.
- 게시글 신고 5건이 서로 다른 유효 사용자에게 누적되면 자동 숨김 처리된다.
- 금칙어가 포함된 채팅과 게시글은 사용자 화면에 마스킹되어 표시된다.
- 관리자 원문 조회는 권한 검사와 감사 기록을 거친다.

### 25.10 관리자, 운영, 데이터

- 관리자는 신고 검토, 게시글 숨김/복구, 커뮤니케이션 정지, 계정정지, 금칙어 생성·조회·수정·삭제를 수행할 수 있다.
- 카드 효과, 확률, 팩 가격 관리자 편집은 MVP에서 제공하지 않는다.
- 요청 제한, 운영 정보 접근 제한, 서비스 상태와 주요 운영 지표 관측이 적용된다.
- 일반 운영 기록과 지표는 비밀번호, 인증 비밀값, 이메일 원문, 채팅·게시글 원문과 난수 생성의 내부 비밀값을 포함하지 않는다.
- 일반 채팅 기록은 7일 보관 후 삭제 또는 비식별화한다.
- 신고 증거 사본은 신고 처리 완료 후 30일 뒤 삭제 또는 비식별화한다.
- 같은 사용자 오류는 화면과 실시간 기능에서 일관된 의미로 안내된다.
- 반복된 사용자 행동, 동시 신고, 동시 카드팩 개봉과 동시에 발생한 경기 종료 조건에서도 결과가 중복 적용되지 않아야 한다.

## 26. 제품 정의와 후속 결정 경계

### 26.1 확정 제품 정책

- MVP 카드 12장의 제품 정의는 `docs/cards`에 확정한다.
- 카드 식별자는 영구적이고 같은 버전의 의미는 바뀌지 않는다. 비용, 사용 시기, 발동 조건, 후보 규칙, 효과, 실패 시 소비와 공개 의미가 바뀌면 새 버전을 사용한다.
- 경기 시작 시 적용할 카드 식별자와 버전을 고정하며 이후 카드 변경은 진행 중 경기와 과거 기보에 영향을 주지 않는다.
- 과거 기보는 경기 당시 확정된 카드 발동, 소비와 효과 결과를 그대로 보여 주며 현재 카드 효과로 다시 해석하지 않는다.
- 비공개 MMR 수치 정책은 다음과 같이 확정한다.
  - 신규 사용자의 최초 MMR은 1400이다.
  - 표준 Elo 기대 승률 공식과 `K=32`를 사용한다.
  - 승리는 1, 패배는 0, 무승부는 0.5의 실제 결과값을 사용한다.
  - 시간패와 몰수패는 패배로 처리하고, 서버 장애로 무효 처리된 경기는 MMR에 반영하지 않는다.
  - 매칭 시작 후 0~10초는 ±50, 이후 10초마다 허용 범위를 두 배로 확대한다.
- 카드·카드팩 정의의 저장, 배포, 변경 이력과 과거 기보 제공 방식은 위 제품 정책을 만족하도록 하위 기술·아키텍처 문서에서 결정한다.

### 26.2 MVP 확정 제약 요약

- MVP는 같은 사이트의 웹 서비스로 제공한다.
- 모바일 앱과 외부 공개 서비스 연동은 지원하지 않는다.
- MVP 카드팩 가격은 0골드이며 유료 결제, 환불, 골드 구매는 제공하지 않는다.
- MVP에서 서비스 장애 후 진행 중 경기의 완전 복구는 필수 요구사항이 아니다.
- 서비스 장애로 계속할 수 없는 경기는 무효 처리하고 전적·MMR에 반영하지 않는다.
- 관리자 권한은 승인된 운영 절차로 부여한다.
- 카드 효과는 표준 체스 승패 판정을 무시하지 않는다.
- 커뮤니티는 게시글 중심이며 댓글과 좋아요는 MVP에서 제외한다.
- 자동화된 카드 제작·관리 도구는 MVP 범위에서 제외한다.

## 27. MVP 제외 및 이후 확장 후보

아래 항목은 MVP 범위, 수용 기준, 테스트 범위에 포함하지 않는다. 추후 별도 요구사항 정의와 보안/운영 검토 후 확장한다. 아래 확장 후보는 현재 확정한 단일 인스턴스, 세션 쿠키, 고정 카드 데이터, 게시글 중심 커뮤니티, 제한된 관리자 기능을 대체하지 않는다.

### 27.1 게임/소셜 확장

- 솔로플레이 봇
- 친구 대전 및 초대 코드
- 친구/팔로우
- 관전
- 공개 기보

관전과 공개 기보를 추가할 때는 사용 전 레전드, 손패, 덱 구성과 비공개 MMR 등 숨김 정보가 조회자에게 노출되지 않아야 한다.

### 27.2 랭크/보상/경제 확장

- 공개 랭크 티어
- 시즌
- 리더보드
- 출석 보상
- 경기 보상
- 골드 구매

MVP의 비공개 MMR은 빠른 매치 내부 점수이며, 공개 랭크나 시즌 시스템과 다르다.

### 27.3 커뮤니티 확장

- 커뮤니티 댓글
- 게시글 좋아요
- 댓글 좋아요
- 커뮤니티 이미지 업로드

이미지 업로드를 추가할 때는 파일 제한, 악성 파일 방지, 접근 권한과 신고·삭제 정책을 별도 정의해야 한다.

### 27.4 카드/카드팩/밸런싱 확장

- MVP 이후 신규 카드 세트 추가
- 기본팩 외 신규 카드팩 추가
- MVP 이후 카드 상세 밸런스 패치
- 카드 삽입/삭제 및 카드팩 추가/수정 관리자 도구
- 카드 효과/확률/가격 관리자 편집

MVP 카드 12장의 상세 정의는 확장 후보가 아니라 현재 확정된 제품 정책이다. 관리자 편집 도구를 추가할 때는 버전 고정, 기존 기보 재현성, 변경 승인, 감사 기록과 변경 취소 정책을 별도 정의해야 한다.

### 27.5 관리자/운영 도구 확장

- 고급 관리자 검색, 필터, 정렬, 엑셀 다운로드
- 고급 금칙어 필터링
- 공백/특수문자 우회 탐지
- 초성 분리
- Unicode confusable 대응
- 외부 필터 서비스 연동

외부 필터 서비스를 연동할 때는 개인정보 전송 범위, 장애 시 처리, 오탐 대응과 감사 정책을 별도 정의해야 한다.

### 27.6 인증·계정·외부 연동 확장

- 설정 화면에서 자체 로그인 계정과 OAuth 계정 연동
- 모바일 앱
- 외부 공개 서비스 연동

모바일 앱과 외부 연동을 추가할 때는 기존 웹 로그인과 분리된 보안·권한 요구사항을 별도로 정의한다.

### 27.7 확장성/인프라 확장

- 대규모 동시 접속과 경기 처리
- 여러 서버에 걸친 실시간 연결과 상태 동기화
- 분산 시간·매칭 처리
- 진행 중 경기의 장애 복구 고도화

이 확장에서는 경기 행동 순서, 중복 방지, 장애 복구와 실시간 연결 일관성 요구사항을 별도로 정의해야 한다.

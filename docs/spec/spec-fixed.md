# TCG Online Chess Backend MVP 요구사항 명세

## 1. 개요

TCG Online Chess는 표준 체스 규칙을 기반으로, 경기 중 카드 효과를 제한적으로 결합한 실시간 1:1 온라인 웹 게임이다. 본 문서는 백엔드 MVP 구현을 위한 요구사항 명세이며, React 프론트엔드와 Spring Boot 백엔드, STOMP WebSocket 기반 실시간 통신을 전제로 한다.

기본 언어는 한국어이며, 서비스 운영 기준 시간대는 KST(Asia/Seoul)로 한다.

## 2. 문서 목적과 범위

이 문서는 TCG Online Chess 백엔드 MVP의 구현, 테스트, 검토 기준을 정의한다. 프론트엔드 화면 상세나 운영 배포 자동화 절차가 아니라, 백엔드가 제공해야 하는 도메인 규칙, 인증, 실시간 통신, 데이터 저장, 운영 보호 장치를 중심으로 작성한다.

- 대상 시스템은 Spring Boot 백엔드와 STOMP WebSocket 기반 실시간 게임 서버다.
- React 프론트엔드는 이 문서의 API 및 실시간 이벤트 원칙을 사용하는 클라이언트로 전제한다.
- MVP는 1:1 온라인 PvP 경험을 완성하는 것을 우선한다.
- MVP 제외 기능은 구현하지 않되, 이후 확장 시 충돌하지 않도록 데이터와 권한 경계를 고려한다.

## 3. 용어 정의

문서, 코드, API, 대화에서 같은 개념을 다른 단어로 부르지 않도록 다음 표준 용어를 사용한다.

| 표준 용어 | 동의어/이전 표현 | 정의 | 관련 코드/API 용어 |
| --- | --- | --- | --- |
| 사용자 | 회원, 유저, 플레이어 | 서비스 계정을 가진 주체다. 경기 안에서는 플레이어 역할을 가진다. | `User`, `Member`, `Player` |
| 자체 로그인 사용자 | 이메일 로그인 사용자 | 이메일과 비밀번호로 가입한 사용자다. 이메일 인증 완료 전에는 주요 기능 사용이 제한된다. | `LocalAccount` |
| OAuth 사용자 | 소셜 로그인 사용자 | Google 또는 Kakao OAuth 2.0으로 로그인하는 사용자다. 서비스 고유 닉네임 설정이 필요하다. | `OAuthAccount` |
| 닉네임 | 표시 이름, 유저명 | 서비스 내 공개 식별자다. MVP에서는 가입 후 변경할 수 없다. | `nickname` |
| 빠른 매치 | 매치메이킹, PvP 매칭 | 유효한 덱을 가진 사용자를 비공개 MMR 기준으로 자동 매칭하는 MVP의 유일한 매칭 방식이다. | `QuickMatch` |
| 경기 | 게임, 대국, 매치 | 백/흑 두 플레이어가 하나의 체스 보드와 카드 상태를 공유하며 진행하는 단위다. | `Game`, `Match` |
| 턴 | 차례 | 한 플레이어가 이동 전 카드 행동, 말 이동, 이동 후 행동을 순서대로 수행하는 진행 단위다. | `Turn` |
| 이동 전 카드 행동 | 선 카드 행동 | 말 이동 전에 카드 1장 사용 또는 아무것도 하지 않음을 선택하는 단계다. | `preMoveAction` |
| 이동 후 행동 | 후 행동, 이동 후 카드 행동 | 말 이동 후 카드 1장 사용, 일반 카드 1장 드로우, 아무것도 하지 않음 중 하나를 선택하는 단계다. | `postMoveAction` |
| 일반 카드 | 공개 카드 | 경기 중 드로우할 수 있고 상대에게 공개되는 카드다. | `NormalCard` |
| 레전드 카드 | 궁극기 카드, 비공개 카드 | 경기당 1장만 사용 가능한 1회성 카드이며 사용 전까지 상대에게 공개되지 않는다. | `LegendCard` |
| 마나 | 비용 자원 | 카드 사용 비용으로 쓰는 경기 중 자원이며 체스 행동 결과로 획득한다. | `mana` |
| 덱 | 카드 덱 | 한 경기에서 사용할 레전드 카드 1장과 일반 카드 5장의 조합이다. | `Deck` |
| 손패 | 핸드 | 경기 중 플레이어가 현재 보유한 카드 목록이다. 일반 카드 손패는 상대에게 공개된다. | `Hand` |
| 기보 | 경기 기록, 수순 기록 | 말 이동, 카드 사용, 마나 변화, 주요 상태 변화를 포함한 경기 재현용 기록이다. | `GameRecord` |
| 시간패 | 시간사 | 개인 시계가 0초가 된 뒤 유효 행동이 완료되지 않아 발생하는 패배다. | `TIMEOUT` |
| 몰수패 | 접속 종료 패배 | 연결 끊김 후 제한 시간 안에 재접속하지 못해 발생하는 패배다. | `FORFEIT` |
| 신고 | 리포트 | 채팅, 게시글, 댓글, 사용자 프로필을 관리자 검토 대상으로 제출하는 행위다. | `Report` |
| 차단 | 블록 | 특정 사용자와의 매칭 및 채팅 노출을 제한하는 사용자 설정이다. | `Block` |
| 제재 | 정지, 관리자 조치 | 관리자가 사용자에게 적용하는 채팅정지 또는 계정정지 조치다. | `Sanction` |

## 4. 사용자 역할과 권한

사용자 상태와 역할에 따라 기능 접근 범위를 다르게 적용한다.

| 역할/상태 | 허용 기능 | 제한 기능 |
| --- | --- | --- |
| 비회원 | 로그인, 회원가입, OAuth 로그인 진입 | 매칭, 경기, 카드팩, 덱, 마이페이지, 커뮤니티 조회/작성 |
| 이메일 미인증 사용자 | 로그인, 로그아웃, 이메일 인증 메일 재요청, 비밀번호 재설정, 제한 안내 조회, 최소 계정 상태 조회 | 매칭, 경기, 카드팩, 덱, 마이페이지 주요 기능, 커뮤니티 조회/작성/댓글/좋아요/신고 |
| 일반 사용자 | 빠른 매치, 경기, 카드팩, 컬렉션, 덱 편성, 마이페이지, 커뮤니티, 신고, 차단 | 관리자 기능 |
| 채팅정지 사용자 | 로그인, 경기 진행, 조회성 기능 | 대전 채팅 전송, 제재 정책에 따라 커뮤니티 작성 |
| 계정정지 사용자 | 로그인 후 제한 안내 확인 | 매칭, 경기, 카드팩, 커뮤니티 작성, 채팅, 신고 등 주요 기능 |
| 관리자 | 신고 검토, 숨김/복구, 사용자 제재, 카드 목록 조회, 금칙어 관리 | 카드 효과, 확률, 팩 가격 수정 |
| 탈퇴 사용자 | 로그인 불가 | 모든 사용자 기능. 기존 기록의 작성자 표시는 탈퇴 사용자로 처리 |

관리자 권한은 초기 MVP에서 DB 수동 부여를 전제로 한다. 권한 검사는 REST API와 STOMP WebSocket 명령 처리에서 모두 서버가 수행한다.

## 5. 주요 사용자 시나리오

### 5.1 가입 및 인증

1. 사용자는 이메일/비밀번호 또는 OAuth로 가입한다.
2. 자체 로그인 사용자는 이메일 인증을 완료한다.
3. OAuth 사용자는 서비스 고유 닉네임을 설정한다.
4. 모든 가입 방식은 필수 약관 동의를 완료한다.
5. 인증과 필수 설정이 완료된 사용자는 주요 기능을 사용할 수 있다.

### 5.2 빠른 매치와 경기 시작

1. 사용자는 유효한 덱을 보유한 상태로 빠른 매치 대기열에 진입한다.
2. 서버는 비공개 MMR과 차단 관계를 고려해 상대를 찾는다.
3. 매칭이 확정되면 서버가 경기를 생성하고 백/흑을 무작위 배정한다.
4. 서버는 양쪽 클라이언트에 경기 시작과 초기 상태를 전송한다.

### 5.3 턴 진행과 카드 사용

1. 현재 턴 플레이어는 이동 전 카드 행동을 선택한다.
2. 서버는 카드 조건, 마나, 체크 상태 제한을 검증한다.
3. 플레이어는 합법 말 이동을 수행한다.
4. 서버는 이동 결과, 마나 획득, 체크/무승부/승패 여부를 확정한다.
5. 플레이어는 이동 후 행동으로 카드 사용, 일반 카드 드로우, 아무것도 하지 않음 중 하나를 선택한다.
6. 서버는 턴 종료 후 다음 플레이어에게 턴을 넘긴다.

### 5.4 카드팩, 컬렉션, 덱 편성

1. 사용자는 MVP에서 0골드 카드팩을 구매하고 개봉한다.
2. 서버는 슬롯별 독립 확률로 카드 3장을 지급하고 감사 로그를 남긴다.
3. 중복 카드는 보유 수량으로 누적한다.
4. 사용자는 보유 카드와 중복 제한을 만족하는 덱을 편성한다.

### 5.5 커뮤니티 신고와 관리자 처리

1. 로그인 사용자는 게시글, 댓글, 사용자 프로필, 대전 채팅을 신고할 수 있다.
2. 게시글 또는 댓글 신고가 5건 누적되면 자동 임시 숨김 처리한다.
3. 관리자는 신고 내용을 검토하고 숨김 유지, 복구, 사용자 제재를 결정한다.

## 7. 인증 및 계정

### 7.1 인증 방식

MVP의 웹 인증은 서버 세션과 쿠키를 사용한다.

- 세션 쿠키는 `HttpOnly`, `Secure`, `SameSite` 속성을 사용한다.
- 프론트엔드와 백엔드는 같은 사이트 배포를 전제로 한다.
- REST의 상태 변경 요청은 CSRF 검증을 적용한다.
- STOMP WebSocket `CONNECT` 요청도 세션과 CSRF를 검증한다.
- WebSocket 연결이 성공하면 서버는 STOMP 세션에 인증 사용자 식별자를 연결한다.

JWT Access/Refresh Token 방식은 MVP에서 사용하지 않는다. 향후 모바일 앱, 외부 API, 멀티 클라이언트 지원이 필요해질 때 별도 인증 방식으로 추가 검토한다.

### 7.2 회원가입 및 로그인

- 자체 로그인은 이메일, 비밀번호, 고유 닉네임을 사용한다.
- 닉네임은 서비스 내 공개 식별자이며 가입 후 변경할 수 없다.
- 닉네임은 2~16자이며 한글, 영문, 숫자, 언더스코어만 허용한다.
- 닉네임 입력값의 앞뒤 공백은 제거한다.
- 닉네임 중복은 대소문자를 구분하지 않고 검사한다.
- 금칙어가 포함된 닉네임은 사용할 수 없다.
- 자체 로그인 사용자는 이메일 인증을 완료해야 주요 기능을 사용할 수 있다.
- 이메일 미인증 사용자는 로그인, 로그아웃, 이메일 인증 메일 재요청, 비밀번호 재설정, 제한 안내 조회, 최소 계정 상태 조회만 사용할 수 있다.
- OAuth 제공자는 Google, Kakao를 지원한다.
- OAuth 계정도 서비스 고유 닉네임을 설정해야 한다.
- OAuth 사용자는 별도 이메일 인증을 요구하지 않으며, 서비스 고유 닉네임 설정과 필수 약관 동의를 완료하면 주요 기능을 사용할 수 있다.
- 모든 가입 방식은 서비스 이용약관과 개인정보 처리방침 동의를 필수로 요구한다.
- 약관 동의 시각과 약관 버전을 저장한다.

### 7.3 비밀번호 및 로그인 보안

- 비밀번호는 8~72자로 제한한다.
- 비밀번호는 BCrypt 계열의 안전한 해시 알고리즘으로 저장한다.
- 같은 계정 기준 5회 연속 로그인 실패 시 10분 동안 계정을 잠근다.
- 계정 잠금 중에도 비밀번호 재설정 요청은 허용한다.
- 비밀번호 재설정이 완료되면 로그인 실패 횟수와 계정 잠금을 초기화한다.
- IP 기반 Rate Limit은 계정 잠금과 별도로 적용한다.
- 이메일 인증 토큰과 비밀번호 재설정 토큰은 모두 5분 만료 시간을 가진다.
- 이메일 인증 토큰과 비밀번호 재설정 토큰은 1회 사용 후 즉시 무효화한다.
- 새 이메일 인증 토큰을 발급하면 같은 사용자의 기존 미사용 이메일 인증 토큰은 모두 무효화한다.
- 새 비밀번호 재설정 토큰을 발급하면 같은 사용자의 기존 미사용 재설정 토큰은 모두 무효화한다.
- 이메일 인증 메일과 비밀번호 재설정 메일은 SMTP 인터페이스로 발송한다.
- SMTP 호스트, 포트, 계정, TLS 설정은 환경 변수로 주입한다.
- 이메일 발송은 사용자 요청 API에서 즉시 1회 시도하고, 실패하면 실패 응답을 반환한다.
- MVP에서는 이메일 발송 백그라운드 자동 재시도 큐를 두지 않는다.

### 7.4 계정 탈퇴

- 사용자는 계정 탈퇴를 요청할 수 있다.
- MVP에서는 계정을 물리 삭제하지 않고 비활성화한다.
- 전적, 기보, 커뮤니티 기록은 서비스 무결성을 위해 보존하되, 작성자 표시는 탈퇴 사용자로 처리한다.
- 동일 이메일 또는 동일 OAuth 식별자로 재가입하면 새 계정을 만들지 않고 기존 비활성화 계정을 재활성화한다.
- 탈퇴 계정 재활성화 시 기존 닉네임을 그대로 유지한다.

## 8. 실시간 통신

### 8.1 WebSocket 방식

- Spring WebSocket + STOMP를 사용한다.
- 클라이언트는 행동 의도만 서버에 전송한다.
- 서버는 모든 게임 상태, 합법 수, 카드 조건, 타이머, 승패를 권위적으로 계산한다.
- 클라이언트가 보낸 상태 계산 결과는 신뢰하지 않는다.

### 8.2 주요 실시간 이벤트

서버는 최소한 다음 유형의 이벤트를 실시간으로 전달해야 한다.

- 매칭 상태 변경
- 경기 시작
- 현재 게임 상태 스냅샷
- 합법 행동 성공
- 행동 실패 및 실패 사유
- 턴 변경
- 타이머 변경
- 카드 사용 결과
- 드로우 결과
- 체크, 체크메이트, 무승부, 시간패, 몰수패, 항복
- 채팅 메시지
- 재접속 상태

## 9. API 또는 인터페이스 요구사항

백엔드는 REST API와 STOMP WebSocket을 함께 제공한다. REST API는 조회, 설정, 관리, 비실시간 명령에 사용하고, STOMP WebSocket은 경기 중 실시간 상태 변화와 플레이어 행동 처리에 사용한다.

### 9.1 REST API 원칙

- REST API는 서버 세션 기반 인증을 사용한다.
- 상태 변경 요청은 CSRF 검증을 통과해야 한다.
- 사용자, 인증, 카드 컬렉션, 덱, 마이페이지, 전적, 기보, 커뮤니티, 신고, 관리자 기능은 REST API로 제공한다.
- REST API 응답은 클라이언트가 화면을 구성하는 데 필요한 서버 확정 상태를 반환한다.
- 클라이언트가 계산한 게임 상태, 카드 결과, 승패 결과는 REST API와 STOMP 명령 모두에서 신뢰하지 않는다.

### 9.2 STOMP WebSocket 원칙

- STOMP `CONNECT`는 서버 세션과 CSRF 검증을 통과해야 한다.
- 경기 참가, 행동 제출, 채팅 전송, 재접속 상태 동기화는 STOMP 메시지로 처리한다.
- 클라이언트는 행동 의도만 전송하고, 서버는 검증된 결과 이벤트를 브로드캐스트한다.
- 서버는 사용자가 구독할 수 있는 경기 채널과 개인 응답 채널을 분리한다.
- 실패한 행동은 상태를 변경하지 않고 실패 사유 이벤트로 응답한다.

### 9.3 인터페이스 안정성

- MVP 단계에서 외부 공개 API 호환성은 보장하지 않는다.
- 프론트엔드와 백엔드는 같은 제품 릴리스 단위로 함께 변경할 수 있다.
- 단, 도메인 이벤트 이름과 주요 상태 필드는 테스트에서 검증 가능하도록 일관되게 유지한다.

## 10. 매치메이킹

### 10.1 빠른 매치

- MVP는 빠른 매치만 제공한다.
- 친선 코드, 랭크 모드, 관전 모드는 제공하지 않는다.
- 사용자는 유효한 덱을 보유해야 빠른 매치 대기열에 진입할 수 있다.
- 매칭 전에는 사용자가 대기열을 취소할 수 있다.
- 매칭 확정 후 서버는 즉시 경기 생성 및 입장 상태로 전환한다.

### 10.2 매칭 기준

- 사용자는 비공개 단일 MMR을 가진다.
- 빠른 매치는 MMR이 가까운 상대를 우선한다.
- 대기 시간이 길어질수록 허용 MMR 범위를 점진적으로 넓힌다.
- 경기 결과에 따라 비공개 MMR을 갱신한다.
- 공개 티어나 랭크 표시는 MVP에 포함하지 않는다.
- 차단 관계가 있는 두 사용자는 빠른 매치에서 서로 매칭되지 않는다.

### 10.3 색 배정

- 서버가 경기 시작 시 백/흑을 무작위 배정한다.
- 백이 먼저 둔다.

## 11. 체스 룰

### 11.1 기본 규칙

- 보드는 표준 8x8 체스 보드다.
- 기물 종류, 수량, 초기 배치는 표준 체스를 따른다.
- 기본 승리 조건은 체크메이트다.
- 킹을 직접 잡는 방식으로 승리하지 않는다.
- 자기 턴에는 합법 수 하나를 반드시 둬야 한다.
- 둘 수 있는 합법 수가 없을 때는 체크메이트 또는 스테일메이트로 판정한다.

### 11.2 특수 수

MVP는 표준 체스 특수 규칙을 모두 지원한다.

- 캐슬링
- 앙파상
- 프로모션

프로모션은 폰이 끝 행에 도달한 즉시 플레이어가 퀸, 룩, 비숍, 나이트 중 하나를 선택해야 한다.

### 11.3 무승부

다음 표준 체스 무승부 조건을 지원한다.

- 스테일메이트
- 기물 부족
- 50수 규칙
- 3회 반복

플레이어 간 합의 무승부는 MVP에서 제외한다.

### 11.4 항복

- 플레이어는 경기 중 항복할 수 있다.
- 항복은 확인 단계를 거친 뒤 확정된다.
- 항복이 확정되면 즉시 해당 플레이어의 패배로 종료한다.

### 11.5 체스 룰 엔진

- 서버는 체스 룰 검증에 검증된 Java 라이브러리를 사용한다.
- 기본 후보는 `bhlangonijr/chesslib`이다.
- 카드 효과는 체스 라이브러리의 기본 합법 수 검증을 감싸는 별도 도메인 계층에서 처리한다.

## 12. 턴 진행

한 턴은 다음 순서로 진행한다.

1. 이동 전 카드 행동
2. 말 이동
3. 이동 후 행동
4. 턴 종료

### 12.1 이동 전 카드 행동

플레이어는 다음 중 하나를 선택한다.

- 카드 1장 사용
- 아무것도 하지 않음

이 단계에서 사용할 수 있는 카드는 카드 조건, 마나, 체크 상태 제한을 모두 만족해야 한다.

### 12.2 말 이동

- 말 이동은 항상 필수다.
- 서버는 이동의 합법성을 검증한다.
- 이동 결과 체크, 체크메이트, 무승부, 캡처, 프로모션, 앙파상 여부를 판정한다.
- 프로모션이 필요한 경우 승격 기물 선택이 완료되어야 다음 단계로 넘어간다.

### 12.3 이동 후 행동

플레이어는 다음 중 하나를 선택한다.

- 카드 1장 사용
- 일반 카드 1장 드로우
- 아무것도 하지 않음

이동 후 행동은 하나만 수행할 수 있다.

### 12.4 체크 상태의 카드 제한

- 자신의 킹이 체크 상태일 때는 체크 해소에 기여하는 카드만 사용할 수 있다.
- 카드 사용 후에도 최종적으로 킹이 체크 상태라면 해당 행동은 불법이다.
- 일반 카드와 레전드 카드 모두 이 제한을 따른다.

## 13. 시간 규칙

### 13.1 기본 시간

- 각 플레이어는 개인 시계 30분을 가진다.
- 자기 턴 동안에만 자신의 시간이 감소한다.
- 턴 시작 시 남은 시간이 10초 미만이면 10초로 보정한다.
- 남은 시간이 10초 이상이면 별도 보정하지 않는다.

### 13.2 시간패

- 서버 기준 개인 시간이 0초에 도달하면 3초 유예 시간을 둔다.
- 유예 시간 안에 유효한 행동이 완료되지 않으면 시간패 처리한다.
- 클라이언트 표시 시간보다 서버 시간이 우선한다.

### 13.3 연결 끊김

- 연결이 끊겨도 개인 시계는 계속 진행한다.
- 사용자가 20초 안에 재접속하면 현재 경기 상태로 복귀한다.
- 20초 안에 재접속하지 않으면 몰수패 처리한다.
- 연결 끊김으로 시간도 0초가 되면 시간패 또는 몰수패 중 서버가 먼저 확정한 종료 사유를 적용한다.

## 14. 카드 시스템

### 14.1 카드 종류

카드는 일반 카드와 레전드 카드로 나눈다.

- MVP 초기 카드 규모는 일반 카드 10장, 레전드 카드 2장으로 둔다.
- 본 문서에는 일반 카드 1장과 레전드 카드 1장만 예시로 작성한다.
- 실제 카드명, 상세 효과, 사용 조건, 마나 비용, 구현 방식은 별도 카드 상세 정의에서 다룬다.
- 일반 카드
  - 체스 룰을 깨지 않는 보조 효과 중심이다.
  - 상대에게 공개된다.
  - 경기 중 일반 카드 덱에서 드로우할 수 있다.
- 레전드 카드
  - 경기당 1장만 사용 가능한 1회성 궁극기다.
  - 사용 전까지 상대에게 공개되지 않는다.
  - 마나 외 추가 사용 조건을 가질 수 있다.
  - 이름에 맞는 큰 전투 개입 효과를 가질 수 있다.
  - 킹 직접 제거, 즉시 승리 효과는 MVP 카드 목록에서 제외한다.

### 14.2 덱 구성

- 한 경기용 덱은 레전드 카드 1장과 일반 카드 5장으로 구성한다.
- 레전드 카드는 덱에 지정된 1장이 게임 시작 시 확정 지급된다.
- 일반 카드는 경기 시작 시 덱에서 무작위 1장을 지급한다.
- 일반 카드 덱은 카드별 최대 2장까지 중복 편성할 수 있다.
- 덱 편성은 보유 카드 수량과 중복 제한을 모두 만족해야 한다.
- 유효하지 않은 덱으로는 빠른 매치에 참가할 수 없다.

### 14.3 경기 중 카드 공개 정보

- 자신의 모든 손패는 자신에게 보인다.
- 상대의 일반 카드 손패는 공개된다.
- 상대의 레전드 카드는 사용 전까지 비공개다.
- 일반 카드 덱 잔량은 상대에게 보인다.

### 14.4 드로우

- 일반 카드 드로우는 이동 후 행동의 선택지 중 하나다.
- 드로우 시 일반 카드 덱에서 1장을 뽑는다.
- 덱이 비어 있으면 드로우 행동은 실패한다.
- 드로우한 일반 카드는 상대에게도 공개된다.

### 14.5 카드 데이터 필드

카드는 최소한 다음 정보를 가진다.

- 카드 ID
- 카드명
- 카드 종류: 일반, 레전드
- 마나 비용
- 사용 가능 타이밍: 이동 전, 이동 후, 둘 다
- 대상 조건
- 추가 사용 조건
- 효과 타입
- 효과 파라미터
- 공개 여부
- 활성 여부

### 14.6 효과 범위

일반 카드의 효과 범위는 다음과 같이 제한한다.

- 특정 기물의 다음 이동 보조
- 특정 기물 보호
- 드로우 보조
- 마나 획득 또는 비용 감소
- 제한적인 정보 확인
- 체크 룰과 최종 킹 안전성을 깨지 않는 보조 효과

레전드 카드의 효과 범위는 다음을 포함할 수 있다.

- 상대 기물 제거
- 기물 위치 교환
- 특정 기물 강제 이동
- 특정 상태 해제
- 강한 보호 또는 반격 효과

레전드 카드라도 다음 효과는 MVP에서 만들지 않는다.

- 상대 킹 직접 제거
- 즉시 승리
- 표준 체스 승패 판정을 무시하는 효과

모든 카드 효과는 체스 규칙을 1순위로 따른다. 서버는 카드 효과 적용 후의 최종 보드 상태를 기준으로 체스 규칙 위반 여부를 검증하고, 다음 중 하나라도 해당하면 카드 사용을 실패 처리한다.

- 자기 킹이 체크 상태로 남음
- 상대 킹을 직접 제거함
- 양쪽 킹 수가 비정상임
- 현재 플레이어가 합법적인 말 이동 없이 턴을 넘기게 됨
- 표준 체스 승패 또는 무승부 판정을 우회함

### 14.7 샘플 카드

샘플 카드는 구현 기준이 아니라 카드 구조 이해를 위한 예시다.

- 일반 카드 예시: 전술 정찰
  - 비용: 1마나
  - 타이밍: 이동 전 또는 이동 후
  - 효과: 상대의 공개 일반 카드와 일반 덱 잔량을 최신 상태로 다시 동기화하고, 자신의 다음 합법 수 후보를 서버 계산 기준으로 요청할 수 있다.
- 레전드 카드 예시: 왕실 친위대
  - 비용: 6마나
  - 추가 조건: 자신의 룩 1개 이상이 생존
  - 타이밍: 이동 전
  - 효과: 이번 턴 동안 지정한 아군 기물 1개가 카드 효과로 제거되지 않는다.

## 15. 마나 시스템

### 15.1 기본 규칙

- 각 플레이어의 시작 마나는 0이다.
- 최대 보유 마나는 10이다.
- 마나는 체스 행동 결과로 획득한다.
- 마나 획득은 서버가 이동 결과를 확정한 뒤 계산한다.

### 15.2 획득량

- 합법 이동: 1마나
- 캡처: 2마나
- 체크: 2마나
- 앙파상: 3마나

한 수가 여러 조건을 동시에 만족하면 모두 합산한다.

예: 이동하면서 캡처하고 체크까지 만들면 1 + 2 + 2 = 5마나를 얻는다.

최대 보유량을 초과하는 마나는 버린다.

## 16. 카드팩 및 컬렉션

### 16.1 골드

- 서비스 재화 이름은 골드다.
- MVP에서는 골드 잔액 필드만 준비한다.
- 카드팩 가격은 0골드로 설정한다.
- 출석 보상, 경기 보상, 골드 구매는 MVP 이후 확장 대상으로 둔다.

### 16.2 카드팩

- MVP에서는 기본 카드팩 1종만 제공한다.
- 카드팩 1개는 카드 3장을 지급한다.
- 각 카드 슬롯은 독립적으로 레전드 여부를 판정한다.
- MVP의 레전드 확률은 각 슬롯당 1%다.
- 따라서 카드팩 1개에서 레전드 카드가 0장, 1장, 2장, 3장 모두 나올 수 있다.
- 레전드가 아닌 슬롯은 일반 카드로 지급한다.
- MVP 카드팩은 0골드로 구매할 수 있다.
- 사용자별 카드팩 개봉은 KST 기준 하루 10회로 제한한다.
- 신규 카드팩 종류 추가는 MVP 이후 확장 대상으로 둔다.

### 16.3 중복 카드

- 이미 보유한 카드를 다시 얻으면 보유 수량이 증가한다.
- 중복 카드를 자동 환급하거나 결과를 보정하지 않는다.
- 덱 편성 시 보유 수량과 덱 중복 제한을 함께 검증한다.

### 16.4 스타터 컬렉션

- 신규 사용자는 즉시 플레이 가능한 스타터 덱을 지급받는다.
- 스타터 덱은 기본 레전드 카드 1장과 일반 카드 5장으로 구성한다.

### 16.5 카드팩 감사 로그

카드팩 개봉 결과는 DB에 기록한다.

- 사용자
- 개봉 시각
- 사용 골드
- 결과 카드 3장
- 각 카드의 일반/레전드 여부
- 확률 버전
- 중복 여부

## 18. 대전 채팅

### 18.1 기본 정책

- 대전 중 두 플레이어는 자유 텍스트 채팅을 사용할 수 있다.
- 채팅 메시지는 경기 채널에 실시간 전달된다.
- 채팅 로그는 7일간 보관한다.
- 차단한 사용자의 채팅은 표시하지 않는다.

### 18.2 금칙어 처리

- 서버는 기본 금칙어 필터를 가진다.
- 금칙어 사전은 DB에 저장한다.
- MVP의 금칙어 매칭은 대소문자 구분 없는 단순 부분 문자열 포함 방식으로 처리한다.
- 공백, 특수문자 우회 탐지, 초성 분리, 고급 패턴 매칭은 MVP 이후 확장 대상으로 둔다.
- 사용자에게 송출되는 채팅 메시지는 금칙어가 마스킹된 내용이다.
- 금칙어는 금칙어 전체 길이만큼 `*`로 치환한다.
- DB에는 신고 검토를 위해 원문 메시지를 저장한다.
- 관리자 화면에서는 원문과 마스킹 결과를 검토할 수 있어야 한다.

## 19. 커뮤니티

### 19.1 접근 권한

- 커뮤니티 조회와 작성은 모두 로그인 사용자만 가능하다.
- 계정정지 사용자는 커뮤니티를 사용할 수 없다.
- 채팅정지 사용자의 커뮤니티 작성 제한 여부는 제재 정책에 따른다.

### 19.2 게시글

게시글은 다음 기능을 제공한다.

- 작성
- 목록 조회
- 상세 조회
- 수정
- 삭제
- 검색
- 신고
- 좋아요 토글

이미지 업로드는 MVP에서 제외한다.

### 19.3 댓글

댓글은 다음 기능을 제공한다.

- 작성
- 조회
- 삭제
- 신고

댓글 좋아요는 MVP에서 제외한다.

### 19.4 검색

- 제목, 내용, 작성자 닉네임을 대상으로 검색한다.
- MVP는 DB 단순 검색을 사용한다.
- 전문 검색 엔진 또는 고급 랭킹은 이후 확장 대상으로 둔다.

### 19.5 좋아요

- 좋아요는 게시글에만 적용한다.
- 로그인 사용자는 게시글마다 좋아요를 한 번 누를 수 있다.
- 이미 누른 좋아요는 취소할 수 있다.

### 19.6 신고 및 자동 숨김

- 게시글과 댓글을 신고할 수 있다.
- 신고가 5건 누적되면 자동 임시 숨김 처리한다.
- 관리자는 신고 내용을 검토하고 숨김 유지, 복구, 사용자 제재를 결정한다.

### 19.7 커뮤니티 금칙어 처리

- 게시글과 댓글 원문은 DB에 저장한다.
- 사용자 화면에는 금칙어가 마스킹된 내용을 표시한다.
- 관리자 검토 시 원문을 확인할 수 있다.
- 마스킹 방식은 대전 채팅 금칙어 처리와 동일하게 금칙어 전체 길이만큼 `*`로 치환한다.

## 20. 신고, 차단, 관리자

### 20.1 사용자 차단

- 사용자는 다른 사용자를 차단할 수 있다.
- 차단 관계가 있는 사용자는 빠른 매치에서 서로 매칭되지 않는다.
- 차단한 사용자의 대전 채팅은 표시하지 않는다.

### 20.2 신고

신고 대상은 다음을 포함한다.

- 대전 채팅
- 커뮤니티 게시글
- 커뮤니티 댓글
- 사용자 프로필

신고는 관리자 검토 대상으로 저장한다.

### 20.3 관리자 기능

MVP는 최소 관리자 기능을 제공한다.

- 신고 목록 조회
- 신고 상세 조회
- 게시글 숨김 및 복구
- 댓글 숨김 및 복구
- 사용자 채팅정지
- 사용자 계정정지
- 카드 목록 조회
- 금칙어 목록 조회
- 금칙어 추가, 수정, 삭제

카드 효과, 확률, 팩 가격 수정은 MVP 관리자 기능에 포함하지 않는다.
관리자 화면은 기능 중심의 단순 목록, 상세, 처리 화면으로 제공한다.

관리자 목록 화면은 MVP에서 다음 검색 및 필터만 제공한다.

- 신고 목록: 상태, 대상 유형, 신고일 범위
- 사용자 제재 관련 검색: 닉네임 또는 사용자 ID
- 카드 목록: 카드명, 카드 종류
- 금칙어 목록: 키워드

복잡한 정렬, 다중 조건 조합, 엑셀 다운로드는 MVP에서 제외한다.

### 20.4 제재

- 채팅정지는 기간제로 적용한다.
- 계정정지도 기간제로 적용한다.
- 계정정지 사용자는 로그인 이후 주요 기능 사용이 제한된다.
- 관리자 권한은 초기에는 DB에서 수동 부여한다.

## 23. 데이터 저장 원칙

### 23.1 경기 상태

- MVP는 단일 백엔드 인스턴스 운영을 전제로 한다.
- 활성 경기 상태는 메모리에 유지한다.
- 주요 이벤트와 스냅샷은 DB에 저장한다.
- 서버 재시작 시 완전한 활성 경기 복구는 MVP 필수 요구사항이 아니다.
- 단, 종료된 경기의 결과와 기보는 DB에 보존되어야 한다.
- 수평 확장, Redis/pub-sub 기반 경기 상태 공유, 다중 인스턴스 STOMP 세션 처리, 분산 타이머는 MVP 이후 확장 대상으로 둔다.

### 23.2 핵심 저장 대상

MVP에서 최소한 다음 도메인 데이터는 영속 저장 대상이다.

- 사용자 계정, 인증 제공자, 이메일 인증 상태, 계정 상태
- 닉네임, 사용자 설정, 차단 관계
- 카드 마스터, 사용자 카드 보유 수량, 덱 구성
- 빠른 매치 결과, 경기 결과, 비공개 MMR
- 기보, 카드 사용 로그, 마나 변화 로그, 주요 상태 변화 로그
- 카드팩 개봉 감사 로그와 확률 버전
- 커뮤니티 게시글, 댓글, 좋아요, 신고, 숨김 상태
- 채팅 로그 원문과 마스킹 결과
- 관리자 제재 이력
- 운영 관측을 위한 로그와 메트릭

### 23.3 개인정보 및 로그

- 채팅 로그는 7일 보관 후 삭제 또는 비식별화한다.
- 신고 검토에 필요한 원문은 보관 기간 안에서 관리자만 접근할 수 있다.
- 탈퇴 사용자의 기록은 탈퇴 사용자로 표시한다.

## 24. 예외 및 엣지 케이스

서버는 다음 예외 상황을 클라이언트 상태와 무관하게 권위적으로 판정하고, 실패 시 상태를 변경하지 않는 응답을 반환해야 한다.

### 24.1 인증 및 권한

- 이메일 미인증 사용자가 주요 기능을 요청하면 거부한다.
- 이메일 미인증 사용자는 이메일 인증 관련 기능과 최소 계정 상태 조회만 사용할 수 있다.
- 계정정지 사용자가 보호 기능을 요청하면 제한 안내와 함께 거부한다.
- 채팅정지 사용자의 채팅 전송은 거부한다.
- 관리자 권한이 없는 사용자의 관리자 API 접근은 거부한다.
- 로그아웃 또는 만료된 세션으로 REST API나 STOMP 명령을 요청하면 거부한다.

### 24.2 매칭 및 경기

- 유효하지 않은 덱으로 빠른 매치 대기열 진입을 요청하면 실패한다.
- 이미 대기열에 있는 사용자가 중복으로 대기열 진입을 요청하면 중복 등록하지 않는다.
- 매칭 확정 후 취소 요청이 도착하면 이미 생성된 경기 상태를 우선한다.
- 차단 관계가 있는 두 사용자는 매칭 후보에서 제외한다.
- 경기 참가자가 아닌 사용자의 경기 상태 조회나 행동 제출은 거부한다.

### 24.3 턴, 카드, 체스 규칙

- 자기 턴이 아닌 사용자의 행동은 거부한다.
- 이동 전 카드 행동, 말 이동, 이동 후 행동 순서를 벗어난 요청은 거부한다.
- 마나가 부족하거나 카드 조건을 만족하지 못한 카드 사용은 실패한다.
- 체크 상태에서 체크 해소에 기여하지 않는 카드 사용은 실패한다.
- 카드 사용 후 최종적으로 킹이 체크 상태라면 해당 행동은 실패한다.
- 카드 효과가 상대 킹을 직접 제거하거나 양쪽 킹 수를 비정상 상태로 만들면 실패한다.
- 카드 효과가 표준 체스 승패 또는 무승부 판정을 우회하면 실패한다.
- 일반 카드 덱이 비어 있을 때 드로우 요청은 실패한다.
- 프로모션이 필요한 상황에서 승격 기물을 선택하지 않으면 다음 단계로 진행하지 않는다.
- 동일 행동 요청이 중복 도착하면 서버가 먼저 확정한 결과만 적용한다.

### 24.4 시간, 연결, 종료

- 개인 시간이 0초가 되면 3초 유예 시간 안의 유효 행동만 인정한다.
- 연결 끊김 후 20초 안에 재접속하지 않으면 몰수패 처리한다.
- 시간패와 몰수패 조건이 동시에 경쟁하면 서버가 먼저 확정한 종료 사유를 적용한다.
- 경기 종료 후 도착한 행동, 카드 사용, 채팅 요청은 경기 상태를 변경하지 않는다.

### 24.5 커뮤니티 및 신고

- 삭제되었거나 숨김 처리된 게시글과 댓글에 대한 수정, 삭제, 좋아요, 신고 요청은 현재 상태 기준으로 검증한다.
- 같은 사용자가 같은 게시글 좋아요를 반복 요청하면 토글 규칙에 따라 한 번만 반영한다.
- 신고 5건 누적 경계에서 동시에 신고가 들어와도 자동 임시 숨김은 한 번만 적용한다.
- 금칙어 필터가 적용된 내용은 사용자 화면에는 마스킹 결과를, 관리자 검토에는 원문을 제공한다.

## 25. 핵심 수용 기준

### 25.1 인증

- 사용자는 이메일/비밀번호로 가입할 수 있다.
- 이메일 인증 전 자체 로그인 사용자는 이메일 인증 관련 기능과 최소 계정 상태 조회만 사용할 수 있다.
- Google 또는 Kakao OAuth로 로그인할 수 있다.
- OAuth 사용자는 서비스 고유 닉네임 설정과 필수 약관 동의 후 주요 기능을 사용할 수 있다.
- 모든 가입 방식에서 서비스 이용약관과 개인정보 처리방침 동의 시각 및 버전이 저장된다.
- 닉네임은 2~16자, 한글/영문/숫자/언더스코어만 허용되며 대소문자 구분 없이 중복을 막는다.
- 인증된 세션으로 REST API와 STOMP WebSocket에 접근할 수 있다.
- 로그아웃 후 기존 세션으로 보호 API를 사용할 수 없다.
- 이메일 인증 토큰과 비밀번호 재설정 토큰은 5분 만료, 1회 사용, 최신 토큰만 유효 정책을 따른다.
- 5회 연속 로그인 실패 시 10분 잠금이 적용되고, 비밀번호 재설정 완료 시 실패 횟수와 잠금이 초기화된다.

### 25.2 매칭

- 유효 덱을 가진 사용자는 빠른 매치 대기열에 진입할 수 있다.
- 유효 덱이 없으면 대기열 진입이 실패한다.
- 두 사용자가 매칭되면 서버가 백/흑을 랜덤 배정하고 경기를 생성한다.
- 차단 관계가 있는 사용자는 서로 매칭되지 않는다.

### 25.3 게임 진행

- 서버는 표준 체스 합법 수만 허용한다.
- 자기 턴에는 이동 전 카드 행동, 필수 말 이동, 이동 후 행동 순서가 지켜진다.
- 이동 후 행동에서는 카드 사용, 드로우, 아무것도 안 함 중 하나만 수행된다.
- 체크 상태에서는 체크 해소에 기여하지 않는 카드 행동이 거부된다.
- 체크메이트, 스테일메이트, 50수 규칙, 3회 반복, 기물 부족이 올바르게 판정된다.

### 25.4 마나 및 카드

- MVP 초기 카드 규모는 일반 카드 10장, 레전드 카드 2장이다.
- 이동, 캡처, 체크, 앙파상 마나가 합산 지급된다.
- 마나는 10을 초과해 저장되지 않는다.
- 레전드 카드는 경기 시작 시 지정된 1장이 지급된다.
- 일반 카드는 시작 시 무작위 1장이 지급된다.
- 일반 카드 손패는 상대에게 공개된다.
- 레전드 카드는 사용 전까지 상대에게 비공개다.
- 카드 효과 적용 후 최종 보드 상태가 체스 규칙을 깨면 카드 사용이 실패한다.

### 25.5 시간 및 재접속

- 각 플레이어는 개인 시계 30분을 가진다.
- 자기 턴 동안에만 시간이 감소한다.
- 턴 시작 시 남은 시간이 10초 미만이면 10초로 보정된다.
- 시간이 0초가 된 뒤 3초 안에 유효 행동이 없으면 시간패 처리된다.
- 연결 끊김 후 20초 안에 복귀하면 경기 상태가 재동기화된다.
- 20초 안에 복귀하지 않으면 몰수패 처리된다.

### 25.6 카드팩

- MVP 카드팩은 기본 카드팩 1종만 제공한다.
- MVP 카드팩은 0골드로 구매할 수 있다.
- 카드팩 1개는 카드 3장을 지급한다.
- 각 카드 슬롯은 독립적으로 레전드 1% 확률을 적용한다.
- 사용자는 KST 기준 하루 10회까지만 카드팩을 개봉할 수 있다.
- 중복 카드는 보유 수량으로 누적된다.
- 카드팩 개봉 로그가 저장된다.

### 25.7 커뮤니티 및 운영

- 로그인 사용자는 게시글과 댓글을 작성할 수 있다.
- 게시글 검색이 동작한다.
- 게시글 좋아요는 토글 가능하다.
- 신고 5건이 누적된 게시글 또는 댓글은 자동 임시 숨김 처리된다.
- 관리자는 신고를 검토하고 숨김, 복구, 제재를 적용할 수 있다.
- 금칙어가 포함된 채팅, 게시글, 댓글은 사용자 화면에 마스킹되어 표시된다.
- 관리자 검토용 DB 데이터에는 원문이 보존된다.
- 금칙어 사전은 DB에 저장되며 관리자는 단순 CRUD를 수행할 수 있다.
- 금칙어 매칭은 대소문자 구분 없는 단순 부분 문자열 포함 방식이며, 금칙어 길이만큼 `*`로 마스킹된다.

## 26. 미결정 사항과 명시적 가정

다음 항목은 MVP 요구사항의 방향은 정해졌지만, 구현 전에 세부 정책 또는 값 확정이 필요하다.

### 26.1 미결정 사항

- 실제 카드명, 상세 효과, 사용 조건, 마나 비용
- 카드와 카드팩의 구체 구현 방식 및 상세 데이터 관리 방식
- 비공개 MMR 초기값, 갱신 공식, 대기 시간에 따른 허용 MMR 확장 폭
- 카드 관리 자동화 또는 AI 기반 카드 객체화 워크플로우

### 26.2 명시적 가정

- MVP 프론트엔드와 백엔드는 같은 사이트로 배포한다.
- MVP는 모바일 앱과 외부 공개 API를 지원하지 않는다.
- MVP 카드팩 가격은 0골드이며 유료 결제는 제공하지 않는다.
- MVP에서 활성 경기의 서버 재시작 후 완전 복구는 필수 요구사항이 아니다.
- MVP는 단일 백엔드 인스턴스 운영을 전제로 한다.
- MVP의 관리자 권한은 초기 DB 수동 부여 방식으로 운영한다.
- 카드 효과는 표준 체스 승패 판정을 무시하지 않는다.

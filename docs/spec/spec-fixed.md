# TCG Online Chess Backend MVP 요구사항 명세

## 1. 개요

TCG Online Chess는 표준 체스 규칙을 기반으로, 경기 중 카드 효과를 제한적으로 결합한 실시간 1:1 온라인 웹 게임이다. 본 문서는 백엔드 MVP 구현을 위한 요구사항 명세이며, React 프론트엔드와 Spring Boot 백엔드, STOMP WebSocket 기반 실시간 통신을 전제로 한다.

기본 언어는 한국어이며, 서비스 운영 기준 시간대는 KST(Asia/Seoul)로 한다.

## 2. 문서 목적과 범위

이 문서는 TCG Online Chess 백엔드 MVP의 구현, 테스트, 검토 기준을 정의한다. 프론트엔드 화면 상세나 운영 배포 자동화 절차가 아니라, 백엔드가 제공해야 하는 도메인 규칙, 인증, 실시간 통신, 데이터 저장, 운영 보호 장치, 수용 기준을 중심으로 작성한다.

MVP 완료 기준은 인증된 사용자가 유효한 덱으로 빠른 매치에 진입하고, 서버 권위의 실시간 1:1 경기를 끝까지 완료하며, 결과와 기보가 서버에 저장되는 것이다.

- 대상 시스템은 Spring Boot 백엔드와 STOMP WebSocket 기반 실시간 게임 서버다.
- React 프론트엔드는 이 문서의 REST API 및 실시간 이벤트 원칙을 사용하는 같은 사이트 클라이언트로 전제한다.
- MVP는 1:1 온라인 PvP 경험을 완성하는 것을 우선한다.
- 서버는 모든 인증, 권한, 게임 상태, 카드 효과, 타이머, 승패, 보상, 기록을 권위적으로 판단한다.
- 클라이언트는 행동 의도와 표시 상태만 전달하며, 클라이언트가 계산한 게임 상태나 결과는 신뢰하지 않는다.
- UI/UX의 구체적 화면 구성, 디자인 시스템, 애니메이션은 이 문서 범위가 아니다.
- 단, 프론트엔드가 반드시 지켜야 하는 API, STOMP 이벤트, 오류 응답, 재접속 원칙은 백엔드 계약으로 포함한다.
- MVP 제외 기능은 구현하지 않는다. 필요한 경우 이후 확장 시 충돌하지 않도록 권한, 데이터, 로그 경계를 남긴다.

## 3. 용어 정의

문서, 코드, API, 대화에서 같은 개념을 다른 단어로 부르지 않도록 다음 표준 용어를 사용한다.

| 표준 용어 | 동의어/이전 표현 | 정의 | 관련 코드/API 용어 |
| --- | --- | --- | --- |
| 사용자 | 회원, 유저 | 서비스 계정을 가진 주체다. 경기 안에서는 플레이어 역할을 가진다. | `User` |
| 플레이어 | 경기 참가자 | 특정 경기에서 백 또는 흑으로 배정된 사용자다. | `Player` |
| 자체 로그인 사용자 | 이메일 로그인 사용자 | 이메일과 비밀번호로 가입한 사용자다. 이메일 인증 완료 전에는 주요 기능 사용이 제한된다. | `LocalAccount` |
| OAuth 사용자 | 소셜 로그인 사용자 | Google 또는 Kakao OAuth 2.0으로 로그인하는 사용자다. 서비스 고유 닉네임 설정과 필수 약관 동의가 필요하다. | `OAuthAccount` |
| 닉네임 | 표시 이름, 유저명 | 서비스 내 공개 식별자다. MVP에서는 가입 후 변경할 수 없다. | `nickname` |
| 빠른 매치 | 매치메이킹, PvP 매칭 | 유효한 덱을 가진 사용자를 비공개 MMR 기준으로 자동 매칭하는 MVP의 유일한 매칭 방식이다. | `QuickMatch` |
| 경기 | 게임, 대국, 매치 | 백/흑 두 플레이어가 하나의 체스 보드와 카드 상태를 공유하며 진행하는 단위다. `Match`는 별도 도메인이 아니라 `Game`과 같은 의미의 비표준 표현으로만 취급한다. | `Game` |
| 턴 | 차례 | 한 플레이어가 턴 시작, 이동 전 카드 행동, 말 이동, 이동 후 행동을 순서대로 수행하는 진행 단위다. | `Turn` |
| 턴 시작 | 시작 단계 | 현재 턴 플레이어에게 턴 시작 효과를 적용하는 단계다. MVP에서는 마나 증가만 수행한다. | `TURN_START` |
| 이동 전 카드 행동 | 선 카드 행동 | 말 이동 전에 카드 1장 사용 또는 아무것도 하지 않음을 선택하는 단계다. | `PRE_MOVE_ACTION` |
| 이동 후 행동 | 후 행동, 이동 후 카드 행동 | 말 이동 후 카드 1장 사용, 일반 카드 1장 드로우, 아무것도 하지 않음 중 하나를 선택하는 단계다. | `POST_MOVE_ACTION` |
| 일반 카드 | 공개 카드 | 경기 중 드로우할 수 있고 상대에게 공개되는 카드다. | `NormalCard` |
| 레전드 카드 | 궁극기 카드, 비공개 카드 | 경기당 1장만 사용 가능한 1회성 카드이며 사용 전까지 상대에게 공개되지 않는다. | `LegendCard` |
| 마나 | 비용 자원 | 카드 사용 비용으로 쓰는 경기 중 자원이며 턴 시작과 체스 행동 결과로 획득한다. | `mana` |
| 덱 | 카드 덱 | 한 경기에서 사용할 레전드 카드 1장과 일반 카드 5장의 조합이다. | `Deck` |
| 유효한 덱 | 사용 가능 덱 | 보유 카드 수량, 카드 종류, 중복 제한, 필수 레전드/일반 카드 수를 모두 만족해 빠른 매치에 사용할 수 있는 덱이다. | `ValidDeck` |
| 손패 | 핸드 | 경기 중 플레이어가 현재 보유한 카드 목록이다. 일반 카드 손패는 상대에게 공개된다. | `Hand` |
| 비공개 MMR | 내부 MMR | 빠른 매치에만 쓰는 서버 내부 실력 점수다. 사용자에게 수치나 티어로 공개하지 않는다. | `mmr` |
| 공개 정보 | 공개 상태 | 양 플레이어에게 모두 전달 가능한 보드, 일반 카드 손패, 마나, 타이머 등 정보다. | `PublicState` |
| 비공개 정보 | 숨김 상태 | 사용 전 레전드 카드, 상대에게 공개하지 않는 내부 난수/시드/로그 등 정보다. | `PrivateState` |
| 기보 | 경기 기록, 수순 기록 | 말 이동, 카드 사용, 드로우, 마나 변화, 주요 상태 변화를 포함한 경기 재현용 기록이다. | `GameRecord` |
| 시간패 | 시간사 | 개인 시계가 0초가 된 뒤 유효 행동이 완료되지 않아 발생하는 패배다. | `TIMEOUT` |
| 몰수패 | 접속 종료 패배 | 연결 끊김 후 제한 시간 안에 재접속하지 못해 발생하는 패배다. | `FORFEIT` |
| 신고 | 리포트 | 대전 채팅, 커뮤니티 게시글, 사용자 프로필을 관리자 검토 대상으로 제출하는 행위다. | `Report` |
| 차단 | 블록 | 특정 사용자와의 매칭 및 채팅 노출을 제한하는 사용자 설정이다. | `Block` |
| 제재 | 정지, 관리자 조치 | 관리자가 사용자에게 적용하는 커뮤니케이션 정지 또는 계정정지 조치다. | `Sanction` |
| 커뮤니케이션 정지 | 채팅정지, 커뮤니티 작성 제한 | 대전 채팅 전송과 커뮤니티 게시글 작성/수정/삭제/신고를 함께 제한하는 MVP의 단일 기능 제재다. | `COMMUNICATION_SUSPENDED` |
| 관리자 | 운영자 | 신고 처리, 숨김/복구, 제재, 금칙어 관리 등 제한된 운영 기능을 수행하는 사용자다. | `ADMIN` |
| 세션 | 로그인 세션 | 서버가 인증 사용자를 식별하기 위해 발급하고 쿠키로 유지하는 상태다. | `Session` |
| 재접속 유예 시간 | 복귀 제한 시간 | 연결 끊김 후 몰수패 전까지 복귀할 수 있는 시간이다. MVP에서는 20초다. | `ReconnectGracePeriod` |
| 감사 로그 | 운영 감사 기록 | 관리자 행위, 카드팩 개봉, 원문 조회, 제재 등 추적이 필요한 행위를 저장한 기록이다. | `AuditLog` |

## 4. 사용자 역할과 권한

기능 접근은 `Role`, `AccountStatus`, 기능 제재를 함께 판단한다. 계정 상태 제한은 관리자 역할보다 우선한다.

### 4.1 역할

| 역할 | 의미 | 허용 기능 |
| --- | --- | --- |
| `USER` | 일반 사용자 | 인증, 빠른 매치, 경기, 카드팩, 컬렉션, 덱, 마이페이지, 커뮤니티 게시글, 신고, 차단 |
| `ADMIN` | 관리자 | 일반 사용자 기능과 신고 검토, 게시글 숨김/복구, 사용자 제재, 카드 목록 조회, 금칙어 관리 |

관리자 권한은 MVP에서 초기 DB 수동 부여를 전제로 한다. 관리자 권한 부여와 회수는 운영 절차로 수행하며, 운영 감사 대상이다.

### 4.2 계정 상태

| 상태 | 의미 | 허용 기능 | 제한 기능 |
| --- | --- | --- | --- |
| `ACTIVE` | 정상 계정. 인증과 필수 설정을 완료해 일반 기능을 사용할 수 있는 상태다. | 일반 사용자 기능 | 관리자 기능은 `ADMIN` 역할 필요 |
| `UNVERIFIED` | 인증 또는 필수 설정 미완료 상태다. 자체 로그인 이메일 미인증, OAuth 닉네임 미설정, 필수 약관 미동의 등을 포함한다. | 로그인, 로그아웃, 이메일 인증 메일 재요청, 비밀번호 재설정, 제한 안내 조회, 최소 계정 상태 조회 | 매칭, 경기, 카드팩, 덱, 마이페이지 주요 기능, 커뮤니티 조회/작성/신고, 대전 채팅 |
| `SUSPENDED` | 계정정지 상태다. 로그인은 가능하지만 제한 안내와 제재 사유/기간 확인만 허용한다. | 로그인, 제한 안내 조회, 로그아웃 | 매칭, 경기, 카드팩, 덱, 커뮤니티, 채팅, 신고, 마이페이지 주요 기능, 관리자 기능 |
| `WITHDRAWN` | 탈퇴 상태다. 로그인할 수 없고 모든 사용자 기능이 차단된다. 기존 기록의 표시명은 탈퇴 사용자로 처리한다. | 없음 | 모든 사용자 기능 |

### 4.3 기능 제재

- MVP의 기능 제재는 커뮤니케이션 정지 하나로 단순화한다.
- 커뮤니케이션 정지를 당하면 대전 채팅 전송과 커뮤니티 게시글 작성/수정/삭제/신고가 모두 제한된다.
- 커뮤니티 작성 제한도 동일한 제재로 취급하며, 별도 제재 종류로 분리하지 않는다.
- 커뮤니케이션 정지는 조회성 기능과 경기 진행 자체를 막지 않는다.
- 계정정지는 커뮤니케이션 정지보다 강한 제한이며 대부분의 보호 기능을 차단한다.
- REST API뿐 아니라 STOMP `CONNECT`, `SUBSCRIBE`, `SEND` 처리에서도 역할, 계정 상태, 제재를 매번 서버가 검증한다.

## 5. 주요 사용자 시나리오

### 5.1 첫 플레이 흐름

1. 사용자는 이메일/비밀번호 또는 OAuth로 가입한다.
2. 자체 로그인 사용자는 이메일 인증을 완료한다.
3. OAuth 사용자는 서비스 고유 닉네임을 설정한다.
4. 모든 가입 방식은 필수 약관 동의를 완료한다.
5. 서버는 신규 사용자에게 스타터 컬렉션과 기본 덱을 멱등적으로 지급한다.
6. 인증과 필수 설정을 완료한 사용자는 기본 덱으로 빠른 매치에 진입할 수 있다.
7. 서버는 빠른 매치를 통해 상대를 찾고 경기를 생성한다.
8. 사용자는 실시간 경기, 재접속, 종료 처리를 경험한다.
9. 종료된 경기 결과, 전적, MMR, 기보가 서버에 저장된다.
10. 사용자는 다음 경기 또는 덱 편성으로 이어갈 수 있다.

### 5.2 빠른 매치와 경기 시작

1. 사용자는 유효한 덱을 보유한 상태로 빠른 매치 대기열에 진입한다.
2. 서버는 계정 상태, 제재, 덱 유효성, 차단 관계, 비공개 MMR을 검증한다.
3. 서버는 한 사용자당 하나의 대기열 항목만 허용한다.
4. 매칭이 확정되면 서버가 경기 생성, 백/흑 무작위 배정, 덱 스냅샷, 초기 상태 생성을 원자적으로 처리한다.
5. 서버는 양쪽 클라이언트에 경기 시작과 초기 상태 스냅샷을 전송한다.

### 5.3 턴 진행과 카드 사용

1. 서버는 턴 시작 단계에서 마나 지급 등 턴 시작 효과를 적용한다.
2. 현재 턴 플레이어는 이동 전 카드 행동을 선택한다.
3. 서버는 카드 조건, 마나, 체크 상태 제한, 최종 보드 합법성을 검증한다.
4. 플레이어는 합법 말 이동을 수행한다.
5. 프로모션이 필요한 경우 플레이어는 즉시 승격 기물을 선택한다.
6. 서버는 이동 결과, 마나 획득, 체크/체크메이트/무승부/승패 여부를 확정한다.
7. 경기가 끝나지 않았다면 플레이어는 이동 후 행동으로 카드 사용, 일반 카드 드로우, 아무것도 하지 않음 중 하나를 선택한다.
8. 이동 후 카드 사용으로 체크메이트가 충족되면 게임은 즉시 종료된다.
9. 서버는 턴 종료 후 다음 플레이어에게 턴을 넘긴다.

### 5.4 카드팩, 컬렉션, 덱 편성

1. 사용자는 MVP에서 기본 카드팩을 0골드로 개봉한다.
2. 서버는 슬롯별 독립 확률로 카드 3장을 지급하고 감사 로그를 남긴다.
3. KST 기준 하루 10회 성공 개봉 제한을 원자적으로 적용한다.
4. 중복 카드는 보유 수량으로 누적한다.
5. 사용자는 보유 카드와 덱 중복 제한을 만족하는 덱을 편성한다.

### 5.5 커뮤니티 신고와 관리자 처리

3. 게시글 신고가 서로 다른 유효 사용자 5명에게 누적되면 자동 숨김 처리한다.
4. 관리자는 신고 내용을 검토하고 숨김 유지, 복구, 사용자 제재를 결정한다.
5. 커뮤니티 댓글과 좋아요는 MVP에서 제공하지 않는다.

## 7. 인증 및 계정

### 7.1 인증 방식

MVP의 웹 인증은 서버 세션과 쿠키를 사용한다.

- 세션 쿠키는 `HttpOnly`, `Secure`, `SameSite=Lax`를 기본값으로 한다.
- 프론트엔드와 백엔드는 같은 사이트 배포를 전제로 한다.
- REST의 상태 변경 요청은 CSRF 검증을 적용한다.
- STOMP WebSocket `CONNECT` 요청도 서버 세션과 CSRF를 검증한다.
- WebSocket 연결이 성공하면 서버는 STOMP 세션에 인증 사용자 식별자를 연결한다.
- WebSocket `Origin`은 허용 목록으로 검증한다.
- 서버는 STOMP `CONNECT`, `SUBSCRIBE`, `SEND`마다 인증, 권한, 계정 상태, 제재를 검증한다.
- 로그인, OAuth 완료, 비밀번호 재설정 완료 후에는 세션 ID를 재발급한다.
- 로그아웃, 탈퇴, 계정정지, 비밀번호 재설정 완료 시 기존 세션과 STOMP 연결을 무효화한다.

JWT Access/Refresh Token 방식은 MVP에서 사용하지 않는다. 향후 모바일 앱, 외부 API, 멀티 클라이언트 지원이 필요해질 때 별도 인증 방식으로 추가 검토한다.

### 7.2 회원가입 및 로그인

- 자체 로그인은 이메일, 비밀번호, 고유 닉네임을 사용한다.
- 닉네임은 서비스 내 공개 식별자이며 가입 후 변경할 수 없다.
- 닉네임은 2~16자이며 한글, 영문, 숫자, 언더스코어만 허용한다.
- 닉네임 입력값의 앞뒤 공백은 제거한다.
- 닉네임 중복은 대소문자를 구분하지 않고 검사한다.
- 금칙어가 포함된 닉네임은 사용할 수 없다.
- 자체 로그인 사용자는 이메일 인증을 완료해야 주요 기능을 사용할 수 있다.
- OAuth 제공자는 Google, Kakao를 지원한다.
- OAuth 계정은 `provider + providerUserId` 조합으로 고유하게 식별한다.
- OAuth 사용자는 별도 이메일 인증을 요구하지 않는다.
- OAuth 사용자는 서비스 고유 닉네임 설정과 필수 약관 동의를 완료해야 주요 기능을 사용할 수 있다.
- 모든 가입 방식은 서비스 이용약관과 개인정보 처리방침 동의를 필수로 요구한다.
- 약관 동의 시각과 약관 버전을 저장한다.

### 7.3 이메일, 비밀번호, 토큰 보안

- 비밀번호는 8~72자로 제한한다.
- 비밀번호는 BCrypt 계열의 안전한 해시 알고리즘으로 저장한다.
- 같은 계정 기준 5회 연속 로그인 실패 시 10분 동안 계정을 잠근다.
- 계정 잠금 중에도 비밀번호 재설정 요청은 허용한다.
- 비밀번호 재설정이 완료되면 로그인 실패 횟수와 계정 잠금을 초기화한다.
- IP 기반 Rate Limit은 계정 잠금과 별도로 적용한다.
- 이메일 인증 토큰과 비밀번호 재설정 토큰은 평문이 아니라 해시로 저장한다.
- 이메일 인증 토큰과 비밀번호 재설정 토큰은 모두 5분 만료 시간을 가진다.
- 이메일 인증 토큰과 비밀번호 재설정 토큰은 1회 사용 후 즉시 무효화한다.
- 새 이메일 인증 토큰을 발급하면 같은 사용자의 기존 미사용 이메일 인증 토큰은 모두 무효화한다.
- 새 비밀번호 재설정 토큰을 발급하면 같은 사용자의 기존 미사용 재설정 토큰은 모두 무효화한다.
- 비밀번호 재설정 요청 응답은 이메일 존재 여부를 노출하지 않는다.
- 이메일 인증 메일과 비밀번호 재설정 메일은 SMTP 인터페이스로 발송한다.
- SMTP 호스트, 포트, 계정, TLS 설정은 환경 변수로 주입한다.
- 이메일 발송은 사용자 요청 API에서 즉시 1회 시도하고, 실패하면 실패 응답을 반환한다.
- MVP에서는 이메일 발송 백그라운드 자동 재시도 큐를 두지 않는다.

### 7.4 OAuth와 동일 이메일 충돌

- MVP에서는 자체 로그인 계정과 OAuth 계정을 자동 병합하지 않는다.
- 동일 이메일의 활성 자체 로그인 계정이 이미 존재하면, 새 OAuth 가입은 차단하고 기존 로그인 방식 안내를 제공한다.
- 동일 이메일이더라도 `provider + providerUserId`가 다르면 같은 계정으로 자동 연결하지 않는다.
- 탈퇴 계정 재활성화는 기존 가입 방식에 맞는 검증을 거쳐 수행한다.
- 설정 화면에서 자체 로그인 계정과 OAuth 계정을 연동하여 같은 사용자 계정으로 로그인할 수 있게 하는 기능은 이후 확장 후보로 둔다.

### 7.5 계정 탈퇴

- 사용자는 계정 탈퇴를 요청할 수 있다.
- MVP에서는 계정을 물리 삭제하지 않고 `WITHDRAWN` 상태로 비활성화한다.
- 탈퇴 사용자는 로그인할 수 없다.
- 전적, 기보, 커뮤니티 기록은 서비스 무결성을 위해 보존하되, 작성자 표시는 탈퇴 사용자로 처리한다.
- 동일 이메일 또는 동일 OAuth 식별자로 재가입하면 새 계정을 만들지 않고 기존 비활성화 계정을 재활성화한다.
- 탈퇴 계정 재활성화 시 기존 닉네임을 그대로 유지한다.

## 8. 실시간 통신

### 8.1 WebSocket 방식

- Spring WebSocket + STOMP를 사용한다.
- 클라이언트는 행동 의도만 서버에 전송한다.
- 서버는 모든 게임 상태, 합법 수, 카드 조건, 타이머, 승패를 권위적으로 계산한다.
- 클라이언트가 보낸 상태 계산 결과는 신뢰하지 않는다.
- 클라이언트는 서버 확정 이벤트를 받기 전까지 행동 결과를 `pending` 상태로 표시할 수 있다.
- 서버 확정 이벤트가 도착하면 클라이언트는 서버 이벤트의 상태 버전을 기준으로 화면 상태를 확정한다.
- 서버가 행동을 거부하면 클라이언트는 pending 표시를 해제하고 서버가 내려준 최신 상태로 되돌린다.

### 8.2 이벤트 envelope

모든 주요 STOMP 이벤트는 최소한 다음 공통 필드를 가진다.

- `eventId`: 서버가 부여한 이벤트 고유 ID
- `eventType`: 이벤트 종류
- `gameId`: 경기 식별자
- `sequence` 또는 `stateVersion`: 경기 내 단조 증가 버전
- `serverTime`: 서버 기준 이벤트 시각
- `payload`: 이벤트별 데이터
- `reasonCode`: 실패 또는 종료 사유가 있을 때 사용하는 코드

서버는 경기별 `sequence` 또는 `stateVersion`을 단조 증가시킨다. 클라이언트는 누락된 이벤트나 버전 불일치를 감지하면 최신 스냅샷을 요청하거나 재동기화한다.

### 8.3 클라이언트 행동 요청

클라이언트가 보내는 게임 행동 요청은 다음 정보를 포함한다.

- `actionId`: 클라이언트가 생성한 멱등성 키
- `gameId`: 대상 경기
- `intent`: 수행하려는 행동 의도
- `baseStateVersion`: 클라이언트가 행동을 결정한 기준 상태 버전
- 행동별 입력값

같은 `actionId`와 같은 payload가 재전송되면 서버는 기존 처리 결과를 반환하거나 중복으로 응답하고, 상태를 다시 변경하지 않는다. 같은 `actionId`에 다른 payload가 오면 충돌로 거부한다.

### 8.4 주요 실시간 이벤트

서버는 최소한 다음 유형의 이벤트를 실시간으로 전달해야 한다.

- 매칭 상태 변경
- 경기 시작
- 현재 게임 상태 스냅샷
- 합법 행동 성공
- 행동 실패 및 실패 사유
- 턴 시작, 턴 변경, 턴 종료
- 서버 기준 타이머 상태
- 카드 사용 결과
- 카드 드로우 결과
- 체크, 체크메이트, 무승부 판정, 시간패, 몰수패, 항복
- 채팅 메시지
- 재접속 상태
- 경기 종료

### 8.5 스냅샷과 델타

- 서버는 경기 시작, 재접속, 이벤트 누락 감지 시 전체 스냅샷을 제공한다.
- 델타 이벤트는 허용하지만 서버 스냅샷이 최종 권위 상태다.
- 공개 정보는 경기 채널로 전달할 수 있다.
- 사용 전 레전드 카드 등 비공개 정보는 개인 채널로만 전달한다.
- 타이머 이벤트는 `serverTime`, `turnDeadlineAt`, `remainingMillis`처럼 서버 기준 값을 포함한다.

## 9. API 또는 인터페이스 요구사항

백엔드는 REST API와 STOMP WebSocket을 함께 제공한다. REST API는 조회, 설정, 관리, 비실시간 명령에 사용하고, STOMP WebSocket은 경기 중 실시간 상태 변화와 플레이어 행동 처리에 사용한다.

### 9.1 REST API 원칙

- REST API는 서버 세션 기반 인증을 사용한다.
- 상태 변경 요청은 CSRF 검증을 통과해야 한다.
- 사용자, 인증, 카드 컬렉션, 덱, 마이페이지, 전적, 기보, 커뮤니티, 신고, 관리자 기능은 REST API로 제공한다.
- REST API는 리소스 소유권, 역할, 계정 상태, 제재를 서버에서 검증한다.
- REST API 응답은 클라이언트가 화면을 구성하는 데 필요한 서버 확정 상태를 반환한다.
- 클라이언트가 계산한 게임 상태, 카드 결과, 승패 결과는 REST API와 STOMP 명령 모두에서 신뢰하지 않는다.

### 9.2 STOMP WebSocket 원칙

- STOMP `CONNECT`는 서버 세션, CSRF, `Origin` 검증을 통과해야 한다.
- 경기 참가, 행동 제출, 채팅 전송, 재접속 상태 동기화는 STOMP 메시지로 처리한다.
- 클라이언트는 행동 의도만 전송하고, 서버는 검증된 결과 이벤트를 전송한다.
- 서버는 사용자가 구독할 수 있는 경기 채널과 개인 응답 채널을 분리한다.
- 경기 채널에는 공개 확정 상태만 전송한다.
- 비공개 정보와 개인 오류는 개인 채널로 전송한다.
- 개인 채널 라우팅은 클라이언트가 보낸 userId가 아니라 인증 세션 기준으로 수행한다.
- 실패한 행동은 도메인 상태를 변경하지 않고 실패 사유 이벤트로 응답한다.
- 허용되지 않은 destination에 대한 `SUBSCRIBE` 또는 `SEND`는 거부한다.

### 9.3 오류 응답 원칙

- REST와 STOMP는 동일한 `errorCode` 체계를 사용한다.
- 사용자에게 전달하는 오류 메시지는 내부 구현, SQL, 스택 트레이스, 보안 정책 세부를 노출하지 않는다.
- 상세 원인은 서버 로그와 감사 로그에 남긴다.
- 실패 응답은 가능한 경우 최신 서버 상태 또는 `stateVersion`을 포함해 클라이언트가 재동기화할 수 있게 한다.

### 9.4 인터페이스 안정성

- MVP 단계에서 외부 공개 API 호환성은 보장하지 않는다.
- 프론트엔드와 백엔드는 같은 제품 릴리스 단위로 함께 변경할 수 있다.
- 단, 도메인 이벤트 이름과 주요 상태 필드는 테스트에서 검증 가능하도록 일관되게 유지한다.
- REST 계약은 OpenAPI 또는 이에 준하는 방식으로 관리한다.
- STOMP 이벤트와 요청은 JSON Schema, fixture, 계약 테스트 등으로 검증 가능해야 한다.
- API/STOMP handler는 인증, 권한, 입력 변환, 오류 포맷을 담당하고, 게임 규칙과 카드 효과는 도메인 또는 애플리케이션 서비스에서 처리한다.

## 10. 매치메이킹

### 10.1 빠른 매치

- MVP는 빠른 매치만 제공한다.
- 친선 코드, 랭크 모드, 관전 모드는 제공하지 않는다.
- 사용자는 유효한 덱을 보유해야 빠른 매치 대기열에 진입할 수 있다.
- 사용자당 활성 빠른 매치 대기열 항목은 하나만 허용한다.
- 이미 대기열에 있는 사용자가 다시 진입을 요청하면 새 항목을 만들지 않고 현재 대기 상태를 반환한다.
- 매칭 전에는 사용자가 대기열을 취소할 수 있다.
- 매칭 확정 후 취소 요청은 실패하거나 무시되며, 이미 생성된 경기 상태가 우선한다.

### 10.2 대기열 상태

빠른 매치 대기열은 최소한 다음 상태를 가진다.

- `WAITING`: 상대를 기다리는 상태
- `CANCELED`: 사용자 요청 또는 조건 불만족으로 취소된 상태
- `MATCH_CONFIRMED`: 양 플레이어 매칭이 확정된 상태
- `GAME_CREATED`: 경기 생성이 완료된 상태
- `FAILED`: 서버 오류 또는 최종 검증 실패로 매칭이 실패한 상태

### 10.3 매칭 기준

- 사용자는 비공개 단일 MMR을 가진다.
- 빠른 매치는 MMR이 가까운 상대를 우선한다.
- 대기 시간이 길어질수록 허용 MMR 범위를 점진적으로 넓힌다.
- 비공개 MMR 초기값, 갱신 공식, 대기 시간별 허용 MMR 확장 폭은 구현 전 확정이 필요하다.
- 경기 결과에 따라 비공개 MMR을 갱신한다.
- 공개 티어나 랭크 표시는 MVP에 포함하지 않는다.
- 차단 관계가 있는 두 사용자는 빠른 매치에서 서로 매칭되지 않는다.
- 차단 관계는 후보 탐색 시점과 매칭 확정 직전에 모두 검사한다.
- MVP에서는 대기 시간이 길어져도 차단 관계를 무시하지 않는다.

### 10.4 매칭 확정과 경기 생성

- 대기열 진입 시점과 매칭 확정 직전에 계정 상태, 커뮤니케이션 외 제재, 덱 유효성, 차단 관계를 다시 검증한다.
- 매칭 확정, 경기 생성, 백/흑 배정, 덱 스냅샷, 초기 상태 생성은 하나의 원자적 서버 처리로 수행한다.
- 경기가 생성되면 선택된 덱 스냅샷을 고정한다.
- 경기 생성 후 사용자가 덱을 수정해도 진행 중 경기에는 영향을 주지 않는다.

### 10.5 색 배정

- 서버가 경기 시작 시 서버 난수로 백/흑을 무작위 배정한다.
- MVP에서는 최근 색 이력 보정이나 공평성 누적 보정을 하지 않는다.
- 백이 먼저 둔다.

## 11. 체스 룰

### 11.1 기본 규칙

- 보드는 표준 8x8 체스 보드다.
- 기물 종류, 수량, 초기 배치는 표준 체스를 따른다.
- 기본 승리 조건은 체크메이트다.
- 킹을 직접 잡는 방식으로 승리하지 않는다.
- 자기 턴에는 합법 수 하나를 반드시 둬야 한다.
- 둘 수 있는 합법 수가 없을 때는 체크메이트 또는 스테일메이트로 판정한다.

### 11.2 특수 수

MVP는 표준 체스 특수 규칙을 모두 지원한다.

- 캐슬링
- 앙파상
- 프로모션

프로모션은 표준 말 이동으로 폰이 끝 행에 도달한 경우에만 발생한다. 프로모션이 필요한 말 이동 직후 플레이어는 퀸, 룩, 비숍, 나이트 중 하나를 즉시 선택해야 한다.

카드 효과로 폰이 끝 행에 도달하더라도 프로모션은 발생하지 않는다.

카드 효과로 특정 폰이 앙파상으로 잡힐 수 있는 위치에 도달한 경우, 서버가 앙파상 가능 상태를 기록하면 다음 적용 가능한 말 이동 기회에 앙파상을 허용한다. 카드 효과로 만들어진 앙파상 가능 상태는 다음 적용 가능한 말 이동 기회에만 유효하다.

카드 효과가 킹 또는 룩의 이동 이력에 영향을 줄 수 있는 경우, 캐슬링 권리는 보수적으로 제거한다.

### 11.3 무승부

다음 표준 체스 무승부 조건을 지원한다.

- 스테일메이트
- 기물 부족
- 50수 규칙
- 3회 반복

카드 효과로 보드 상태가 바뀌면 해당 변화도 보드 이력에 반영해 50수 규칙과 3회 반복 판단에 사용한다.

플레이어 간 합의 무승부는 MVP에서 제외한다.

### 11.4 체크, 체크메이트, 게임 종료

- 말 이동 결과뿐 아니라 카드 효과 적용 후의 보드 결과도 체크, 체크메이트, 스테일메이트, 무승부 판단 대상이다.
- 이동 전 카드 행동으로 체크메이트가 충족되면 말 이동 단계로 가지 않고 즉시 게임을 종료한다.
- 말 이동으로 체크메이트 또는 무승부가 충족되면 이동 후 행동으로 가지 않고 즉시 게임을 종료한다.
- 이동 후 카드 사용으로 체크메이트가 충족되면 즉시 게임을 종료한다.
- 카드 효과는 표준 체스 승패와 무승부 판정을 우회할 수 없다.

### 11.5 항복

- 플레이어는 경기 중 항복할 수 있다.
- 항복은 경기 참가자만 요청할 수 있다.
- 항복은 확인 단계를 거친 뒤 확정된다.
- 항복이 확정되면 즉시 해당 플레이어의 패배로 종료한다.
- 중복 항복 요청은 멱등적으로 처리한다.

### 11.6 체스 룰 엔진과 상태 저장

- 서버는 체스 룰 검증에 검증된 Java 라이브러리를 사용한다.
- 기본 후보는 `bhlangonijr/chesslib`이다.
- 카드 효과는 체스 라이브러리의 기본 합법 수 검증을 감싸는 별도 도메인 계층에서 처리한다.
- 보드 상태는 FEN 또는 이에 준하는 표준 표현으로 저장할 수 있다.
- 카드, 마나, 턴 단계, 타이머, 비공개 정보는 보드 상태와 분리해 서버 상태로 관리한다.

## 12. 턴 진행

한 턴은 다음 순서로 진행한다.

1. 턴 시작
2. 이동 전 카드 행동 대기
3. 말 이동 대기
4. 프로모션 선택 대기
5. 이동 후 행동 대기
6. 턴 종료
7. 경기 종료

`프로모션 선택 대기`는 표준 말 이동으로 프로모션이 필요한 경우에만 발생한다. `경기 종료`는 어느 단계에서든 종료 조건이 확정되면 즉시 진입한다.

### 12.1 턴 시작

- 턴 시작 단계에서는 현재 턴 플레이어에게 턴 시작 효과를 적용한다.
- MVP의 턴 시작 효과는 현재 턴 플레이어의 마나를 1 증가시키는 것이다.
- 단, 최초 턴 플레이어의 최초 턴에는 턴 시작 마나를 지급하지 않는다.
- 마나 증가에는 최대 보유량 10 제한을 적용한다.
- 턴 시작 처리는 서버가 자동으로 확정하며 중복 처리하지 않는다.

### 12.2 이동 전 카드 행동

플레이어는 다음 중 하나를 선택한다.

- 카드 1장 사용
- 아무것도 하지 않음

이 단계에서 사용할 수 있는 카드는 카드 조건, 마나, 체크 상태 제한, 최종 보드 합법성을 모두 만족해야 한다.

아무것도 하지 않음을 선택하면 이동 전 카드 행동 단계는 소비되며 같은 턴의 이 단계로 돌아갈 수 없다.

이동 전 카드 행동 후 게임이 끝나지 않았다면 말 이동은 필수다.

### 12.3 말 이동

- 말 이동은 항상 필수다.
- 서버는 이동의 합법성을 검증한다.
- 이동 결과 체크, 체크메이트, 무승부, 캡처, 프로모션, 앙파상 여부를 판정한다.
- 프로모션이 필요한 경우 승격 기물 선택이 완료되어야 다음 단계로 넘어간다.
- 말 이동으로 게임 종료 조건이 충족되면 이동 후 행동 없이 즉시 종료한다.

### 12.4 프로모션 선택

- 프로모션은 표준 말 이동으로 폰이 끝 행에 도달한 경우에만 발생한다.
- 프로모션 선택은 말 이동 후 즉시 수행해야 한다.
- 프로모션 선택이 완료되기 전에는 이동 후 행동, 턴 종료, 상대 행동을 처리하지 않는다.
- 프로모션 선택 후 최종 보드를 기준으로 체크, 체크메이트, 무승부를 판정한다.

### 12.5 이동 후 행동

플레이어는 다음 중 하나를 선택한다.

- 카드 1장 사용
- 일반 카드 1장 드로우
- 아무것도 하지 않음

이동 후 행동은 하나만 수행할 수 있다. 카드 사용 또는 드로우 또는 패스가 성공적으로 확정되면 턴을 종료한다.

일반 카드 덱이 비어 있을 때 드로우를 요청하면 실패하며, 이동 후 행동 슬롯은 소비하지 않는다.

### 12.6 체크 상태의 카드 제한

- 자신의 킹이 체크 상태일 때는 체크 해소에 기여하는 카드만 사용할 수 있다.
- 카드 사용 후에도 최종적으로 자신의 킹이 체크 상태라면 해당 행동은 불법이다.
- 일반 카드와 레전드 카드 모두 이 제한을 따른다.
- 이동 전 카드 행동에서 체크가 발생하거나 해소되는 경우도 최종 보드 기준으로 판단한다.

### 12.7 동시성 및 멱등성

- 모든 턴 명령은 `actionId` 또는 이에 준하는 멱등성 키를 포함한다.
- 같은 단계에서 동시에 요청이 도착하면 서버가 먼저 커밋한 요청만 적용한다.
- 나머지 요청은 중복 또는 단계 불일치로 실패한다.
- 실패한 카드 사용, 드로우, 말 이동은 보드, 손패, 덱, 마나, 턴 단계, 결과를 변경하지 않는다.

## 13. 시간 규칙

### 13.1 기본 시간

- 각 플레이어는 개인 시계 30분을 가진다.
- 자기 턴 동안에만 자신의 시간이 감소한다.
- 상대 턴 동안에는 자신의 시간이 감소하지 않는다.
- 턴 시작, 이동 전 카드 행동, 말 이동, 프로모션 선택, 이동 후 행동은 모두 현재 턴 플레이어의 시간을 소비한다.
- 클라이언트 시계는 신뢰하지 않으며, 모든 시간 판정은 서버 기준 시간으로 수행한다.
- 서버는 deadline 기반으로 시간을 계산한다.

### 13.2 시간패

- 개인 시간이 0초가 되면 3초 유예 시간을 적용한다.
- 0초 도달 후 3000ms 이내에 유효 행동이 서버에서 완료되면 해당 행동을 인정한다.
- 3000ms를 초과하면 시간패를 확정한다.
- 시간패 판정은 서버 커밋 시각 또는 서버 처리 완료 시각을 기준으로 한다.
- 시간패가 확정되면 이후 복귀나 지연 도착 행동으로 되돌릴 수 없다.

### 13.3 연결 끊김과 재접속

- 서버가 해당 사용자의 모든 활성 WebSocket 연결 종료를 감지하면 연결 끊김으로 본다.
- 같은 사용자의 다른 탭 또는 다른 기기가 아직 연결되어 있으면 연결 끊김으로 보지 않는다.
- 연결 끊김 후 20초 안에 재접속하면 서버는 최신 경기 상태를 재동기화한다.
- 연결 끊김 중에도 해당 플레이어의 턴이면 시간은 계속 감소한다.
- 연결 끊김 중 상대 턴이면 해당 플레이어의 개인 시계는 감소하지 않는다.
- 연결 끊김 후 복귀했더라도 이미 시간이 초과되어 시간패가 확정되었으면 시간패는 그대로 유지한다.
- 연결 끊김 후 20초 안에 재접속하지 못하면 몰수패를 확정한다.

### 13.4 종료 사유 경쟁

- 시간패와 몰수패 조건이 경쟁하면 실제 deadline이 더 이른 종료 사유를 적용한다.
- 동일 시각으로 판단되면 시간패를 우선한다.
- 경기 종료 사유는 원자적으로 한 번만 확정한다.

## 14. 카드 시스템

### 14.1 카드 종류

- MVP 초기 카드 규모는 일반 카드 10장, 레전드 카드 2장이다.
- 일반 카드는 공개형 전술 보조 수단이다.
- 레전드 카드는 비공개 1회성 변수다.
- 실제 MVP 카드 12장의 카드명, 상세 효과, 사용 조건, 마나 비용은 구현 전 별도 확정해야 한다.

### 14.2 덱 구성

- 한 덱은 레전드 카드 1장과 일반 카드 5장으로 구성한다.
- 덱은 사용자가 보유한 카드 수량 안에서만 구성할 수 있다.
- 동일 일반 카드의 덱 내 중복 가능 여부와 최대 중복 수는 카드 상세 정책에서 확정한다.
- 빠른 매치 진입과 경기 생성 직전 모두 덱 유효성을 검증한다.
- 경기 생성 시 덱 스냅샷을 고정한다.

### 14.3 경기 시작 카드 상태

- 경기 시작 시 각 플레이어는 선택한 레전드 카드 1장을 가진다.
- 경기 시작 시 각 플레이어는 일반 카드 덱에서 무작위 일반 카드 1장을 손패로 받는다.
- 일반 카드 덱 순서는 서버가 경기 시작 시 셔플하고 고정한다.
- 클라이언트는 드로우 결과, 셔플 결과, 난수 결과를 제출할 수 없다.
- 일반 카드 손패는 상대에게 공개한다.
- 레전드 카드는 사용 전까지 상대에게 공개하지 않는다.

### 14.4 공개 정보와 비공개 정보

- 상대에게 공개되는 일반 카드 정보는 이름, 비용, 효과 설명, 사용 조건 등 의사결정에 필요한 정보를 포함한다.
- 남은 일반 카드 덱은 장수만 공개하고, 남은 구성과 순서는 공개하지 않는다.
- 사용 전 레전드 카드는 상대 응답, 상대 이벤트, 상대 재접속 스냅샷에 포함하지 않는다.
- 사용되지 않은 상대 레전드 카드는 경기 종료 후 기보에서도 공개하지 않는다.

### 14.5 카드 사용

- 이동 전 카드 행동에서 최대 1장, 이동 후 행동에서 최대 1장을 사용할 수 있다.
- 한 턴에 사용할 수 있는 카드는 최대 2장이다.
- 카드 사용은 카드 보유, 카드 타이밍, 대상 조건, 추가 조건, 마나, 체크 상태, 최종 보드 합법성을 모두 검증한다.
- 카드 사용이 성공하면 카드 소비, 마나 차감, 보드/상태 변경, 이벤트 저장이 원자적으로 처리된다.
- 카드 사용이 실패하면 카드, 마나, 보드, 손패, 덱, 행동 슬롯을 소비하지 않는다.

### 14.6 카드 효과 불변식

카드 효과는 다음 규칙을 깨뜨릴 수 없다.

- 양쪽 킹은 정상적으로 존재해야 한다.
- 상대 킹을 직접 제거할 수 없다.
- 자신의 킹이 체크 상태로 남는 결과를 만들 수 없다.
- 표준 체스의 체크메이트, 스테일메이트, 무승부 판정을 우회할 수 없다.
- 카드 효과 후 체크메이트가 되면 즉시 게임이 종료된다.
- 카드 효과로 폰이 끝 행에 도달해도 프로모션은 발생하지 않는다.
- 카드 효과가 앙파상 가능 상태를 만들 수 있으며, 서버가 이를 기록하면 다음 적용 가능한 말 이동에서만 유효하다.
- 카드 효과가 캐슬링 권리에 영향을 줄 수 있으면 보수적으로 캐슬링 권리를 제거한다.

### 14.7 드로우

- 이동 후 행동에서 일반 카드 1장을 드로우할 수 있다.
- 드로우는 서버가 고정된 일반 카드 덱 순서에 따라 처리한다.
- 일반 카드 덱이 비어 있으면 드로우 요청은 실패한다.
- 실패한 드로우는 이동 후 행동 슬롯을 소비하지 않는다.

### 14.8 구현 방식

- MVP에서는 카드별 서버 핸들러와 공통 검증 파이프라인을 사용한다.
- 범용 카드 DSL, 관리자 편집형 카드 효과, AI 기반 카드 객체화 워크플로우는 MVP 구현 필수가 아니다.
- 카드 효과 실행 후에는 공통 검증 파이프라인으로 체스 불변식과 게임 종료 조건을 재검증한다.

## 15. 마나 시스템

### 15.1 기본 규칙

- 각 플레이어의 시작 마나는 0이다.
- 최대 보유 마나는 10이다.
- 최대 보유량을 초과하는 마나는 버린다.
- 마나는 서버가 계산하며 클라이언트 입력을 신뢰하지 않는다.
- 모든 마나 변화는 원자적으로 적용하고 변화 사유를 기록한다.

### 15.2 획득 규칙

- 최초 턴 플레이어의 최초 턴에는 턴 시작 마나를 받지 않는다.
- 그 이후 모든 턴 시작마다 현재 턴 플레이어는 마나 1을 얻는다.
- 합법적인 말 이동 자체는 마나를 주지 않는다.
- 기물을 잡으면 이동한 플레이어가 마나 1을 얻는다.
- 앙파상은 캡처 보상 1에 추가 앙파상 보상 1을 더해 총 2마나를 얻는다.
- 체크를 건 플레이어는 마나를 얻지 않는다.
- 각 플레이어는 자신이 체크를 2회 당할 때마다 마나 1을 얻는다.
- 체크메이트도 체크를 당한 1회로 계산한 뒤 게임을 종료한다.
- 한 행동에서 더블 체크가 발생해도 체크를 당한 횟수는 1회로 계산한다.
- 카드 효과로 발생한 체크도 체크를 당한 횟수에 포함한다.

### 15.3 예시

- 턴 시작 마나 1, 캡처 1이면 총 2마나를 얻는다.
- 앙파상 캡처는 캡처 1과 앙파상 추가 1을 합산해 총 2마나를 얻는다.
- 어떤 플레이어가 체크를 두 번째로 당하면, 체크를 당한 플레이어가 마나 1을 얻는다.
- 체크를 건 플레이어는 체크 자체로 마나를 얻지 않는다.

## 16. 카드팩 및 컬렉션

### 16.1 골드

- 서비스 재화 이름은 골드다.
- MVP에서는 골드 잔액 필드를 미래 확장용으로 둘 수 있다.
- MVP 기본 카드팩 가격은 0골드다.
- 출석 보상, 경기 보상, 골드 구매는 MVP 이후 확장 대상으로 둔다.

### 16.2 카드팩

- MVP에서는 기본 카드팩 1종만 제공한다.
- 카드팩 1개는 카드 3장을 지급한다.
- 각 카드 슬롯은 독립적으로 레전드 여부를 판정한다.
- MVP의 레전드 확률은 각 슬롯당 1%다.
- 따라서 카드팩 1개에서 레전드 카드가 0장, 1장, 2장, 3장 모두 나올 수 있다.
- 레전드 슬롯은 활성 레전드 카드 중 균등 무작위로 지급한다.
- 일반 슬롯은 활성 일반 카드 중 균등 무작위로 지급한다.
- MVP 카드팩은 0골드로 개봉할 수 있다.
- 사용자별 카드팩 개봉은 KST 기준 하루 10회 성공 개봉으로 제한한다.
- 일일 제한은 KST 00:00:00에 초기화한다.
- 신규 카드팩 종류 추가는 MVP 이후 확장 대상으로 둔다.

### 16.3 개봉 처리

- 카드팩 개봉 결과는 서버가 결정한다.
- 클라이언트는 카드 ID, 확률, 결과를 제출할 수 없다.
- 카드팩 개봉은 일일 카운터 증가, 골드 확인/차감, 카드 지급, 감사 로그 저장을 하나의 트랜잭션으로 처리한다.
- 0골드 카드팩이라도 동시 요청으로 하루 10회 성공 제한을 넘을 수 없어야 한다.
- 중복 요청을 식별할 수 있도록 요청 ID 또는 이에 준하는 멱등성 키를 사용할 수 있다.

### 16.4 중복 카드

- 이미 보유한 카드를 다시 얻으면 보유 수량이 증가한다.
- 중복 카드를 자동 환급하거나 결과를 보정하지 않는다.
- 천장, pity, 확률 보정은 MVP에서 제공하지 않는다.
- 덱 편성 시 보유 수량과 덱 중복 제한을 함께 검증한다.

### 16.5 스타터 컬렉션

- 신규 사용자는 즉시 플레이 가능한 스타터 컬렉션과 기본 덱을 지급받는다.
- 스타터 덱은 기본 레전드 카드 1장과 일반 카드 5장으로 구성한다.
- 스타터 지급은 멱등적으로 처리한다.
- 이미 지급된 사용자에게 중복 지급하지 않는다.

### 16.6 카드팩 감사 로그

카드팩 개봉 결과는 DB에 감사 로그로 기록한다.

- `requestId`
- 사용자
- 카드팩 종류
- 개봉 시각
- KST 기준 날짜
- 사용 골드와 개봉 전/후 골드
- 결과 카드 3장과 슬롯 순서
- 각 카드의 일반/레전드 여부
- 확률 버전
- 개봉 전/후 보유 수량
- 중복 여부
- 실패한 경우 실패 사유

## 17. 전적 및 기보

### 17.1 전적 요약

마이페이지에서는 자신의 전적만 제공한다.

- 총 종료 경기 수
- 승리 수
- 패배 수
- 무승부 수
- 승률
- 최근 경기 목록

승률은 승리 수를 종료 경기 수로 나눈 값이다. 무승부는 분모에 포함한다.

최근 경기 목록은 종료 시각 내림차순으로 정렬하고 페이지네이션한다.

최근 경기 목록은 최소한 다음 정보를 포함한다.

- 상대 닉네임
- 백/흑 여부
- 경기 시작 시각
- 경기 종료 시각
- 종료 사유
- 결과

전적에는 서버가 최종 종료한 경기만 반영한다. 진행 중, 취소, 미시작, 무효 경기는 제외한다.

### 17.2 경기 종료 저장

- 경기 종료 트랜잭션은 `GameEnded` 이벤트, 경기 결과, 플레이어 전적, MMR 갱신, 기보 봉인을 원자적으로 처리한다.
- 경기 결과, 종료 사유, 종료 시각은 서버만 확정한다.
- 서버 재시작으로 활성 경기가 유실되어 `SERVER_ABORTED` 또는 `NO_RESULT` 처리된 경기는 전적과 MMR에 반영하지 않는다.

### 17.3 기보 저장

- 기보의 원천은 append-only `GameEvent`다.
- `GameSnapshot`은 재접속, 이벤트 누락 복구, 재생 최적화를 위한 보조 저장소다.
- 스냅샷은 이벤트 원천을 대체하지 않는다.
- 말 이동, 카드 사용, 카드 드로우, 마나 변화, 턴 단계, 보드 상태, 종료 이벤트를 저장한다.
- 기보는 MVP에서 해당 경기 당사자만 조회할 수 있다.
- 관리자가 기보를 조회하는 권한은 MVP에서 별도로 부여하지 않는다.
- 공개 기보 기능은 이후 확장 대상으로 둔다.

### 17.4 기보 공개 범위

- 기보 응답은 조회자 기준 projection을 사용한다.
- 사용되지 않은 상대 레전드 카드는 경기 종료 후에도 공개하지 않는다.
- RNG seed, 내부 로그, 비공개 MMR, IP, 세션, 차단 관계는 기보 응답에 포함하지 않는다.
- 탈퇴 사용자는 탈퇴 사용자로 표시한다.
- 차단 관계가 있어도 과거 경기 기보 접근 권한은 제거하지 않는다.
- 단, 기보 응답에서 차단 여부를 노출하지 않는다.

## 18. 대전 채팅

### 18.1 기본 정책

- 대전 중 두 플레이어는 자유 텍스트 채팅을 사용할 수 있다.
- 채팅 전송과 구독은 경기 참가자만 가능하다.
- 서버는 클라이언트가 보낸 senderId, 닉네임, 권한 값을 신뢰하지 않는다.
- 커뮤니케이션 정지 사용자는 채팅을 전송할 수 없다.
- 계정정지 사용자는 채팅을 포함한 주요 기능을 사용할 수 없다.
- 경기 종료 후 채팅 전송은 거부한다.
- 채팅 로그는 일반 보관본 기준 7일간 보관한다.

### 18.2 차단과 전달

- 채팅은 수신자별 개인 채널로 전달한다.
- 서버는 차단한 사용자의 채팅을 차단자에게 전달하지 않는다.
- 차단 여부는 송신자에게 노출하지 않는다.
- 차단은 채팅 저장 여부와 신고 증거 저장 정책을 우회하지 않는다.

### 18.3 메시지 형식과 제한

- 채팅은 plain text만 허용한다.
- 빈 문자열과 공백만 있는 메시지는 거부한다.
- 채팅 메시지 최대 길이는 100자다.
- 사용자 화면과 관리자 화면 모두 HTML escape를 적용한다.
- Rate Limit은 사용자+경기 기준 1초 3건, 10초 10건을 적용한다.

### 18.4 금칙어 처리

- 서버는 기본 금칙어 필터를 가진다.
- 금칙어 사전은 DB에 저장한다.
- MVP의 금칙어 매칭은 대소문자 구분 없는 단순 부분 문자열 포함 방식으로 처리한다.
- 중첩 금칙어는 긴 단어 우선으로 처리한다.
- 모든 금칙어 출현 위치를 마스킹한다.
- 공백, 특수문자 우회 탐지, 초성 분리, Unicode confusable, 고급 패턴 매칭은 MVP 이후 확장 대상으로 둔다.
- 사용자에게 송출되는 채팅 메시지는 금칙어가 마스킹된 내용이다.
- 금칙어는 금칙어 전체 길이만큼 `*`로 치환한다.
- DB에는 원문과 마스킹 결과를 모두 저장한다.
- 관리자 원문 조회는 권한 검사를 거치고 감사 로그를 남긴다.

### 18.5 신고 증거

채팅 신고 증거 스냅샷은 일반 채팅 로그 보관 기간과 별도로 관리한다.

- `messageId`
- `gameId`
- 발신자
- 신고자
- 원문
- 마스킹 결과
- 발신 시각
- 신고 사유

신고 증거는 신고 처리 완료 후 30일이 지나면 삭제 또는 비식별화한다.

## 19. 커뮤니티

### 19.1 접근 권한

- 커뮤니티 조회와 작성은 모두 로그인 사용자만 가능하다.
- 계정정지 사용자는 커뮤니티를 사용할 수 없다.
- 커뮤니케이션 정지 사용자는 게시글 작성, 수정, 삭제, 신고를 할 수 없다.
- 커뮤니티 댓글, 게시글 좋아요, 댓글 좋아요는 MVP에서 제외한다.

### 19.2 게시글

게시글은 다음 기능을 제공한다.

- 작성
- 목록 조회
- 상세 조회
- 수정
- 삭제
- 검색
- 신고
- 자동 숨김
- 관리자 숨김/복구

이미지 업로드는 MVP에서 제외한다.

### 19.3 게시글 상태

게시글은 최소한 다음 상태를 가진다.

- `ACTIVE`: 정상 노출
- `DELETED`: 작성자 또는 관리자가 삭제
- `AUTO_HIDDEN`: 신고 기준 충족으로 자동 숨김
- `MANUAL_HIDDEN`: 관리자가 수동 숨김

숨김 또는 삭제된 게시글은 일반 목록과 검색에서 제외한다. 직접 접근은 거부하거나 숨김 안내만 제공한다.

### 19.4 작성, 수정, 삭제

- 게시글 작성자는 자신의 게시글을 수정하거나 삭제할 수 있다.
- 관리자는 게시글을 숨김 또는 복구할 수 있다.
- 사용자와 관리자 화면 모두 plain text를 HTML escape하여 표시한다.
- 제목과 본문은 원문과 마스킹 결과를 저장한다.
- 사용자 API는 마스킹 결과만 반환한다.
- 금칙어 사전 변경은 기존 게시글에 자동 소급 적용하지 않는다.
- 게시글 생성 또는 수정 시점의 금칙어 사전을 기준으로 마스킹한다.

### 19.5 검색

- 제목, 내용, 작성자 닉네임을 대상으로 검색한다.
- MVP는 DB 단순 검색을 사용한다.
- 검색 결과는 `ACTIVE` 게시글만 포함한다.
- 사용자 API는 마스킹 결과만 반환한다.
- 전문 검색 엔진 또는 고급 랭킹은 이후 확장 대상으로 둔다.

### 19.6 신고 및 자동 숨김

- 신고 대상은 게시글만이다.
- 한 사용자는 같은 게시글을 한 번만 신고할 수 있다.
- 숨김 또는 삭제된 게시글은 신고할 수 없다.
- 서로 다른 유효 사용자 5명이 신고하면 게시글은 `AUTO_HIDDEN` 상태가 된다.
- 자동 숨김은 한 번만 적용한다.
- 신고 수와 임계치 진행 상황은 일반 사용자에게 노출하지 않는다.
- 신고 저장, 신고 수 계산, 자동 숨김 처리는 원자적으로 수행한다.
- 허위 신고 자동 제재, 이의 제기 절차는 MVP에서 제외한다.

### 19.7 커뮤니티 금칙어 처리

- 게시글 원문은 DB에 저장한다.
- 사용자 화면에는 금칙어가 마스킹된 내용을 표시한다.
- 관리자 검토 시 원문을 확인할 수 있다.
- 관리자 원문 조회는 권한 검사를 거치고 감사 로그를 남긴다.
- 마스킹 방식은 대전 채팅 금칙어 처리와 동일하게 금칙어 전체 길이만큼 `*`로 치환한다.

## 20. 신고, 차단, 관리자

### 20.1 사용자 차단

- 사용자는 다른 사용자를 차단할 수 있다.
- 자기 자신은 차단할 수 없다.
- 중복 차단 요청은 멱등적으로 처리한다.
- 사용자는 자신이 차단한 목록만 조회하고 관리할 수 있다.
- 자신을 차단한 사용자 목록은 조회할 수 없다.
- 차단 해제를 제공한다.
- 차단 관계가 있는 사용자는 빠른 매치에서 서로 매칭되지 않는다.
- 매칭 후보 탐색과 매칭 확정 직전에 모두 차단 관계를 검사한다.
- 차단한 사용자의 대전 채팅은 서버가 차단자에게 전달하지 않는다.
- 차단 여부는 상대에게 노출하지 않는다.

### 20.2 신고

MVP 신고 대상은 다음으로 제한한다.

- 대전 채팅
- 커뮤니티 게시글
- 사용자 프로필

커뮤니티 댓글 신고는 MVP에서 제외한다.

- 채팅 신고는 `chatMessageId`와 신고 증거 스냅샷을 기준으로 저장한다.
- 게시글 신고는 사용자당 게시글당 한 번만 허용한다.
- 게시글 신고 5건은 서로 다른 유효 사용자 기준으로 계산한다.
- 사용자 프로필 신고는 관리자 검토 대상으로만 저장하며 자동 숨김은 없다.
- 숨김, 삭제, 만료된 대상에는 추가 신고를 허용하지 않는다.
- 신고 대상자에게 신고자 정보나 차단 관계를 노출하지 않는다.

### 20.3 관리자 기능

MVP는 최소 관리자 기능을 제공한다.

- 신고 목록 조회
- 신고 상세 조회
- 게시글 숨김 및 복구
- 사용자 커뮤니케이션 정지
- 사용자 계정정지
- 사용자 제재 이력 조회
- 카드 목록 조회
- 금칙어 목록 조회
- 금칙어 추가, 수정, 삭제

카드 효과, 확률, 팩 가격 수정은 MVP 관리자 기능에 포함하지 않는다.

관리자 화면은 기능 중심의 단순 목록, 상세, 처리 화면으로 제공한다.

관리자 목록 화면은 MVP에서 다음 검색 및 필터만 제공한다.

- 신고 목록: 상태, 대상 유형, 신고일 범위
- 사용자 제재 관련 검색: 닉네임 또는 사용자 ID
- 카드 목록: 카드명, 카드 종류
- 금칙어 목록: 키워드

복잡한 정렬, 다중 조건 조합, 엑셀 다운로드는 MVP에서 제외한다.

### 20.4 제재

- 커뮤니케이션 정지는 기간제로 적용한다.
- 계정정지도 기간제로 적용한다.
- 제재는 시작 시각, 종료 시각, 사유, 처리 관리자, 출처 신고 ID를 저장한다.
- 제재는 REST 요청, STOMP 메시지, 기존 세션에 즉시 반영한다.
- 계정정지 사용자는 로그인 이후 제한 안내 외 주요 기능 사용이 제한된다.
- 커뮤니케이션 정지는 채팅 전송과 커뮤니티 게시글 작성/수정/삭제/신고를 함께 제한한다.
- 관리자 권한은 초기에는 DB에서 수동 부여한다.

### 20.5 관리자 감사

다음 관리자 행위는 감사 로그를 남긴다.

- 신고 상세 조회와 원문 조회
- 게시글 숨김 및 복구
- 커뮤니케이션 정지
- 계정정지
- 제재 해제
- 금칙어 추가, 수정, 삭제
- 카드 관련 관리자 조회

금칙어는 빈 값, 중복 값, 과도하게 긴 값을 거부한다. 금칙어 변경 시 변경 사유와 변경 전/후 값을 저장한다.

## 21. 마이페이지 및 설정

### 21.1 마이페이지

마이페이지는 별도 핵심 도메인이 아니라 사용자 자신의 데이터 조회와 설정 변경 허브다. 각 기능은 해당 도메인의 기존 권한과 projection 규칙을 따른다.

마이페이지는 다음 기능을 제공한다.

- 내 카드 컬렉션 조회
- 덱 편성 및 변경
- 현재 유효 덱 확인
- 전적 요약 조회
- 최근 경기 조회
- 당사자 기보 조회
- 사용자 설정 변경

모든 마이페이지와 설정 API는 현재 세션 사용자 기준으로만 동작한다. 임의 `userId`로 다른 사용자 정보를 조회하거나 수정할 수 없다.

닉네임 변경은 제공하지 않는다.

### 21.2 사용자 설정

사용자는 다음 설정만 변경할 수 있다.

- 대전 채팅 표시 여부
- 인앱 알림 수신 여부
- 이메일 알림 수신 여부

알 수 없는 필드나 mass assignment 필드는 거부한다. `role`, `accountStatus`, `email`, `nickname`, `mmr` 같은 민감 필드는 설정 API로 변경할 수 없다.

대전 채팅 표시 여부는 로컬 표시 선호도다. 채팅 전송 권한, 서버 저장, 신고, 차단 필터에는 영향을 주지 않는다.

인앱 알림 수신 여부는 선택 알림에만 적용한다. 제재, 보안, 관리자 중요 공지 등 필수 안내는 수신 설정과 무관하게 전달할 수 있다.

이메일 알림 수신 여부는 향후 서비스 이메일에 적용한다. 이메일 인증과 비밀번호 재설정 메일에는 적용하지 않는다.

### 21.3 알림

- MVP의 서비스 알림은 인앱 알림 중심으로 제공한다.
- MVP 인앱 알림 대상은 신고 처리 결과, 관리자 공지, 제재/제한 안내다.
- 커뮤니티 댓글과 좋아요 알림은 MVP에서 제외한다.
- 자체 로그인 이메일 인증과 비밀번호 재설정은 이메일로 처리한다.
- 계정정지 사용자는 주요 마이페이지와 설정 기능을 사용할 수 없다.
- 탈퇴 사용자는 로그인할 수 없고 알림도 받을 수 없다.

### 21.4 이후 설정 확장

- 설정 화면에서 자체 로그인 계정과 OAuth 계정을 연동하는 기능은 MVP 이후 확장 후보로 둔다.
- 연동 기능이 추가되면 자체 로그인과 연동된 OAuth 로그인 모두 같은 사용자 계정으로 로그인되어야 한다.

## 22. 운영 및 보호 장치

### 22.1 Rate Limit

다음 민감 기능에는 사용자, IP, 또는 도메인 키 단위 요청 제한을 적용한다.

- 로그인: IP + 로그인 식별자
- 회원가입: IP + 이메일
- 이메일 인증 요청: IP + 이메일
- 비밀번호 재설정 요청: IP + 이메일
- 채팅 전송: 사용자 + 경기, 1초 3건, 10초 10건
- 게시글 작성: 사용자 기준과 보조 IP 기준
- 카드팩 개봉: 사용자 + KST 날짜 기준 하루 10회와 동시 요청 방지
- STOMP 인증 실패, 과도한 `SEND`, 과도한 메시지 크기

짧은 abuse 방어용 Rate Limit은 MVP에서 인메모리로 구현할 수 있으며, 서버 재시작 시 초기화되어도 허용한다. 카드팩 일일 제한은 DB 원자 카운터로 관리한다.

REST Rate Limit 초과는 `429`와 `Retry-After`를 반환한다. STOMP Rate Limit 초과는 개인 오류 이벤트로 반환한다.

### 22.2 관측

서버는 다음 관측 기능을 제공한다.

- 구조화 애플리케이션 로그
- 에러 로그
- Spring Actuator health endpoint
- Prometheus 메트릭
- Grafana 대시보드 연동을 위한 메트릭 제공

애플리케이션은 Prometheus가 scrape할 수 있는 `/actuator/prometheus` 메트릭 엔드포인트를 제공한다. Prometheus 서버와 Grafana 대시보드의 실제 배포 구성은 운영 환경별 설정으로 둔다.

### 22.3 Actuator 접근

- `/actuator/health`는 최소 상태만 외부 접근을 허용한다.
- 상세 health 정보는 내부망 또는 인증된 관리자/모니터링 주체만 접근할 수 있어야 한다.
- `/actuator/prometheus`는 외부 공개 접근을 제한하고, 내부망 또는 허용된 모니터링 주체만 접근할 수 있어야 한다.
- 민감 actuator endpoint는 외부 공개하지 않는다.

### 22.4 로그와 보안

구조화 로그는 최소한 trace/request ID, path, status, duration, 오류 코드를 포함한다.

로그에는 다음 값을 남기지 않는다.

- 비밀번호
- 인증 토큰
- 비밀번호 재설정 토큰
- 이메일 인증 토큰
- 세션 쿠키
- CSRF 토큰
- 이메일 원문
- 채팅/게시글 원문
- RNG seed

관리자 원문 조회, 제재, 숨김/복구, 금칙어 변경, Rate Limit 차단, 권한 거부는 감사 또는 보안 로그 대상이다.

### 22.5 주요 메트릭

최소한 다음 메트릭을 수집한다.

- 현재 WebSocket 연결 수: gauge
- 매칭 대기열 사용자 수: gauge
- 진행 중 경기 수: gauge
- 경기 생성 수: counter
- 경기 종료 사유별 카운트: counter
- STOMP 메시지 처리 실패 수: counter
- 로그인 실패 수: counter
- Rate Limit 차단 수: counter
- 카드팩 개봉 수: counter
- 카드팩 일일 제한 초과 시도 수: counter
- 채팅 전송 차단 수: counter
- 신고 생성 수: counter
- 관리자 제재 수: counter
- API 응답 시간: timer/histogram
- 서버 에러 수: counter

메트릭 label에는 사용자 ID, 이메일, 게임 ID, IP, 토큰 등 민감하거나 고카디널리티인 값을 넣지 않는다.

## 23. 데이터 저장 원칙

### 23.1 경기 상태 저장

- MVP는 단일 백엔드 인스턴스와 단일 writer 운영을 전제로 한다.
- 활성 경기 상태는 빠른 처리와 타이머/턴 검증을 위해 서버 메모리에 유지한다.
- 확정된 경기 이벤트는 `GameEvent`로 DB에 append-only 저장한다.
- `GameEvent`는 기보, 재생, 감사, 버그 분석, 이벤트 누락 복구의 원천이다.
- `GameSnapshot`은 재접속, 누락 이벤트 복구, 재생 최적화를 위한 보조 저장소다.
- 스냅샷은 이벤트 원천을 대체하지 않는다.
- 서버 메모리는 현재 상태의 빠른 처리를 위해 사용하고, DB 이벤트 저장은 이후 기보와 확정 기록을 위해 사용한다.

### 23.2 서버 재시작과 활성 경기

- 서버 재시작 시 완전한 활성 경기 복구는 MVP 필수 요구사항이 아니다.
- 서버 재시작 또는 프로세스 crash로 활성 메모리 경기가 유실되면 해당 경기는 `SERVER_ABORTED` 또는 `NO_RESULT`로 무효 처리한다.
- 무효 처리된 경기는 전적과 MMR에 반영하지 않는다.
- 종료된 경기의 결과와 기보는 DB에 보존되어야 한다.
- 수평 확장, Redis/pub-sub 기반 경기 상태 공유, 다중 인스턴스 STOMP 세션 처리, 분산 타이머, 활성 경기 외부 저장소 복구는 MVP 이후 확장 대상으로 둔다.

### 23.3 핵심 저장 대상

MVP에서 최소한 다음 도메인 데이터는 영속 저장 대상이다.

- 사용자 계정, 인증 제공자, 이메일 인증 상태, 계정 상태
- 닉네임, 사용자 설정, 차단 관계
- 역할, 제재 이력, 관리자 감사 로그
- 카드 마스터, 사용자 카드 보유 수량, 덱 구성
- 카드팩 개봉 감사 로그와 확률 버전
- 빠른 매치 결과, 경기 결과, 비공개 MMR
- `GameEvent`, `GameSnapshot`, 기보 봉인 상태
- 커뮤니티 게시글, 게시글 신고, 게시글 숨김 상태
- 채팅 로그 원문과 마스킹 결과
- 신고 증거 스냅샷
- 운영 관측을 위한 로그와 메트릭

커뮤니티 댓글과 좋아요 저장은 MVP에서 제외한다.

### 23.4 개인정보, 원문, 보관 기간

- 일반 채팅 로그는 7일 보관 후 삭제 또는 원문 null 처리한다.
- 신고 증거 스냅샷은 신고 처리 완료 후 30일이 지나면 삭제 또는 비식별화한다.
- 게시글 원문은 관리자 검토와 금칙어 감사 필요에 따라 보관한다.
- 관리자 원문 조회는 권한 검사를 거치고 감사 로그를 남긴다.
- 탈퇴 사용자의 기록은 탈퇴 사용자로 표시하되 내부 FK는 기록 무결성을 위해 보존할 수 있다.
- 사용되지 않은 상대 레전드처럼 내부에는 저장되지만 조회자에게 숨겨야 하는 정보는 projection에서 항상 마스킹한다.

### 23.5 카드 마스터와 난수

- 카드 마스터, 카드팩 마스터, 확률 버전의 저장/배포/버전 관리 방식은 구현 전 확정해야 한다.
- 카드팩 RNG는 서버에서만 수행한다.
- 카드팩 감사 로그에는 결과와 확률 버전을 저장한다.
- RNG seed는 일반 로그, API 응답, 기보 응답에 노출하지 않는다.

## 24. 예외 및 엣지 케이스

서버는 다음 예외 상황을 클라이언트 상태와 무관하게 권위적으로 판정한다. 실패 시 도메인 상태를 변경하지 않는 응답을 반환해야 한다.

도메인 상태가 변경되지 않는다는 것은 보드, 손패, 덱, 마나, 턴 단계, 타이머 결과, 경기 결과, `GameEvent`가 변경되지 않는다는 의미다. 단, Rate Limit 카운터, 보안 감사 로그, 실패 응답 이벤트, 멱등성 기록은 저장될 수 있다.

### 24.1 인증 및 권한

- 이메일 미인증 또는 필수 설정 미완료 사용자가 주요 기능을 요청하면 거부한다.
- 계정정지 사용자가 보호 기능을 요청하면 제한 안내와 함께 거부한다.
- 커뮤니케이션 정지 사용자의 채팅 전송과 커뮤니티 게시글 작성/수정/삭제/신고는 거부한다.
- 관리자 권한이 없는 사용자의 관리자 API 접근은 거부한다.
- 로그아웃 또는 만료된 세션으로 REST API나 STOMP 명령을 요청하면 거부한다.
- STOMP `CONNECT`, `SUBSCRIBE`, `SEND`는 인증, 권한, 계정 상태, 제재를 모두 검증한다.
- 로그아웃, 계정정지, 역할 회수는 기존 WebSocket 연결에도 반영되어야 하며 필요 시 연결을 종료한다.

### 24.2 매칭 및 경기

- 유효하지 않은 덱으로 빠른 매치 대기열 진입을 요청하면 실패한다.
- 이미 대기열에 있는 사용자가 중복으로 대기열 진입을 요청하면 중복 등록하지 않는다.
- 매칭 확정 후 취소 요청이 도착하면 이미 생성된 경기 상태를 우선한다.
- 대기열 진입 후 덱을 수정해 경기 생성 직전 덱이 유효하지 않으면 매칭 또는 경기 생성을 실패 처리한다.
- 새 차단 관계가 매칭 확정 직전에 생기면 경기 생성을 취소한다.
- 경기 참가자가 아닌 사용자의 경기 상태 조회나 행동 제출은 거부한다.
- 경기 종료 후 도착한 행동 요청은 종료 상태를 반환하고 도메인 상태를 변경하지 않는다.

### 24.3 턴, 카드, 체스 규칙

- 자기 턴이 아닌 사용자의 행동은 거부한다.
- 턴 시작 중복 처리는 거부한다.
- 최초 턴 플레이어의 최초 턴에는 턴 시작 마나를 지급하지 않는다.
- 이동 전 카드 행동, 말 이동, 프로모션 선택, 이동 후 행동 순서를 벗어난 요청은 거부한다.
- 마나가 부족하거나 카드 조건을 만족하지 못한 카드 사용은 실패한다.
- 체크 상태에서 체크 해소에 기여하지 않는 카드 사용은 실패한다.
- 카드 사용 후 최종적으로 킹이 체크 상태라면 해당 행동은 실패한다.
- 카드 효과가 상대 킹을 직접 제거하거나 양쪽 킹 수를 비정상 상태로 만들면 실패한다.
- 카드 효과가 표준 체스 승패 또는 무승부 판정을 우회하면 실패한다.
- 이동 전 카드 사용, 말 이동, 이동 후 카드 사용으로 체크메이트가 발생하면 즉시 게임을 종료한다.
- 말 이동으로 체크메이트 또는 무승부가 발생하면 이동 후 행동은 생략한다.
- 카드 효과로 만들어진 앙파상 가능 상태는 다음 적용 가능한 말 이동 기회에만 유효하다.
- 카드 효과로 폰이 끝 행에 도달해도 프로모션은 발생하지 않는다.
- 프로모션이 필요한 상황에서 승격 기물을 선택하지 않으면 다음 단계로 진행하지 않는다.
- 일반 카드 덱이 비어 있을 때 드로우 요청은 실패한다.
- 실패한 카드 사용 또는 드로우는 마나, 카드, 행동 슬롯, 보드 상태를 소비하지 않으며 같은 단계에서 재시도할 수 있다.

### 24.4 시간, 연결, 종료

- 개인 시간이 0초가 되면 3초 유예 시간 안의 유효 행동만 인정한다.
- 연결 끊김 후 20초 안에 재접속하지 않으면 몰수패 처리한다.
- 연결 끊김 중 시간이 초과되어 시간패가 확정되면, 이후 재접속해도 결과를 되돌리지 않는다.
- 시간패와 몰수패 조건이 동시에 경쟁하면 서버가 먼저 도래한 deadline을 적용한다.
- 동일 시각이면 시간패를 우선한다.
- 종료 조건 경쟁은 경기 lock 또는 command queue 안에서 처리하며 종료 이벤트는 한 번만 저장한다.
- 서버 재시작으로 활성 경기가 유실되면 `SERVER_ABORTED` 또는 `NO_RESULT` 처리하고 전적/MMR에 반영하지 않는다.

### 24.5 커뮤니티 및 신고

- 삭제되었거나 숨김 처리된 게시글에 대한 수정, 삭제, 신고 요청은 현재 상태 기준으로 검증한다.
- 댓글과 좋아요 관련 요청은 MVP에서 제공하지 않으므로 거부한다.
- 같은 사용자가 같은 게시글을 중복 신고하면 기존 신고를 반환하거나 중복 오류로 처리한다.
- 신고 5건 누적 경계에서 동시에 신고가 들어와도 자동 숨김은 한 번만 적용한다.
- 금칙어 필터가 적용된 내용은 사용자 화면에는 마스킹 결과를, 관리자 검토에는 원문을 제공한다.
- Rate Limit 초과, actuator 접근 거부, 카드팩 일일 제한 초과는 공통 오류 체계로 응답한다.

### 24.6 명령 동시성과 이벤트 저장

- REST와 STOMP 변경 명령은 가능한 경우 `commandId` 또는 `actionId`, `gameId`, `turnNo`, `phase`, `baseStateVersion`을 검증한다.
- 같은 `commandId`와 같은 payload는 기존 결과를 반환한다.
- 같은 `commandId`에 다른 payload가 오면 `COMMAND_CONFLICT`로 거부한다.
- 경기 명령은 경기별 lock 또는 command queue로 직렬화한다.
- `GameEvent` 저장에 실패하면 해당 명령은 확정하지 않는다.
- 성공한 도메인 변경만 `GameEvent`에 저장한다.
- 거부된 명령은 별도 command/audit/security log에 기록할 수 있다.

### 24.7 공통 오류 코드

공통 오류 코드는 최소한 다음을 포함한다.

## 25. 핵심 수용 기준

### 25.1 인증, 계정, 세션

- 사용자는 이메일/비밀번호로 가입할 수 있다.
- 이메일 인증 전 자체 로그인 사용자는 이메일 인증 관련 기능과 최소 계정 상태 조회만 사용할 수 있다.
- Google 또는 Kakao OAuth로 로그인할 수 있다.
- OAuth 사용자는 서비스 고유 닉네임 설정과 필수 약관 동의 후 주요 기능을 사용할 수 있다.
- 동일 이메일의 활성 자체 로그인 계정이 있으면 OAuth 신규 가입은 자동 병합되지 않고 차단된다.
- 모든 가입 방식에서 서비스 이용약관과 개인정보 처리방침 동의 시각 및 버전이 저장된다.
- 닉네임은 2~16자, 한글/영문/숫자/언더스코어만 허용되며 대소문자 구분 없이 중복을 막는다.
- 세션 쿠키는 `HttpOnly`, `Secure`, `SameSite=Lax`를 사용한다.
- REST 상태 변경 요청과 STOMP `CONNECT`는 CSRF를 검증한다.
- 로그아웃, 탈퇴, 계정정지, 비밀번호 재설정 후 기존 세션과 STOMP 연결이 무효화된다.

### 25.2 계정 상태와 제재

- `UNVERIFIED`, `SUSPENDED`, `WITHDRAWN` 상태별 허용/거부 기능이 테스트 가능하다.
- 계정정지 사용자는 로그인 후 제한 안내 외 주요 기능을 사용할 수 없다.
- 탈퇴 사용자는 로그인할 수 없다.
- 커뮤니케이션 정지는 채팅 전송과 커뮤니티 게시글 작성/수정/삭제/신고를 함께 제한한다.
- 제재는 기존 세션과 STOMP 메시지 처리에 즉시 반영된다.

### 25.3 첫 플레이와 매칭

- 신규 사용자는 스타터 컬렉션과 기본 덱을 멱등적으로 지급받는다.
- 유효 덱을 가진 사용자는 빠른 매치 대기열에 진입할 수 있다.
- 유효 덱이 없으면 대기열 진입이 실패한다.
- 한 사용자는 활성 대기열 항목을 하나만 가진다.
- 중복 대기열 진입은 현재 상태를 반환하고 중복 등록하지 않는다.
- 매칭 확정 전 취소는 가능하고, 매칭 확정 후 취소는 경기 상태가 우선한다.
- 두 사용자가 매칭되면 서버가 백/흑을 랜덤 배정하고 경기를 원자적으로 생성한다.
- 차단 관계가 있는 사용자는 서로 매칭되지 않는다.
- 경기 생성 후 덱 수정은 진행 중 경기에 영향을 주지 않는다.

### 25.4 게임 진행, 체스, 카드

- 서버는 표준 체스 합법 수만 허용한다.
- 턴은 턴 시작, 이동 전 카드 행동, 말 이동, 프로모션 선택, 이동 후 행동, 턴 종료 순서를 따른다.
- 최초 턴 플레이어의 최초 턴에는 턴 시작 마나를 받지 않는다.
- 이동 전 카드 행동 후 게임이 끝나지 않으면 말 이동이 필수다.
- 프로모션은 표준 말 이동 후 즉시 선택해야 한다.
- 카드 효과로 폰이 끝 행에 도달해도 프로모션은 발생하지 않는다.
- 카드 효과로 앙파상 가능 상태가 만들어지면 다음 적용 가능한 말 이동에서만 유효하다.
- 이동 전 카드 사용, 말 이동, 이동 후 카드 사용 모두 체크메이트를 만들 수 있고 즉시 게임을 종료한다.
- 체크메이트, 스테일메이트, 50수 규칙, 3회 반복, 기물 부족이 올바르게 판정된다.
- 카드 효과 적용 후 최종 보드 상태가 체스 규칙을 깨면 카드 사용이 실패한다.

### 25.5 마나

- 시작 마나는 양 플레이어 모두 0이다.
- 마나는 10을 초과해 저장되지 않는다.
- 턴 시작 마나, 캡처 마나, 앙파상 추가 마나가 서버에서 계산된다.
- 체크를 건 플레이어는 마나를 얻지 않는다.
- 체크를 2회 당할 때마다 체크를 당한 플레이어가 마나 1을 얻는다.
- 카드 효과로 발생한 체크도 체크를 당한 횟수에 포함한다.
- 모든 마나 변화는 서버 이벤트와 기보에 사유와 함께 기록된다.

### 25.6 시간과 재접속

- 각 플레이어는 개인 시계 30분을 가진다.
- 자기 턴 동안에만 시간이 감소한다.
- 모든 턴 단계는 현재 턴 플레이어의 시간을 소비한다.
- 시간이 0초가 된 뒤 3초 안에 유효 행동이 없으면 시간패 처리된다.
- 연결 끊김 후 20초 안에 복귀하면 경기 상태가 재동기화된다.
- 연결 끊김 중에도 본인 턴이면 시간은 계속 감소한다.
- 복귀 전에 시간패가 확정되었으면 복귀해도 결과는 되돌리지 않는다.
- 20초 안에 복귀하지 않으면 몰수패 처리된다.

### 25.7 카드팩과 컬렉션

- MVP 카드팩은 기본 카드팩 1종만 제공한다.
- MVP 카드팩은 0골드로 개봉할 수 있다.
- 카드팩 1개는 카드 3장을 지급한다.
- 각 카드 슬롯은 독립적으로 레전드 1% 확률을 적용한다.
- 사용자는 KST 기준 하루 10회까지만 카드팩을 성공 개봉할 수 있다.
- 동시 요청으로 하루 10회 제한을 초과할 수 없다.
- 중복 카드는 보유 수량으로 누적된다.
- 카드팩 개봉 감사 로그가 저장된다.

### 25.8 전적과 기보

- 종료된 경기만 전적에 반영된다.
- 경기 종료는 결과, 전적, MMR, `GameEnded`, 기보 봉인을 원자적으로 처리한다.
- 기보는 `GameEvent` 기반으로 저장된다.
- 경기 참가자만 자신의 경기 기보를 조회할 수 있다.
- 사용되지 않은 상대 레전드는 기보에서도 공개되지 않는다.
- 서버 재시작으로 무효 처리된 활성 경기는 전적과 MMR에 반영되지 않는다.

### 25.9 채팅, 커뮤니티, 신고

- 채팅 메시지는 최대 100자다.
- 채팅은 사용자+경기 기준 Rate Limit을 적용한다.
- 차단한 사용자의 채팅은 서버가 차단자에게 전달하지 않는다.
- 커뮤니케이션 정지 사용자는 채팅과 커뮤니티 게시글 작성/수정/삭제/신고를 할 수 없다.
- 커뮤니티는 게시글 작성, 목록, 상세, 수정, 삭제, 검색, 신고, 자동 숨김을 제공한다.
- 커뮤니티 댓글과 좋아요는 MVP에서 제공하지 않는다.
- 게시글 신고 5건이 서로 다른 유효 사용자에게 누적되면 자동 숨김 처리된다.
- 금칙어가 포함된 채팅과 게시글은 사용자 화면에 마스킹되어 표시된다.
- 관리자 원문 조회는 권한 검사와 감사 로그를 거친다.

### 25.10 관리자, 운영, 데이터

- 관리자는 신고 검토, 게시글 숨김/복구, 커뮤니케이션 정지, 계정정지, 금칙어 CRUD를 수행할 수 있다.
- 카드 효과, 확률, 팩 가격 관리자 편집은 MVP에서 제공하지 않는다.
- Rate Limit, actuator 접근 제한, Prometheus 메트릭, 구조화 로그가 적용된다.
- 로그와 메트릭은 비밀번호, 토큰, 쿠키, CSRF, 이메일 원문, 채팅/게시글 원문, RNG seed를 포함하지 않는다.
- 일반 채팅 로그는 7일 보관 후 삭제 또는 비식별화한다.
- 신고 증거 스냅샷은 신고 처리 완료 후 30일 뒤 삭제 또는 비식별화한다.
- 공통 오류 코드는 REST와 STOMP에서 일관되게 사용된다.
- 명령 멱등성, 중복 요청, 동시 신고, 동시 카드팩 개봉, 동시 게임 종료 경쟁이 테스트된다.

## 26. 구현 전 필수 확정 사항과 MVP 확정 제약

다음 항목은 MVP 방향은 정해졌지만, 구현 착수 또는 테스트 작성 전에 세부 정책 또는 값 확정이 필요하다.

### 26.1 구현 전 필수 확정 사항

- MVP 카드 12장의 상세 정의
  - 카드명
  - 카드 종류
  - 마나 비용
  - 사용 가능 타이밍
  - 대상 조건
  - 추가 조건
  - 효과
  - 실패 조건
  - UI 표시 문구
  - 테스트 fixture
- 비공개 MMR 수치 정책
  - 초기 MMR
  - 승/패/무승부/시간패/몰수패별 갱신 여부
  - 갱신 공식
  - 대기 시간별 허용 MMR 범위 확장 주기와 상한
  - 서버 장애 무효 경기의 MMR 미반영 기준
- 카드/카드팩 마스터 데이터 관리 방식
  - DB seed, migration, 정적 파일 중 저장 방식
  - 카드 버전과 확률 버전 관리
  - 배포 후 카드 효과 수정 가능 여부
  - 기존 기보 재현성 보장 방식
  - 카드팩 확률 데이터 배포와 감사 로그 연결 방식

### 26.2 MVP 확정 제약 요약

- MVP 프론트엔드와 백엔드는 같은 사이트로 배포한다.
- MVP는 서버 세션 쿠키 인증을 사용하며 모바일 앱과 외부 공개 API용 JWT를 지원하지 않는다.
- MVP 카드팩 가격은 0골드이며 유료 결제, 환불, 골드 구매는 제공하지 않는다.
- MVP에서 활성 경기의 서버 재시작 후 완전 복구는 필수 요구사항이 아니다.
- 서버 재시작으로 유실된 활성 경기는 `SERVER_ABORTED` 또는 `NO_RESULT`로 무효 처리하고 전적/MMR에 반영하지 않는다.
- MVP는 단일 백엔드 인스턴스와 단일 writer 운영을 전제로 한다.
- MVP의 관리자 권한은 초기 DB 수동 부여 방식으로 운영한다.
- 카드 효과는 표준 체스 승패 판정을 무시하지 않는다.
- 커뮤니티는 게시글 중심이며 댓글과 좋아요는 MVP에서 제외한다.
- 카드 관리 자동화 또는 AI 기반 카드 객체화 워크플로우는 MVP 구현 차단 항목이 아니라 이후 확장 후보로 둔다.

## 27. MVP 제외 및 이후 확장 후보

아래 항목은 MVP 범위, 수용 기준, 테스트 범위에 포함하지 않는다. 추후 별도 요구사항 정의와 보안/운영 검토 후 확장한다. 아래 확장 후보는 현재 확정한 단일 인스턴스, 세션 쿠키, 고정 카드 데이터, 게시글 중심 커뮤니티, 제한된 관리자 기능을 대체하지 않는다.

### 27.1 게임/소셜 확장

- 솔로플레이 봇
- 친구 대전 및 초대 코드
- 친구/팔로우
- 관전
- 공개 기보

관전과 공개 기보를 추가할 때는 사용 전 레전드, 손패, 덱 구성, 비공개 MMR 등 숨김 정보가 노출되지 않는 viewer projection이 필요하다.

### 27.2 랭크/보상/경제 확장

- 공개 랭크 티어
- 시즌
- 리더보드
- 출석 보상
- 경기 보상
- 골드 구매

MVP의 비공개 MMR은 빠른 매치 내부 점수이며, 공개 랭크나 시즌 시스템과 다르다.

### 27.3 커뮤니티 확장

- 커뮤니티 댓글
- 게시글 좋아요
- 댓글 좋아요
- 커뮤니티 이미지 업로드

이미지 업로드를 추가할 때는 파일 크기, MIME type, 악성 파일 검사, 저장소 권한, CDN 캐시 무효화, 신고/삭제 정책을 별도 정의해야 한다.

### 27.4 카드/카드팩/밸런싱 확장

- MVP 이후 신규 카드 세트 추가
- 기본팩 외 신규 카드팩 추가
- MVP 이후 카드 상세 밸런스 패치
- 카드 삽입/삭제 및 카드팩 추가/수정 관리자 도구
- 카드 효과/확률/가격 관리자 편집

MVP 카드 12장의 상세 정의는 확장 후보가 아니라 구현 전 필수 확정 사항이다. 관리자 편집 도구를 추가할 때는 버전 고정, 기존 기보 재현성, 승인 워크플로우, 감사 로그, 롤백 정책을 별도 정의해야 한다.

### 27.5 관리자/운영 도구 확장

- 고급 관리자 검색, 필터, 정렬, 엑셀 다운로드
- 고급 금칙어 필터링
- 공백/특수문자 우회 탐지
- 초성 분리
- Unicode confusable 대응
- 외부 필터 서비스 연동

외부 필터 서비스를 연동할 때는 개인정보 전송 범위, 장애 시 fallback, 오탐 처리, 감사 로그를 별도 정의해야 한다.

### 27.6 인증/계정/API 확장

- 설정 화면에서 자체 로그인 계정과 OAuth 계정 연동
- 모바일 앱
- 외부 공개 API
- 외부 API용 JWT 인증

JWT 인증은 MVP의 같은 사이트 세션 쿠키 방식을 대체하는 것이 아니라 별도 클라이언트/API 지원을 위한 확장이다.

### 27.7 확장성/인프라 확장

- 수평 확장용 Redis/pub-sub
- 다중 인스턴스 STOMP 세션 처리
- 분산 타이머
- 분산 매치메이킹 큐
- 활성 경기 상태 외부 저장소
- 활성 경기 복구 고도화

이 항목들은 단순 인프라 추가가 아니라 현재 단일 인스턴스와 인메모리 활성 경기 상태 전제를 바꾸는 아키텍처 확장이다. 이벤트 순서, 중복 처리, 단일 writer 보장, 장애 복구, STOMP 세션 라우팅 정책을 별도 정의해야 한다.
